**프로젝트가 여러 개면 Codex 하나로는 안 돼.**

앞에서 Claude + Codex 2인 체제를 배웠어.
근데 실무에서는 프로젝트가 하나가 아니거든.

---

## 어떤 상황이야?

쇼핑몰을 만들고 있다고 치자.

처음엔 `packages/backend` 하나, `packages/frontend` 하나였어.
근데 서비스가 커지면서 이렇게 나뉘어:

```
my-shop/
├── projects/
│   ├── shop-api/       ← 고객용 API (상품 조회, 주문)
│   ├── admin-api/      ← 관리자용 API (상품 등록, 주문 관리)
│   ├── noti-api/       ← 알림 API (주문 확인 문자, 배송 알림)
│   ├── shop-web/       ← 고객이 보는 쇼핑 페이지
│   └── admin-web/      ← 사장님이 보는 관리 페이지
└── _shared/
    └── models/         ← 상품, 주문, 회원 데이터 (3개 API가 공유)
```

API 3개, 프론트 2개, 공유 데이터 1개.
이걸 한 곳에서 관리하면서 Codex를 프로젝트별로 보내야 해.

```
Claude (컨트롤타워)
  │
  ├── Codex → shop-api (고객 API)
  ├── Codex → admin-api (관리자 API)
  ├── Codex → noti-api (알림 API)
  ├── Codex → shop-web (고객 웹)
  └── Codex → admin-web (관리 페이지)
```

---

## 뭐가 문제야?

프로젝트 하나면 간단해:

```bash
bash scripts/run-codex.sh projects/shop-api "..."
```

근데 5개면?

```bash
# 매번 이렇게 치면 미쳐
bash scripts/run-codex.sh projects/shop-api "..."
bash scripts/run-codex.sh projects/admin-api "..."
bash scripts/run-codex.sh projects/noti-api "..."
bash scripts/run-codex.sh projects/shop-web "..."
bash scripts/run-codex.sh projects/admin-web "..."
```

그리고 더 큰 문제가 있어.

**공유 데이터.**

`상품(Product)` 모델을 shop-api, admin-api, noti-api 3개가 같이 써.
`_shared/models`에서 상품 모델을 수정하면 3개 프로젝트가 다 영향을 받아.

근데 Codex는 그걸 몰라.
자기가 작업하는 디렉토리만 보거든.

---

## 어떻게 해결해?

**프로젝트 레지스트리**를 만들어.

프로젝트 이름, 경로, 타입, 공유 관계를 JSON 하나에 정리해.
래퍼 스크립트가 이걸 읽고 자동으로 처리해.

```json
{
  "projects": {
    "shop-api": {
      "path": "projects/shop-api",
      "type": "express",
      "shared": ["models"]
    },
    "admin-api": {
      "path": "projects/admin-api",
      "type": "express",
      "shared": ["models"]
    },
    "noti-api": {
      "path": "projects/noti-api",
      "type": "express",
      "shared": ["models"]
    },
    "shop-web": {
      "path": "projects/shop-web",
      "type": "next",
      "shared": []
    },
    "admin-web": {
      "path": "projects/admin-web",
      "type": "next",
      "shared": []
    }
  },
  "aliases": {
    "shop": "shop-api",
    "admin": "admin-web",
    "noti": "noti-api"
  },
  "shared_modules": {
    "models": {
      "path": "_shared/models",
      "consumers": ["shop-api", "admin-api", "noti-api"]
    }
  }
}
```

이 파일이 있으면 3가지가 해결돼:

1. **단축명** — `shop`만 쳐도 `projects/shop-api`로 자동 변환
2. **공유 모듈 경고** — `_shared/models` 수정하면 "영향받는 프로젝트: 3개" 자동 안내
3. **관계 파악** — Claude가 어떤 프로젝트끼리 연결돼 있는지 알 수 있어

---

## 래퍼 스크립트는 어떻게 바뀌어?

기존 `run-codex.sh`를 확장해.

```bash
# 이전 — 전체 경로만 가능
bash scripts/run-codex.sh projects/shop-api "..."

# 이후 — 뭘로 넘기든 알아서 해석
bash scripts/run-codex.sh shop "..."              # 단축명
bash scripts/run-codex.sh shop-api "..."           # 정식명
bash scripts/run-codex.sh projects/shop-api "..."  # 전체 경로
bash scripts/run-codex.sh _shared/models "..."     # 공유 모듈
```

해석 순서:

1. `projects/...` 또는 `_shared/...`면 → 그대로 사용
2. `aliases`에 있으면 → 정식명으로 변환 → 경로 조회
3. `projects`에 있으면 → 경로 조회
4. 다 없으면 → 에러

핵심은 `resolve_project()` 함수야.
단축명이든 정식명이든 경로든 받아서, 실제 디렉토리 경로로 바꿔주는 거야.

````bash
resolve_project() {
  local input="$1"

  # 이미 전체 경로
  if [[ "$input" == projects/* ]] || [[ "$input" == _shared/* ]]; then
    echo "$REPO_ROOT/$input"
    return
  fi

  # projects.json에서 해석
  if [[ -f "$PROJECTS_JSON" ]]; then
    local resolved
    resolved=$(python3 -c "
import json
data = json.load(open('$PROJECTS_JSON'))
name = '$input'
if name in data.get('aliases', {}):
    name = data['aliases'][name]
if name in data.get('projects', {}):
    print(data['projects'][name]['path'])
elif name in data.get('shared_modules', {}):
    print(data['shared_modules'][name]['path'])
" 2>/dev/null)

    if [[ -n "$resolved" ]]; then
      echo "$REPO_ROOT/$resolved"
      return
    fi
  fi

  # 디렉토리 직접 탐색
  if [[ -d "$REPO_ROOT/projects/$input" ]]; then
    echo "$REPO_ROOT/projects/$input"
  fi
}
````

## 공유 모듈은 어떻게 처리해?

이게 핵심이야.

`_shared/models`에서 상품 모델에 `review_count` 필드를 추가한다고 치자.
shop-api, admin-api, noti-api 3개가 이 모델을 쓰고 있어.

근데 Codex는 그걸 몰라.
자기가 작업하는 `_shared/models` 디렉토리만 보거든.

래퍼 스크립트가 자동으로 **경고를 프롬프트에 주입**해:

```
⚠️ 이 모듈은 공유 모듈(_shared/models)이야.
영향받는 프로젝트: shop-api, admin-api, noti-api
수정할 때 이 프로젝트들과의 호환성을 반드시 유지해.
```

Codex가 이걸 보면 호환성을 깨는 수정을 안 해.
필드 이름을 바꾸거나, 기존 필드를 삭제하는 위험한 짓을 방지할 수 있어.

## 멀티프로젝트 실행 패턴이 뭐야?

5가지 패턴이 있어. 상황에 따라 골라 써.

### 패턴 1: 단일 프로젝트

한 프로젝트만 건드리는 작업. 제일 간단해.

```bash
# "상품 검색에 카테고리 필터 추가해줘"
bash scripts/run-codex.sh shop "상품 검색에 카테고리 필터 추가"
```

### 패턴 2: 순차 (API → 프론트)

API를 먼저 만들고, 그 결과를 프론트에 넘겨.

예를 들어 "상품 리뷰 기능"을 만든다고 치자.
API가 먼저 있어야 프론트에서 호출할 수 있잖아.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# 1단계: API 먼저
bash scripts/run-codex.sh shop-api \
  "TASK: 상품 리뷰 등록/조회 API 구현
   EXPECTED: POST /products/:id/reviews, GET /products/:id/reviews
   MUST NOT: 기존 상품 API 수정 금지
   CONTEXT: Product 모델은 _shared/models/product.js" > "$WORK/step1.json"

STATUS=$(python3 -c "import json; print(json.load(open('$WORK/step1.json'))['status'])")
SUMMARY=$(python3 -c "import json; print(json.load(open('$WORK/step1.json'))['summary'])")

# 2단계: API 완료되면 프론트
if [ "$STATUS" = "done" ]; then
  bash scripts/run-codex.sh shop-web \
    "TASK: 상품 상세 페이지에 리뷰 섹션 추가
     CONTEXT: 리뷰 API 완료됨 — $SUMMARY"
fi

rm -rf "$WORK"
```

핵심은 **1단계 결과를 2단계 CONTEXT에 넘기는 거야.**
Codex가 API 응답 형식을 알아야 프론트를 정확하게 만들거든.

### 패턴 3: 병렬 (독립 프로젝트)

서로 안 건드리는 프로젝트는 동시에 돌려.

shop-web에 배너를 추가하고, admin-web에 통계 차트를 추가한다고 치자.
둘은 서로 전혀 관계없으니까 동시에 돌려도 돼.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

bash scripts/run-codex.sh shop-web "메인 배너 슬라이더 추가" > "$WORK/shop.json" &
bash scripts/run-codex.sh admin-web "주문 통계 차트 추가" > "$WORK/admin.json" &
wait  # 둘 다 끝날 때까지 대기

# 결과 확인
python3 -c "import json; print(json.load(open('$WORK/shop.json'))['status'])"
python3 -c "import json; print(json.load(open('$WORK/admin.json'))['status'])"

rm -rf "$WORK"
```

:::warning
**같은 프로젝트 내 작업은 병렬 금지.** 두 Codex가 같은 파일을 건드리면 나중 거가 먼저 걸 덮어써.
:::

### 패턴 4: 공유 모듈 → 소비자

공유 모듈을 먼저 수정하고, 영향받는 프로젝트를 업데이트해.

예: 상품 모델에 `review_count` 필드를 추가하는 거야.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# 1단계: 공유 모듈 수정 (반드시 순차)
bash scripts/run-codex.sh _shared/models \
  "TASK: Product 모델에 review_count 필드 추가
   MUST NOT: 기존 필드 삭제 금지" > "$WORK/shared.json"

# 2단계: 소비자 프로젝트 (병렬 가능 — 서로 다른 프로젝트니까)
bash scripts/run-codex.sh shop-api \
  "TASK: Product review_count 필드 반영
   CONTEXT: _shared/models에 review_count 추가됨" > "$WORK/shop.json" &
bash scripts/run-codex.sh admin-api \
  "TASK: Product review_count 필드 반영
   CONTEXT: _shared/models에 review_count 추가됨" > "$WORK/admin.json" &
wait

rm -rf "$WORK"
```

:::warning
**공유 모듈은 절대 병렬로 수정하지 마.** 순차 실행 후에 소비자를 업데이트해.
:::

### 패턴 5: 풀스택 기능

데이터 모델 → API → 프론트 전체를 한 번에 만드는 거야.

"상품 리뷰 기능"을 처음부터 끝까지 만든다고 치자.
Phase로 쪼개서 순서를 보장해.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# Phase 1: 데이터 모델 (순차 — 기반 작업)
bash scripts/run-codex.sh _shared/models "Review 모델 추가" > "$WORK/p1.json"

# Phase 2: API 구현 (병렬 — 서로 다른 프로젝트)
bash scripts/run-codex.sh shop-api "리뷰 등록/조회 API" > "$WORK/p2-shop.json" &
bash scripts/run-codex.sh admin-api "리뷰 관리 API (삭제, 숨김)" > "$WORK/p2-admin.json" &
wait

# Phase 3: 프론트 구현 (병렬 — 서로 다른 프로젝트)
bash scripts/run-codex.sh shop-web "상품 상세에 리뷰 섹션" > "$WORK/p3-shop.json" &
bash scripts/run-codex.sh admin-web "리뷰 관리 페이지" > "$WORK/p3-admin.json" &
wait

rm -rf "$WORK"
```

```
Phase 1 (순차)     Phase 2 (병렬)     Phase 3 (병렬)
─────────────     ──────────────     ──────────────
 _shared/models    shop-api ──┐      shop-web ──┐
 (Review 모델)     admin-api ─┘      admin-web ─┘
                       ↓                  ↓
                  API 2개 완료       프론트 2개 완료
```

## CLAUDE.md에 뭘 넣어?

Claude가 알아서 멀티프로젝트 패턴을 쓰게 하려면 `CLAUDE.md`에 박아놔.

```markdown
## Codex 오케스트레이션 (멀티프로젝트)

### 역할 분담
- **Claude**: 전략, 작업 분해, 크로스 프로젝트 조율
- **Codex**: 단일 프로젝트 내 구현

Claude가 직접 수정하는 경우:
- 크로스 프로젝트 연동 (양쪽 컨텍스트 필요)
- _shared/ 수정 시 영향 분석
- 간단한 1~2줄 수정
- Codex 3번 연속 실패 시

### 위임 프로토콜
TASK / EXPECTED / MUST NOT / CONTEXT 4가지 포함.

### 래퍼 스크립트
bash scripts/run-codex.sh <project> <task>
단축명 지원: shop, admin, noti

### 안전 규칙
- 같은 프로젝트 내 작업은 병렬 금지
- _shared/ 수정은 순차 실행 후 소비자 업데이트
- 3번 연속 실패 시 Codex 위임 중단
```

이게 있으면 Claude가 작업을 받았을 때:
1. 어떤 프로젝트에 영향이 있는지 파악하고
2. 순서를 정하고 (모델 → API → 프론트)
3. Codex에 위임하고
4. 결과를 보고 다음 단계를 결정해

## 프로젝트 컨텍스트는 어떻게 줘?

두 가지 방법이 있어.

### 방법 1: AGENTS.md (권장)

프로젝트 디렉토리에 `AGENTS.md`를 두면 **Codex가 자동으로 읽어.**
래퍼 스크립트나 주입 없이도 작동해.

```
my-shop/
├── projects/
│   ├── shop-api/
│   │   ├── AGENTS.md    ← Codex가 자동으로 읽음
│   │   └── src/
│   └── admin-web/
│       ├── AGENTS.md    ← Codex가 자동으로 읽음
│       └── src/
└── _shared/
    └── models/
```

```bash
# 터미널에서 직접 실행해도 AGENTS.md 자동 적용
codex exec -C ./projects/shop-api --full-auto "..."

# 래퍼로 실행해도 마찬가지
bash scripts/run-codex.sh shop "..."
```

`AGENTS.md` 예시:

````markdown
# Shop API — Agent Guide

**Express + MongoDB**

## Critical Rules

1. 에러 코드는 정수만: `ORDER_NOT_FOUND = 1001`
2. `find_by` 사용 금지: `User.where(email: e).first` 사용

## Commands

```bash
npm test
npm run dev
```

## Response Format

```json
{"status":"done","summary":"...","files_modified":[],"next_steps":[],"blocked_reason":""}
```
````

### 방법 2: CLAUDE.md 주입 (래퍼 스크립트)

래퍼 스크립트가 **해당 프로젝트의 `CLAUDE.md`를 프롬프트에 삽입해.**

```bash
bash scripts/run-codex.sh shop "상품 검색에 카테고리 필터 추가"
```

내부에서 이렇게 합쳐져:

```
[shop-api/CLAUDE.md 내용]          ← 프로젝트 규칙 자동 주입
---
상품 검색에 카테고리 필터 추가
---
작업 완료 후 JSON으로만 응답해라...
```

### 어떤 걸 써?

| | AGENTS.md | CLAUDE.md 주입 |
|--|-----------|----------------|
| 터미널 직접 실행 | ✅ 자동 적용 | ❌ 래퍼 필요 |
| `_shared` 경고 주입 | ❌ 정적 파일이라 불가 | ✅ 래퍼가 동적으로 추가 |
| 관리 편의 | 프로젝트별 파일 1개 | 래퍼 스크립트 유지 필요 |

**실무 추천**: 각 프로젝트에 `AGENTS.md` 두고, 래퍼 스크립트는 `_shared` 경고 주입 + 단축명 해석 용도로만 써.

:::tip
`AGENTS.md`와 `CLAUDE.md` 주입을 같이 써도 문제없어.
Codex가 `AGENTS.md`를 먼저 읽고, 래퍼가 `CLAUDE.md`를 뒤에 붙이는 방식이라
규칙이 두 번 들어가는 셈이지만, Codex는 겹쳐도 잘 처리해.
:::

## 실패하면 어떻게 돼?

앞에서 말한 "3번 연속 실패 시 Claude 직접 처리" 규칙은 그대로야.

멀티프로젝트에서 추가로 주의할 거:

**Phase 중간에 실패하면 이후 Phase를 실행하지 마.**

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# Phase 1 실패 → Phase 2 스킵
bash scripts/run-codex.sh _shared/models "Review 모델 추가" > "$WORK/p1.json"
STATUS=$(python3 -c "import json; print(json.load(open('$WORK/p1.json'))['status'])")

if [ "$STATUS" != "done" ]; then
  echo "Phase 1 실패. Phase 2 스킵."
  rm -rf "$WORK"
  exit 1
fi

# Phase 1 성공해야 Phase 2 실행
bash scripts/run-codex.sh shop-api "리뷰 API" > "$WORK/p2-shop.json" &
bash scripts/run-codex.sh admin-api "리뷰 관리 API" > "$WORK/p2-admin.json" &
wait

rm -rf "$WORK"
```

왜? Phase 1에서 Review 모델을 만들어야 Phase 2에서 그 모델을 쓸 수 있거든.
없는 모델을 참조하면 Codex가 에러를 내거나 엉뚱한 걸 만들어.

## 한 줄 정리

| 전략 | 효과 |
|------|------|
| 프로젝트 레지스트리 | 단축명 + 관계 파악 |
| `_shared` 경고 주입 | 호환성 깨짐 방지 |
| `AGENTS.md` (각 프로젝트) | 터미널 직접 실행에도 규칙 자동 적용 |
| `CLAUDE.md` 주입 (래퍼) | `_shared` 경고 등 동적 컨텍스트 추가 |
| Phase 분리 | 모델 → API → 프론트 순서 보장 |
| 병렬 실행 | 독립 프로젝트는 동시에 |

단일 프로젝트 오케스트레이션이 잘 돌아가면 멀티로 넘어와.
핵심은 **관계를 레지스트리에 적어두는 거야.** Claude가 관계를 알아야 순서를 정할 수 있거든.
