**프로젝트가 여러 개면 Codex 하나로는 안 돼.**

앞에서 Claude + Codex 2인 체제를 배웠어.
근데 실무에서는 프로젝트가 하나가 아니거든.

백엔드 API 3개, 프론트 2개, 공유 모듈 5개.
이걸 한 곳에서 관리하면서 Codex를 프로젝트별로 보내야 해.

```
Claude (컨트롤타워)
  │
  ├── Codex → 백엔드 API (Rails)
  ├── Codex → 관리자 프론트 (Nuxt)
  ├── Codex → 사용자 프론트 (Nuxt)
  └── Codex → 공유 모듈 (models, workers)
```

---

## 뭐가 문제야?

단일 프로젝트면 이렇게 하면 끝이야:

```bash
bash scripts/run-codex.sh packages/backend "..."
```

근데 프로젝트가 10개면?

```bash
# 매번 이렇게 치면 미쳐
bash scripts/run-codex.sh projects/commerce-api "..."
bash scripts/run-codex.sh projects/internal-api "..."
bash scripts/run-codex.sh projects/message-api "..."
bash scripts/run-codex.sh projects/admin "..."
bash scripts/run-codex.sh projects/invoice "..."
```

그리고 더 큰 문제가 있어.

**공유 모듈.**

백엔드 3개가 같은 모델 파일을 쓰거든.
공유 모듈을 수정하면 3개 프로젝트가 다 영향을 받아.
Codex는 그걸 몰라.

---

## 어떻게 해결해?

**프로젝트 레지스트리**를 만들어.

프로젝트 이름, 경로, 타입, 공유 모듈 관계를 JSON 하나에 정리해.
래퍼 스크립트가 이걸 읽고 자동으로 처리해.

```json
{
  "projects": {
    "commerce-api": {
      "path": "projects/commerce-api",
      "type": "rails",
      "shared": ["app-models", "locales", "workers"]
    },
    "admin": {
      "path": "projects/admin",
      "type": "nuxt",
      "shared": ["vendor"]
    }
  },
  "aliases": {
    "commerce": "commerce-api",
    "msg": "message-api"
  },
  "shared_modules": {
    "app-models": {
      "path": "_shared/app-models",
      "consumers": ["internal-api", "commerce-api", "message-api"]
    },
    "vendor": {
      "path": "_shared/vendor",
      "consumers": ["admin", "invoice"]
    }
  }
}
```

이 파일이 있으면 3가지가 해결돼:

1. **단축명** — `commerce` 만 쳐도 `projects/commerce-api`로 자동 변환
2. **공유 모듈 경고** — `_shared/app-models` 수정하면 "영향받는 프로젝트: 3개" 자동 안내
3. **관계 파악** — Claude가 어떤 프로젝트끼리 연결돼 있는지 알 수 있어

---

## 래퍼 스크립트는 어떻게 바뀌어?

기존 `run-codex.sh`를 확장해. 핵심 변경은 **프로젝트 해석 로직**.

```bash
# 이전 — 전체 경로만 가능
bash scripts/run-codex.sh projects/commerce-api "..."

# 이후 — 뭘로 넘기든 알아서 해석
bash scripts/run-codex.sh commerce "..."          # alias
bash scripts/run-codex.sh commerce-api "..."       # 정식명
bash scripts/run-codex.sh projects/commerce-api "..." # 전체 경로
bash scripts/run-codex.sh _shared/app-models "..."    # 공유 모듈
```

해석 순서:

1. `projects/...` 또는 `_shared/...`면 → 그대로 사용
2. 절대경로면 → 그대로 사용
3. `aliases`에 있으면 → 정식명으로 변환 → 경로 조회
4. `projects`에 있으면 → 경로 조회
5. `projects/` 디렉토리에 그 이름의 폴더가 있으면 → 사용
6. 다 없으면 → JSON 에러 반환

````bash
resolve_project() {
  local input="$1"

  # 이미 전체 경로
  if [[ "$input" == projects/* ]] || [[ "$input" == _shared/* ]]; then
    echo "$REPO_ROOT/$input"
    return
  fi

  # projects.json에서 해석
  if [[ -f "$PROJECTS_JSON" ]]; then
    local resolved
    resolved=$(python3 -c "
import json
data = json.load(open('$PROJECTS_JSON'))
name = '$input'
if name in data.get('aliases', {}):
    name = data['aliases'][name]
if name in data.get('projects', {}):
    print(data['projects'][name]['path'])
elif name in data.get('shared_modules', {}):
    print(data['shared_modules'][name]['path'])
" 2>/dev/null)

    if [[ -n "$resolved" ]]; then
      echo "$REPO_ROOT/$resolved"
      return
    fi
  fi

  # 디렉토리 직접 탐색
  if [[ -d "$REPO_ROOT/projects/$input" ]]; then
    echo "$REPO_ROOT/projects/$input"
  fi
}
````

## 공유 모듈은 어떻게 처리해?

이게 핵심이야.

`_shared/app-models`를 수정하면 3개 백엔드 프로젝트가 영향을 받아.
근데 Codex는 그걸 몰라. 자기가 작업하는 디렉토리만 보거든.

래퍼 스크립트가 자동으로 **경고를 프롬프트에 주입**해:

```bash
# _shared/ 대상이면 자동으로 이 문구가 Codex 프롬프트에 추가돼
⚠️ 이 모듈은 공유 모듈(_shared/app-models)이다.
영향받는 프로젝트: internal-api, commerce-api, message-api
수정 시 이 프로젝트들과의 호환성을 반드시 유지해라.
```

Codex가 이걸 보면 호환성을 깨는 수정을 안 해.
메서드 이름을 바꾸거나, 필드를 삭제하는 위험한 짓을 방지할 수 있어.

## 멀티프로젝트 실행 패턴이 뭐야?

5가지 패턴이 있어. 상황에 따라 골라 써.

### 패턴 1: 단일 프로젝트

한 프로젝트만 건드리는 작업. 제일 간단해.

```bash
bash scripts/run-codex.sh commerce "..."
```

### 패턴 2: 순차 (백엔드 → 프론트)

API를 먼저 만들고, 그 결과를 프론트에 넘겨.

```bash
# 세션 고유 디렉토리 — 여러 터미널에서 동시에 돌려도 안 겹쳐
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# 1단계: 백엔드
bash scripts/run-codex.sh commerce "API 구현" > "$WORK/step1.json"

STATUS=$(python3 -c "import json; print(json.load(open('$WORK/step1.json'))['status'])")
SUMMARY=$(python3 -c "import json; print(json.load(open('$WORK/step1.json'))['summary'])")

# 2단계: 프론트 (백엔드 결과를 CONTEXT로)
if [ "$STATUS" = "done" ]; then
  bash scripts/run-codex.sh admin \
    "TASK: 프론트 구현.
     CONTEXT: 백엔드 완료 — $SUMMARY"
fi

rm -rf "$WORK"
```

핵심은 **1단계 결과를 2단계 CONTEXT에 넘기는 거야.**
Codex가 API 응답 형식을 알아야 프론트를 정확하게 만들거든.

### 패턴 3: 병렬 (독립 프로젝트)

서로 안 건드리는 프로젝트는 동시에 돌려.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

bash scripts/run-codex.sh commerce "..." > "$WORK/commerce.json" &
bash scripts/run-codex.sh admin "..." > "$WORK/admin.json" &
wait  # 둘 다 끝날 때까지 대기

# 결과 확인
python3 -c "import json; print(json.load(open('$WORK/commerce.json'))['status'])"
python3 -c "import json; print(json.load(open('$WORK/admin.json'))['status'])"

rm -rf "$WORK"
```

:::warning
**같은 프로젝트 내 작업은 병렬 금지.** 두 Codex가 같은 파일을 건드리면 나중 거가 먼저 걸 덮어써.
:::

### 패턴 4: 공유 모듈 → 소비자

공유 모듈을 먼저 수정하고, 영향받는 프로젝트를 업데이트해.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# 1단계: 공유 모듈 수정 (반드시 순차)
bash scripts/run-codex.sh _shared/app-models \
  "TASK: Notification 모델에 read_at 필드 추가
   MUST NOT: 기존 필드 삭제 금지" > "$WORK/shared.json"

# 2단계: 소비자 프로젝트 (병렬 가능 — 서로 다른 프로젝트니까)
bash scripts/run-codex.sh commerce \
  "TASK: Notification read_at 필드 반영
   CONTEXT: _shared/app-models에 read_at 추가됨" > "$WORK/commerce.json" &
bash scripts/run-codex.sh internal \
  "TASK: Notification read_at 필드 반영
   CONTEXT: _shared/app-models에 read_at 추가됨" > "$WORK/internal.json" &
wait

rm -rf "$WORK"
```

:::warning
**공유 모듈은 절대 병렬로 수정하지 마.** 순차 실행 후에 소비자를 업데이트해.
:::

### 패턴 5: 풀스택 기능

모델 → API → 프론트 전체를 한 번에 만드는 거야. Phase로 쪼개.

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# Phase 1: 데이터 모델 (순차 — 기반 작업)
bash scripts/run-codex.sh _shared/app-models "모델 추가" > "$WORK/p1.json"

# Phase 2: API 구현 (병렬 — 서로 다른 프로젝트)
bash scripts/run-codex.sh commerce "사용자 API" > "$WORK/p2-commerce.json" &
bash scripts/run-codex.sh internal "관리자 API" > "$WORK/p2-internal.json" &
wait

# Phase 3: 프론트 구현 (병렬 — 서로 다른 프로젝트)
bash scripts/run-codex.sh admin "관리 UI" > "$WORK/p3-admin.json" &
bash scripts/run-codex.sh invoice "사용자 UI" > "$WORK/p3-invoice.json" &
wait

rm -rf "$WORK"
```

```
Phase 1 (순차)     Phase 2 (병렬)      Phase 3 (병렬)
─────────────     ────────────────    ────────────────
 _shared          commerce-api ─┐     admin ──────┐
 (모델)           internal-api ─┤     invoice ────┤
                  message-api ──┘                  │
                       ↓                           ↓
                  API 3개 완료              프론트 2개 완료
```

## CLAUDE.md에 뭘 넣어?

Claude가 알아서 멀티프로젝트 패턴을 쓰게 하려면 `CLAUDE.md`에 박아놔.

```markdown
## Codex 오케스트레이션 (멀티프로젝트)

### 역할 분담
- **Claude**: 전략, 작업 분해, 크로스 프로젝트 조율
- **Codex**: 단일 프로젝트 내 구현

Claude가 직접 수정하는 경우:
- 크로스 프로젝트 연동 (양쪽 컨텍스트 필요)
- _shared/ 수정 시 영향 분석
- 간단한 1~2줄 수정
- Codex 3번 연속 실패 시

### 위임 프로토콜
TASK / EXPECTED / MUST NOT / CONTEXT 4가지 포함.

### 래퍼 스크립트
bash scripts/agent/run-codex.sh <project> <task>
단축명 지원: commerce, internal, msg, admin, invoice

### 안전 규칙
- 같은 프로젝트 내 작업은 병렬 금지
- _shared/ 수정은 순차 실행 후 소비자 업데이트
- 3번 연속 실패 시 Codex 위임 중단
```

이게 있으면 Claude가 작업을 받았을 때:
1. 어떤 프로젝트에 영향이 있는지 파악하고
2. 순서를 정하고 (모델 → API → 프론트)
3. Codex에 위임하고
4. 결과를 보고 다음 단계를 결정해

## 프로젝트 CLAUDE.md 자동 주입이 뭐야?

래퍼 스크립트가 **해당 프로젝트의 `CLAUDE.md`를 Codex 프롬프트에 자동으로 넣어줘.**

```bash
bash scripts/run-codex.sh commerce "Order 취소 구현"
```

이렇게 호출하면 내부에서:

```
[commerce-api/CLAUDE.md 내용]        ← 프로젝트 규칙 자동 주입
---
Order 취소 구현                      ← 네가 쓴 작업 내용
---
작업 완료 후 JSON으로만 응답해라...   ← JSON 응답 강제
```

이렇게 합쳐져서 Codex에 전달돼.

그래서 **Rails 프로젝트면 Rails 규칙**, **Nuxt 프로젝트면 Nuxt 규칙**이 자동으로 적용돼.
`find_by` 쓰지 마라, Tailwind separator는 `__`다 — 이런 거를 매번 안 써도 돼.

:::tip
프로젝트별 `CLAUDE.md`가 충실할수록 Codex 위임 성공률이 올라가.
"이 프로젝트에서 하면 안 되는 것"을 적어놓으면 Codex가 알아서 피해.
:::

## 실패하면 어떻게 돼?

앞에서 말한 "3번 연속 실패 시 Claude 직접 처리" 규칙은 그대로야.

멀티프로젝트에서 추가로 주의할 거:

**Phase 중간에 실패하면 이후 Phase를 실행하지 마.**

```bash
WORK=$(mktemp -d /tmp/codex-XXXXXX)

# Phase 1 실패 → Phase 2 스킵
bash scripts/run-codex.sh _shared/app-models "..." > "$WORK/p1.json"
STATUS=$(python3 -c "import json; print(json.load(open('$WORK/p1.json'))['status'])")

if [ "$STATUS" != "done" ]; then
  echo "Phase 1 실패. Phase 2 스킵."
  rm -rf "$WORK"
  exit 1
fi

# Phase 1 성공해야 Phase 2 실행
bash scripts/run-codex.sh commerce "..." > "$WORK/p2-commerce.json" &
bash scripts/run-codex.sh internal "..." > "$WORK/p2-internal.json" &
wait

rm -rf "$WORK"
```

왜? Phase 1에서 모델을 만들어야 Phase 2에서 그 모델을 쓸 수 있거든.
없는 모델을 참조하면 Codex가 에러를 내거나 엉뚱한 걸 만들어.

## 한 줄 정리

| 전략 | 효과 |
|------|------|
| 프로젝트 레지스트리 | 단축명 + 관계 파악 |
| _shared 경고 주입 | 호환성 깨짐 방지 |
| CLAUDE.md 자동 주입 | 프로젝트별 규칙 적용 |
| Phase 분리 | 모델 → API → 프론트 순서 보장 |
| 병렬 실행 | 독립 프로젝트는 동시에 |

단일 프로젝트 오케스트레이션이 잘 돌아가면 멀티로 넘어와.
핵심은 **관계를 레지스트리에 적어두는 거야.** Claude가 관계를 알아야 순서를 정할 수 있거든.
