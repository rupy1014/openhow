**[Codex 플러그인](/부록-codex-plugin) 깔았으면 다음은 이거야.**

플러그인만 깔면 Codex가 코드를 짜긴 해. 근데 레이어를 무시하고, 컨벤션을 안 지키고, 문서랑 코드가 따로 놀아. 이걸 잡는 게 하네스 엔지니어링이야.

---

## 하네스가 뭐냐

AI 모델을 감싸는 **전체 인프라**야. 도구, 권한, 문서, 제약, 검증 — 모델 자체를 제외한 모든 것.

말이 아무리 빨라도 마구(harness) 없이는 마차를 못 끌어. AI도 마찬가지야.

OpenAI가 [공식 블로그](https://openai.com/index/harness-engineering/)에서 이걸 방법론으로 정리했어. 5개월 동안 100만 줄 넘는 프로덕션 코드를 — **사람이 직접 쓴 코드 0줄**로 — 만든 비결이야.

> Martin Fowler도 "컨텍스트 엔지니어링 + 아키텍처 제약 + 가비지 컬렉션"이라고 [정리했어](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html).

---

## 3대 축이 뭐냐

### 1. 컨텍스트 엔지니어링

**에이전트가 컨텍스트에서 못 읽으면 존재하지 않는 거야.**

Slack에 있는 설계 논의, 구글독에 있는 스펙, 사람 머리에 있는 암묵지 — 에이전트한테는 다 안 보여. 레포가 유일한 진실이어야 해.

| 파일 | 역할 |
|------|------|
| `CLAUDE.md` / `AGENTS.md` | 에이전트 행동 규칙. 금지 사항, 코딩 컨벤션, 워크플로우 |
| `docs/architecture.md` | 디렉토리 맵 + 레이어 구조 + 의존성 방향 |
| `docs/decisions/` | 설계 결정 기록 (ADR). "왜 이렇게 했냐"를 남겨야 에이전트가 반복 안 해 |

### 2. 아키텍처 제약

**"제안"이 아니라 "차단"이야.**

문서에 "이 레이어는 건드리지 마"라고 써봤자 소용없어. 린터와 구조 테스트로 **기계적으로 막아야** 해.

| 제약 예시 | 어떻게 강제하냐 |
|-----------|----------------|
| 레이어 의존성 방향 | ESLint `import/no-restricted-paths` 또는 구조 테스트 |
| 파일 크기 제한 | pre-commit hook (`wc -l < 300`) |
| 네이밍 컨벤션 | 커스텀 린터 규칙 |
| 구조화된 로깅만 허용 | `console.log` 금지 린터 + 공용 logger 강제 |

### 3. 엔트로피 관리

**AI가 짠 코드는 썩어.**

시간이 지나면 네이밍이 흔들리고, 문서가 코드랑 안 맞고, 데드코드가 쌓여. 사람이 짜도 그런데 AI는 더 빨리 쌓여.

| 청소 대상 | 방법 |
|-----------|------|
| 문서 ↔ 코드 동기화 | `CLAUDE.md`와 실제 구조 비교 에이전트 |
| 데드코드 | `ts-prune` / `knip` 주기 실행 |
| 컨벤션 위반 | 린터 리포트 → 자동 수정 PR |
| 테스트 커버리지 | 커버리지 리포트 → 부족한 파일에 테스트 추가 PR |

---

## 프로젝트마다 다르지 않냐

다르다. 3대 축은 같지만 구체적인 세팅은 완전히 달라.

| 프로젝트 유형 | 핵심 제약 |
|--------------|----------|
| 모노레포 | 패키지 간 의존성 방향 |
| API 서버 | 구조화된 로깅, 에러 핸들링 패턴 |
| 프론트엔드 | 컴포넌트 디렉토리 규칙, 번들 크기 |
| CLI 도구 | 커맨드 등록 패턴, 옵션 파싱 구조 |

그래서 범용 템플릿 대신 **네 프로젝트를 분석해서 맞춤 세팅하는 프롬프트**를 만들었어.

## 한 줄 정리

코드를 잘 짜는 AI가 아니라 AI가 잘 짤 수 있는 환경을 만들어. 이게 하네스 엔지니어링이야.
