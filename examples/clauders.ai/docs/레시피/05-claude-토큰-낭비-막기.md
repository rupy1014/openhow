**토큰이 빨리 녹는 건 AI가 멍청해서가 아니야. 세팅이 새고 있는 거야.**

Meta Alchemist가 2026-03-31에 올린 글을 바탕으로 다시 썼어.
원문은 [여기](https://x.com/meta_alchemist/status/2038919582111670415)야.
필드 이름은 [Anthropic 공식 문서](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)도 같이 확인했어.
모델 이름이랑 세부 조건은 바뀔 수 있으니, 실제 적용 전엔 공식 문서를 한 번 더 봐.

---

## 뭐부터 하면 돼?

복잡하게 생각하지 마.
일단 이 다섯 개부터 해.

| 바로 할 것 | 왜 좋냐 |
|-----------|---------|
| 작은 일은 작은 모델 | 쉬운 작업에 비싼 추론을 안 태워 |
| `effort`를 낮춰 | 필요 없는 깊은 생각을 줄여 |
| 대화 주제가 바뀌면 `/clear` | 옛날 대화를 계속 들고 안 가 |
| 답 길이를 먼저 제한해 | 출력 토큰이 바로 줄어 |
| 큰 고정 규칙은 캐시해 | 매번 같은 설명을 다시 안 읽어 |

이 다섯 개만 해도 체감이 커.
특히 긴 세션에서 차이가 많이 나.

---

## 왜 이렇게 빨리 새냐?

토큰은 수도세랑 비슷해.
조금씩 새면 나중에 크게 나와.

보통 여기서 많이 새:

- 긴 대화를 계속 이어가
- 안 바뀌는 규칙을 매번 다시 넣어
- 간단한 작업도 무조건 무거운 추론으로 돌려
- 출력 형식을 안 정해서 말이 길어져
- 안 쓰는 도구 설명까지 같이 보내

Claude Code에서도 비슷해.
한 주제가 끝났는데 계속 이어가면 이전 문맥을 또 읽어.

그래서 **토큰 절약은 짠돌이 팁이 아니라 문맥 정리 습관**이야.

## 어떻게 세팅하냐?

:::steps

### 작은 일은 가볍게 보내

분류, 한 줄 요약, 형식 맞추기 같은 건 작은 모델이나 낮은 effort로 보내.
깊게 생각할 일이 아닌데 무겁게 돌리면 돈이랑 시간이 같이 나가.

```json
{
  "model": "claude-haiku-4-5",
  "output_config": {
    "effort": "low"
  }
}
```

### 긴 규칙은 캐시해

매번 같은 시스템 규칙, 긴 배경 설명, 자주 쓰는 도구 설명은 캐시 대상이야.
반복해서 쓰는 고정 프롬프트면 효과가 크다.

[복붙용 예시 JSON](./claude-cache-request.example.json)

### 비실시간 작업은 배치로 빼

지금 당장 답 안 받아도 되는 대량 작업 있잖아.
리포트 100개 생성, 로그 분류, 긴 문서 일괄 요약 같은 거.
그건 [Message Batches API](https://platform.claude.com/docs/en/build-with-claude/batch-processing)로 빼.

원문 기준으론 비용 절약폭이 컸고,
공식 문서도 배치와 캐시를 같이 쓰는 패턴을 권장해.

### 출력 길이를 먼저 묶어

"짧게"라고만 하지 마.
숫자로 잘라.

```text
3줄로만 답해.
불릿은 3개까지만.
JSON만 반환해. 설명하지 마.
```

이 한 줄이 바로 돈을 아껴.
출력 토큰이 줄어드니까.

### 주제 바뀌면 세션도 갈아타

Claude Code에선 이게 바로 먹혀.

```text
/clear
/compact
/cost
/effort low
```

- `/clear` → 주제 바뀔 때
- `/compact` → 긴 대화 압축할 때
- `/cost` → 지금 얼마나 쓰는지 볼 때
- `/effort low` → 가벼운 턴으로 내릴 때

:::

## 캐시는 어떻게 거냐?

캐시는 **맨 앞에 오래 안 바뀌는 덩어리**를 저장하는 거야.

예를 들어:

- 우리 서비스 말투 규칙
- 고정 시스템 프롬프트
- 길고 반복되는 배경 문서
- 자주 쓰는 도구 설명

이런 건 좋다.

반대로 이런 건 캐시 기준점으로 잡으면 안 좋아:

- 지금 시간
- 세션 ID
- 사용자마다 매번 바뀌는 입력
- 바로 직전 질문

쉽게 말하면 이거야.

> "매번 똑같은 앞부분"까지만 캐시해야 해.

공식 문서 기준으로는 top-level 자동 캐시도 있고,
블록별 `cache_control`도 있어.

- **자동 캐시** → 긴 대화 전체를 쉽게 다루기 좋아
- **블록 캐시** → 시스템 프롬프트처럼 고정 앞부분만 잡기 좋아

[Prompt caching 공식 문서](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)

## 뭘 하면 캐시가 깨지냐?

여기가 함정이야.
캐시를 걸어도 자꾸 미스 나면 소용없어.

원문이 특히 강조한 건 이거였어:

- 이미지가 들어가고 빠지는 패턴이 바뀌면 캐시가 잘 깨져
- 캐시 기준점 앞 텍스트가 한 글자만 달라도 다시 써야 해
- thinking 설정을 자꾸 바꾸면 일부 캐시 전략이 꼬일 수 있어
- TTL이 지나면 다시 써야 해

Anthropic 공식 문서도 비슷하게 말해.
정확히 같은 prefix가 유지돼야 캐시 히트가 나.

그래서 실무에선 이렇게 나누면 편해:

1. **텍스트만 도는 요청**
2. **이미지가 섞인 요청**
3. **툴 호출이 많은 요청**

이걸 한 세트로 생각하지 마.
서로 다른 흐름으로 봐.

---

## effort는 언제 낮추면 돼?

`effort`는 "얼마나 토큰을 적극적으로 쓰게 둘 거냐"에 가까워.

Anthropic 공식 문서 기준으로 Sonnet 4.6은 `medium`이 기본 추천선에 가깝고,
간단한 일은 `low`가 잘 맞아.

이 기준으로 시작해봐.

| 작업 | 추천 |
|------|------|
| 분류, 라우팅, 짧은 요약 | `low` |
| 일반 코딩, 툴 많이 쓰는 작업 | `medium` |
| 어려운 설계, 깊은 분석 | `high` |

무조건 `high`로 두면 좋을 것 같지?
근데 쉬운 일에도 장비를 과하게 쓰는 거라 낭비가 커.

[Effort 공식 문서](https://platform.claude.com/docs/en/build-with-claude/effort)

## 모델 라우터는 어떻게 짜면 돼?

모든 요청을 같은 모델로 보내지 마.
이건 제일 쉬운 절약 레버야.

아래처럼 나누면 돼:

- 분류 / 라우팅 / 짧은 추출 → Haiku
- 일반 코딩 / 요약 / 툴 쓰는 작업 → Sonnet
- 진짜 어려운 설계 판단 → Opus

[복붙용 라우터 예시](./claude-model-router.example.ts)

핵심은 이거야.

**작은 일은 빨리 끝내고, 큰 일에만 비싼 추론을 써.**

## Claude Code에선 뭘 기억하면 돼?

API 안 써도 바로 적용되는 게 있어.

### 1. 새 주제면 `/clear`

이전 작업을 계속 들고 가면 그만큼 다시 읽어.
주제가 바뀌면 세션도 갈아타.

### 2. 길어졌으면 `/compact`

대화가 길어지면 압축해.
핵심만 남기고 뒤를 이어가면 돼.

### 3. 쉬운 일은 `/effort low`

간단한 수정, 짧은 요약, 경량 작업은 낮춰.
체감 속도도 좋아져.

### 4. 중간중간 `/cost`

안 보면 감이 안 와.
측정 안 하면 습관이 안 바뀌어.

이미 기본 비용 관리만 보고 싶으면 [입문의 비용 관리](../입문/05-비용관리.md)부터 봐.
지금 글은 그 다음 단계야.
