**잘 쓰는 사람은 처음부터 길게 안 싸워. 결과 먼저 잠그고, 에이전트가 헤매지 못하게 레일을 깔아.**

Codex CLI는 "좋은 모델"보다 "좋은 운영"에서 차이가 더 크게 나. 결과부터 박고, 과정은 짧게 쪼개고, 라이브 세션은 병렬로 굴리고, 길어지면 다시 압축하면 돼.

이번 글은 기존 뼈대는 그대로 두고, 실전에서 자주 먹히는 운영 패턴을 더 촘촘하게 채운 버전이야. 혼자 쓰는 사람, 팀에서 쓰는 사람, CI에 붙이는 사람, OMX까지 얹어서 굴리는 사람 기준으로 적어둘게.

---

## AGENTS.md를 어디까지 쪼개야 덜 꼬이냐?

핵심은 3층이야. `~/.codex/AGENTS.md`에 전역 규칙을 두고, 레포 루트 `AGENTS.md`에 프로젝트 공통 규칙을 두고, 더 깊은 서브디렉토리 `AGENTS.md`에 모듈 전용 규칙을 두면 돼.

우선순위는 가까운 쪽이 이겨. 홈 디렉토리보다 레포 루트가 세고, 레포 루트보다 서브디렉토리가 더 세.

그래서 보통 이렇게 나눠:

| 위치 | 뭘 넣냐 |
|------|---------|
| `~/.codex/AGENTS.md` | 응답 포맷, 기본 검증 습관, 커밋 메시지 취향 |
| `<repo>/AGENTS.md` | 아키텍처 규칙, 패키지 경계, 실행 커맨드 |
| `<repo>/apps/foo/AGENTS.md` | 모듈별 네이밍, 테스트 방식, 파일 배치 |

이 구조가 좋은 이유는, 같은 사람이어도 레포마다 작업법이 다르고 같은 레포 안에서도 서브시스템마다 룰이 다르기 때문이야.

여기서 실전 팁이 바로 하나 나와. 프롬프트에 규칙을 매번 복붙하지 마. 그건 토큰도 먹고, 세션마다 문구가 조금씩 바뀌어서 일관성도 깨져. 차라리 AGENTS.md에 한 번만 적고, 프롬프트에는 이번 작업 범위와 완료 조건만 넣어.

예를 들면 이런 식이 좋아:

```text
AGENTS.md 기준으로 작업해.
이번엔 packages/worker 범위만 보면 돼.
완료 기준은 pnpm test, 실패 시 원인까지 적어.
```

이렇게 두면 규칙은 문서가 들고 가고, 프롬프트는 이번 태스크의 경계만 들고 가. 훨씬 덜 흔들려.

특히 빌드와 테스트 커맨드는 꼭 넣어. 에이전트가 스스로 검증까지 밀어붙이려면 "무엇을 실행해 봐야 끝인지"가 문서에 있어야 해. 이 한 줄이 없으면 "아마 이걸 돌리면 되겠지" 모드로 가고, 있으면 "문서에 적힌 검증을 끝까지 실행" 모드로 바뀌어.

예를 들면:

```md
## Verification
- 루트에서 `pnpm lint`
- API 패키지는 `cd core/packages/worker && pnpm build`
- CLI 패키지는 `cd core/packages/cli && pnpm test -- path/to/file.test.ts`
- 테스트가 없으면 최소 `pnpm build`까지 확인
```

실무에선 이 차이가 꽤 커. 문서에 커맨드가 없으면 에이전트가 추정하고, 문서에 있으면 에이전트가 실행하고, 실패해도 어디서 막혔는지 더 정확히 적어.

레퍼런스 코드도 강해. 커뮤니티에서 자주 나오는 팁인데, 잘 짜인 사내 샘플이나 과거 베스트 구현을 워크스페이스 안에 같이 두고, AGENTS.md에서 참조 경로를 찍어주면 패턴 적응 속도가 빨라져.

예를 들면:

```md
## Reference Implementations
- API 에러 응답 패턴: `references/api-error-pattern/`
- Vue 폼 검증 패턴: `references/form-validation/`
- Worker auth 라우트 예시: `references/worker-auth/`
```

이건 "복붙해"가 아니라 "이 레포가 좋아하는 패턴은 여기 있어"를 알려주는 방식이야. 새 에이전트가 코드베이스를 넓게 헤집지 않아도 되고, 첫 수정부터 팀 취향에 더 가깝게 들어와.

실전에서 문서에 꼭 있어야 하는 항목도 정리해 두면 좋아:

| 꼭 넣어야 할 것 | 이유 |
|----------------|------|
| 빌드 커맨드 | 변경 직후 뭘 돌릴지 안 헤매 |
| 테스트 커맨드 | 스스로 검증 루프를 닫아 |
| 작업 금지 범위 | 멀쩡한 영역까지 번지는 걸 막아 |
| 참조 코드 경로 | 팀 패턴 적응 속도가 빨라져 |
| 파일 위치 규칙 | 새 파일이 엉뚱한 데 안 생겨 |

반대로 이런 건 프롬프트에 둘 때 더 좋아:

| 프롬프트에 둘 것 | 이유 |
|------------------|------|
| 이번 작업 범위 | 매번 달라져 |
| 완료 기준 | 태스크마다 달라져 |
| 시간 절약용 제약 | 이번엔 문서만, 이번엔 테스트 제외 |
| 급한 우선순위 | 버그 핫픽스인지 리팩터링인지 |

커뮤니티 쪽에선 AGENTS.md 최적화만 다루는 스레드도 계속 올라와. `agentsmd.net` 같은 사이트가 대표적이야. 좋은 사례를 보면 공통점이 있어. 규칙을 선언형으로 쓰기보다 실행형으로 쓰고, 검증 명령을 빠뜨리지 않고, "좋은 코드 작성" 같은 추상어보다 파일 경로와 커맨드를 적고, 금지사항은 짧고 강하게 적어.

짧게 줄이면 이거야. `AGENTS.md = 프롬프트 절약 장치 + 자가 검증 레일 + 팀 취향 압축본`. 사람 공통 습관은 위에, 코드베이스 규칙은 가운데, 모듈 디테일은 아래에 두고, 검증 명령은 반드시 문서 안에 넣어. 그게 제일 덜 꼬여.

## 승인 모드는 어떻게 올려야 안 망하냐?

Codex CLI는 대충 3단계로 생각하면 편해.

| 모드 | 언제 쓰냐 |
|------|-----------|
| `suggest` 감각 | 처음 방향 볼 때 |
| `auto-edit` 감각 | 수정까지 맡길 때 |
| `full-auto` 감각 | CI, 컨테이너, 격리된 작업 공간 |

실제 최신 CLI 도움말 기준으로 `--full-auto`는 `-a on-request`와 `--sandbox workspace-write`를 묶은 편의 플래그야. 그러니까 이름만 보고 "모든 걸 다 뚫는 모드"로 보면 안 맞아. "낮은 마찰로 자동 실행" 쪽에 더 가까워.

중요한 건 이름보다 운영 방식이야. 처음엔 보수적으로 방향만 맞추고, 작업 반경이 보이면 자동 수정 범위를 올리고, 재현 가능한 격리 환경에선 `full-auto`로 넘기는 흐름이 무난해.

비대화형 실행인 `codex exec`도 여기서 자주 붙어. 정확히 말하면 현재 버전에서 `--full-auto`가 자동 기본값은 아니야. 그래도 실전 배치 패턴에선 거의 기본처럼 붙여. CI, 스크립팅, 임시 worktree 처리에 특히 잘 맞아.

```bash
codex exec --full-auto "lint 에러만 고쳐. 테스트 실패면 멈추고 원인 적어."
```

샌드박스 기본값은 느슨하게 풀지 마. 대부분의 경우 `workspace-write`가 제일 무난해. 읽을 건 읽고, 작업 폴더 안에서만 쓰게 두면 실수 반경이 작아져.

다만 macOS에선 실전 gotcha가 하나 있어. Seatbelt 샌드박스 쪽 제약 때문에 `network_access=true`를 줬는데도 실제 네트워크가 막히는 사례가 커뮤니티에서 꽤 보고돼. 이럴 땐 "왜 config가 안 먹지?" 하고 오래 싸우기보다 우회 경로를 빨리 잡는 쪽이 낫다.

보통 이렇게 처리해:

```bash
codex exec --sandbox danger-full-access "패키지 메타데이터 확인 후 버전만 업데이트해"
```

또는 인라인 오버라이드로:

```bash
codex exec -c 'sandbox_mode="danger-full-access"' "원격 API 스키마 확인하고 타입 재생성해"
```

이 우회는 문제를 해결해 주는 대신 샌드박스 경계를 크게 낮춰. 그러니까 신뢰 가능한 레포, 임시 worktree, 짧은 작업 범위에서만 써.

상황별 추천 조합은 대충 이래:

| 상황 | 추천 조합 |
|------|-----------|
| 새 코드베이스 탐색 | 승인 보수적 + 읽기 위주 |
| 작은 수정 반복 | `workspace-write` + 자동 수정 |
| CI 배치 | `codex exec --full-auto` |
| 외부 네트워크 의존 작업 | 별도 worktree + 샌드박스 재검토 |
| 코드 생성 실험 | 임시 브랜치나 worktree에서 먼저 |

보안 쪽도 빼면 안 돼. 2025년에 `CVE-2025-61260` 커맨드 인젝션 이슈가 공개된 적 있어. 이런 류 취약점은 도구 성격상 영향 반경이 커. 운영 원칙은 단순해. Codex CLI는 최신 버전 유지하고, `danger-full-access`는 짧고 격리된 작업에서만 쓰고, 외부 입력을 프롬프트로 그대로 밀어 넣지 말고, 배치 작업은 별도 브랜치나 worktree에 가둬.

자동화 품질은 모델보다 경계 설정에서 더 자주 갈려. 빨리 맡기는 것보다 어디까지 맡길지 선을 잘 긋는 쪽이 오래 이겨.

## 프롬프트는 뭘 박아야 덜 헤매냐?

복붙보다 참조가 먼저야. 파일 내용을 길게 붙이는 대신 `@file`로 파일을 찍어주면 컨텍스트가 훨씬 깔끔해.

예를 들면 이런 식이야:

```text
@packages/worker/src/routes/share.ts
이 파일 기준으로 만료 링크 검증만 추가해.
테스트 통과하면 끝이야.
다른 라우트는 건드리지 마.
```

복잡한 작업이면 바로 코딩부터 시키지 말고 먼저 계획을 세우게 해. 큰 작업을 바로 손대기 전에 작업 순서, 위험 구간, 검증 포인트부터 맞추는 게 훨씬 덜 새.

디자인이나 UI 작업도 텍스트만 고집할 필요 없어. 스크린샷이나 시안 이미지를 붙이면 Codex가 그걸 보고 구조랑 스타일을 코드로 옮겨.

그리고 제약조건이랑 완료 기준은 꼭 같이 줘. 이 둘이 없으면 세션이 길어질수록 "대충 좋아 보이는 쪽"으로 미끄러져.

잘 먹히는 문장 패턴은 이거야:

| 뭘 적냐 | 예시 |
|---------|------|
| 범위 | `로그인 폼 파일만 수정해` |
| 금지 | `API 응답 스키마는 바꾸지 마` |
| 기준 | `pnpm test 통과하면 끝` |
| 참조 | `@src/store/auth.ts 기준으로 맞춰` |

좋은 프롬프트는 길어서 좋은 게 아니고, 의사결정 순서가 보여서 좋아. 보통 `범위 -> 금지 -> 완료 기준 -> 참조 파일 -> 실패 시 행동` 순서가 잘 먹혀.

예를 들면:

```text
@packages/viewer/src/components/LoginForm.vue
로그인 폼 파일만 수정해.
API 응답 스키마는 바꾸지 마.
완료 기준은 pnpm build 통과야.
기존 검증 패턴은 @references/form-validation 를 따라.
막히면 추측 구현하지 말고 차단 원인부터 적어.
```

실전에서 특히 자주 쓰는 게 배치 패턴이야. `codex exec`를 여러 개 병렬로 던져 놓고, 끝난 것만 리뷰하는 방식이 의외로 생산성이 좋아.

가장 단순한 형태는 이런 식이야:

```bash
for task in "fix lint" "add tests" "update docs"; do
  codex exec --full-auto "$task" &
done
wait
```

이건 작은 태스크 여러 개를 빨리 돌릴 때 좋아. 다만 같은 파일을 건드릴 확률이 낮아야 해. 겹치면 이득보다 병합 비용이 더 커져.

그래서 병렬 배치를 걸기 전엔 태스크를 이렇게 나누는 편이 안전해:

| 안전한 분할 | 위험한 분할 |
|-------------|-------------|
| 테스트 추가 / 문서 수정 / 린트 정리 | 같은 서비스 파일을 둘이 동시에 수정 |
| 패키지별 작업 분리 | 같은 디렉토리 구조를 동시에 개편 |
| 읽기 중심 분석 + 구현 분리 | 스키마 변경 + UI 마이그레이션 동시 진행 |

같은 레포에서 여러 에이전트를 굴릴 땐 Git worktree 격리가 거의 필수야. 한 브랜치에서 여러 세션을 동시에 돌리면 수정 충돌보다 더 무서운 게 맥락 충돌이야.

예를 들면:

```bash
git worktree add ../mdshare-lint fix/lint
git worktree add ../mdshare-tests fix/tests
git worktree add ../mdshare-docs docs/codex-tips
```

그다음 각 worktree에서 따로 돌려:

```bash
cd ../mdshare-lint && codex exec --full-auto "lint 에러만 고쳐"
cd ../mdshare-tests && codex exec --full-auto "CLI 테스트 보강"
cd ../mdshare-docs && codex exec --full-auto "문서 확장"
```

이렇게 하면 충돌 반경이 확 줄어. 작업이 끝난 뒤에도 diff를 비교하기 쉬워서 리뷰가 빨라져.

실무에서 꽤 자주 나오는 패턴이 queue-and-review야. WorkOS 팀이 공유한 운영 감각도 이 흐름과 비슷해. 지금 바로 판단이 필요한 일만 사람 손에 두고, 나머지 3개에서 5개 태스크는 `codex exec`로 큐잉해 두고, 기다리는 동안 다른 업무를 보고, 돌아와서 결과물만 리뷰하는 방식이야.

이 흐름이 먹히는 이유는 "생성 시간"을 사람의 대기 시간이 아니라 "백그라운드 처리 시간"으로 바꿔버리기 때문이야.

좋은 배치 프롬프트는 욕심을 덜 부려. 하나의 에이전트에게 "버그도 고치고 테스트도 추가하고 문서도 업데이트하고 PR 요약도 적어" 이렇게 던지면 중간 품질이 흔들려. 차라리 `1) lint만 고쳐 2) 실패 테스트만 복구해 3) 변경된 동작만 문서화해 4) 마지막에 diff를 5줄로 요약해`처럼 쪼개.

프롬프트가 짧아도 되는 이유는 레이어드 규칙을 AGENTS.md가 이미 들고 있기 때문이야. 짧게 말하면, Codex는 마음을 읽는 쪽보다 경계를 읽는 쪽이 더 강해. "어디까지", "뭘 보면", "뭘 끝으로 볼지"를 먼저 박아두는 게 제일 잘 먹혀.

## 서브에이전트는 어떻게 굴려야 이득이 남냐?

Codex는 필요한 순간에 서브에이전트를 자동으로 스폰할 수 있어. 여기서 중요한 건 "많이"보다 "역할 분리"야.

탐색용 에이전트는 코드베이스를 읽고 패턴을 찾게 두고, 구현용 에이전트는 실제 수정과 검증에 집중하게 두면 충돌이 확 줄어. 리뷰도 따로 빼면 더 좋고, 구현 끝난 뒤 코드 리뷰 전용 에이전트를 하나 더 붙이면 작성한 흐름이랑 검수 시선이 분리돼서 놓친 회귀를 더 잘 잡아.

재귀 위임을 깊게 여는 건 보통 손해가 커. 일 시키는 에이전트가 또 에이전트를 부르고, 걔가 또 부르는 식으로 토큰이 빠지고, 사람은 누가 최신 맥락을 들고 있는지 추적하느라 지쳐.

보통은 이 정도면 충분해:

| 역할 | 맡길 일 |
|------|---------|
| 탐색용 | 관련 파일 찾기, 패턴 요약, 영향 범위 정리 |
| 구현용 | 코드 수정, 테스트 실행, 에러 수습 |
| 리뷰용 | 회귀 검사, 누락 케이스 확인, 설명 보강 |

보통 실전에선 여기서 한 단계만 더 가면 돼. 서브에이전트를 쓰려면 먼저 `config.toml`에서 기능을 켜:

```toml
[features]
multi_agent = true

[agents]
max_threads = 6
max_depth = 1
```

`max_depth = 1`이 기본값이야. 즉 자식 에이전트까지는 되는데, 손자 에이전트는 안 돼. 대부분의 작업은 이걸로 충분해. 깊이를 더 열면 구조가 똑똑해지는 게 아니라 누가 누구한테 뭘 맡겼는지 사람이 추적하기 어려워져.

역할을 자주 나눌 거면 프롬프트마다 즉흥적으로 설명하지 말고, `.codex/agents/`에 커스텀 에이전트를 미리 정의해두는 게 좋아:

```toml
# .codex/agents/reviewer.toml
name = "reviewer"
description = "PR 리뷰 전문. 정확성, 보안, 테스트 누락에 집중."
developer_instructions = """
리뷰할 때 이 순서로 봐:
1. 동작 버그
2. 보안 취약점
3. 회귀 위험
4. 테스트 커버리지 누락
파일명:줄번호 형태로 찾은 것을 정리해.
"""
```

```toml
# .codex/agents/debugger.toml
name = "debugger"
description = "실패 재현하고 최소 수정안 제시."
developer_instructions = """
1. 실패 테스트나 에러 로그를 읽어.
2. 콜스택을 따라가며 근본 원인을 추적해.
3. 가장 작은 수정안을 제시해.
4. 다른 테스트가 깨지지 않는지 확인해.
"""
```

글로벌 설정은 `~/.codex/agents/`, 프로젝트 전용 설정은 `.codex/agents/`에 두면 돼. 둘 다 있으면 프로젝트 쪽이 우선순위가 더 높아. 팀 저장소에서 일할 땐 보통 프로젝트 쪽에 넣어두는 편이 재현성이 좋아.

중요한 점 하나 더. 서브에이전트는 자동 발동 마법이 아니야. "필요하면 알아서 나눠"보다 "어떤 역할로 몇 개를 어떻게 굴릴지"를 프롬프트에 박아주는 쪽이 훨씬 잘 먹혀.

병렬 리뷰를 시키고 싶으면 이렇게 써:

```text
이 PR을 리뷰해줘. 에이전트를 하나씩 스폰해서 병렬로 봐:
1. 동작 버그가 있냐
2. 보안 취약점이 있냐
3. 테스트 커버리지가 빠진 곳이 있냐
셋 다 끝나면 카테고리별로 요약해줘.
```

코드베이스 탐색은 이런 식이 좋아:

```text
인증 흐름이 어떻게 돼 있는지 파악해줘.
에이전트를 병렬로 스폰해서 이 영역을 동시에 봐:
1. 인증 미들웨어와 라우트 가드
2. 토큰 생성과 검증
3. 세션 관리
전부 끝나면 아키텍처 요약을 하나로 합쳐줘.
```

버그 수정은 아예 역할 이름까지 박아두면 더 안정적이야:

```text
설정 모달 저장이 안 되는 문제를 조사해줘.
debugger 에이전트로 재현하고, reviewer로 원인 코드를 추적하고,
원인이 확인되면 최소 수정안을 제시해.
```

핵심은 프롬프트에 아래 3가지를 빠뜨리지 않는 거야:

| 명시할 것 | 예시 |
|-----------|------|
| 병렬인지 순차인지 | "에이전트 3개 병렬로 스폰해" |
| 끝나면 뭘 하냐 | "전부 끝나면 요약해줘" |
| 쓰기 범위 | "src/api만 수정해. 다른 건 읽기만" |

이 셋이 빠지면 에이전트는 열심히 일해도 결과를 합치는 쪽에서 사람이 다시 고생해. 반대로 이 셋만 명확하면 프롬프트가 길지 않아도 돼.

세션 안에서 `/agent`를 치면 현재 활성 에이전트 스레드 목록이 나와. 여기서 스레드 간 전환하면서 각 에이전트가 뭘 보고 있는지, 어디까지 끝냈는지 추적할 수 있어. 병렬 작업을 걸어놓고 메인 스레드에서 계속 일할 때 꽤 유용해.

파일이 여러 개인 리뷰 작업이라면 CSV 배치 패턴도 강해. 예를 들어 이런 CSV를 만들어:

```csv
path,owner
src/components/Button.tsx,frontend
src/components/Modal.tsx,frontend
src/components/Form.tsx,frontend
```

그다음 프롬프트를 이렇게 줘:

```text
/tmp/components.csv의 각 행마다 에이전트를 스폰해서 리뷰해줘.
각 {path}를 리뷰하고, 위험도와 요약을 JSON으로 돌려줘.
결과를 /tmp/review-results.csv로 내보내줘.
```

이 패턴이 좋은 이유는 "파일 목록 분배"와 "결과 수집 포맷"을 사람이 미리 고정해둘 수 있기 때문이야. 큰 PR에서 컴포넌트, 라우트, 테스트 파일을 owner별로 쪼개 던지면 수작업 triage 시간이 꽤 줄어.

병렬화는 강력한데, 같은 파일을 여러 에이전트가 동시에 잡게만 두지 마. 속도보다 충돌 비용이 더 크게 나와.

여기서 커뮤니티 쪽 확장판도 하나 알아둘 만해. `codex-subagents-mcp` 같은 프로젝트는 MCP 서버를 통해 reviewer, debugger, security 성격의 서브에이전트를 스폰하는 패턴을 제공해.

이 계열 도구가 재밌는 이유는 서브에이전트마다 temp workdir를 따로 잡고, 전용 AGENTS.md를 얹어서 격리 실행하는 흐름이 많기 때문이야. 즉, "서브에이전트"를 단순한 대화 분기가 아니라 "작은 독립 작업 공간"으로 보는 거야.

이 접근이 특히 좋은 상황은 이런 때야:

| 상황 | 왜 분리하냐 |
|------|-------------|
| 보안 검토 | 구현 에이전트와 다른 관점이 필요해 |
| 디버깅 | 실패 로그와 재현 실험을 따로 밀 수 있어 |
| 코드 리뷰 | 작성자 시선과 검수 시선을 분리해 |
| 문서화 | 구현 끝난 뒤 diff 기반 요약만 따로 맡겨 |

커뮤니티에선 이런 병렬 작업을 통째로 `Agentmaxxing`이라고 부르기도 해. 여러 에이전트를 동시에 굴리면서 사람은 큐 관리자처럼 움직이는 패턴이야.

이걸 흉내 낼 때 가장 먼저 지켜야 하는 건 "쓰기 권한이 겹치지 않게 자르기"야. 잘못 자르면 A가 타입 정의를 바꾸고, B가 그걸 모른 채 구현을 덮고, C 리뷰어가 이미 오래된 diff를 보고 코멘트를 달고, 사람은 셋 다 읽고 다시 수습하게 돼.

좋게 자르면 흐름이 훨씬 단순해져. 탐색 에이전트가 관련 파일과 패턴을 요약하고, 구현 에이전트가 한 패키지 안에서만 수정하고, 리뷰 에이전트가 diff와 테스트 결과만 보고, 사람은 merge 여부만 결정해.

서브에이전트 전략을 한 장으로 줄이면 이거야:

| 규칙 | 이유 |
|------|------|
| 역할별로 나눠 | 한 에이전트가 다 하려다 품질이 흐려져 |
| 쓰기 범위를 분리해 | 충돌 비용이 급격히 줄어 |
| 리뷰는 따로 둬 | 작성자 편향을 줄여 |
| 깊이는 낮게 둬 | 토큰과 맥락 낭비를 막아 |
| workdir를 격리해 | 병렬성이 실제로 살아 |

"에이전트를 많이 쓴다"보다 "에이전트마다 책임 경계를 좁힌다"가 핵심이야.

## 컨텍스트는 언제 압축해야 덜 늦냐?

길어지면 `/compact`를 먼저 떠올리면 돼. 이건 그냥 청소가 아니라, 남길 핵심만 추려서 다음 턴 효율을 살리는 작업이야.

대화가 거의 한계까지 찬 다음에 누르는 건 늦어. 이미 노이즈가 쌓였고, 모델도 앞쪽 맥락에 덜 민감해져 있을 가능성이 커.

좋은 타이밍은 이런 때야:

1. 큰 기능 하나를 끝냈을 때
2. 탐색 단계에서 구현 단계로 넘어갈 때
3. 실패한 시도 둘셋을 정리하고 다시 들어갈 때

자동 압축은 기대하지 마. 길어지는 세션일수록 사람이 먼저 정리 포인트를 잡아주는 편이 낫다.

핵심은 "터지면 압축"이 아니라 "터지기 전에 정리"야. 큰 기능 하나 끝날 때 한 번, 스테이지 전환할 때 한 번, 막혔을 때 한 번. 이 리듬만 잡아도 체감이 꽤 달라져.

압축할 때 남겨야 할 건 세 가지면 충분해:

| 꼭 남길 것 | 이유 |
|-----------|------|
| 현재 목표 | 다음 턴이 흔들리지 않게 |
| 이미 바꾼 것 | 중복 작업 막으려고 |
| 아직 안 풀린 것 | 다음 액션 바로 이어가려고 |

실전에서 잘 먹히는 압축 문장도 있어:

```text
현재 목표: viewer 로그인 에러 상태 정리
이미 바꾼 것: store 응답 타입 통일, 에러 토스트 공통화
남은 것: 모바일 레이아웃 깨짐, build 확인
금지: API 스키마 변경 금지
```

반대로 안 좋은 압축은 이런 쪽이야. 대화 내용을 줄글 요약으로만 남기고, 파일 경로나 커맨드가 없고, 지금 뭘 해야 하는지가 안 보이고, 실패한 시도가 왜 실패했는지가 빠진 경우야.

컨텍스트 관리는 기술이라기보다 운영 습관에 가까워. 좋은 타이밍에 먼저 정리해 두면 긴 세션이 길어질수록 이득이 더 커져.

## oh-my-codex는 어디에 끼우면 맛이 사냐?

[oh-my-codex](https://github.com/Yeachan-Heo/oh-my-codex), 줄여서 OMX는 Codex CLI 위에 올리는 멀티 에이전트 레이어야. 혼자 한 세션을 오래 끄는 대신, 역할이 갈린 팀처럼 굴리게 도와줘.

강점은 두 갈래야. 하나는 이미 짜인 운영 자산이고, 다른 하나는 팀 모드야. 처음부터 운영 패턴을 다 짤 필요 없이, 필요한 스킬과 흐름을 바로 꺼내 쓰기 편해.

설치 흐름은 짧아:

```bash
npm install -g oh-my-codex
omx setup
omx doctor
```

여기서 많이 쓰는 건 팀 모드와 HUD야:

```bash
omx team 6:executor "리팩토링"
omx hud
```

그런데 OMX는 "스킬 이름을 안다" 수준에서 끝내면 별로 체감이 안 커. 실전은 티어를 나눠서 익히는 쪽이 훨씬 좋아.

### Tier 1: 거의 매일 쓰는 스킬

| 스킬 | 언제 쓰냐 | 왜 많이 쓰냐 |
|------|-----------|--------------|
| `$ralph` | 끝까지 완료 보장을 걸고 싶을 때 | 구현, 검증, 리뷰, 정리까지 루프를 닫아 |
| `$team` | 병렬 팀을 열고 싶을 때 | 역할 분리와 분업 시작점이야 |
| `$plan` | 큰 작업 들어가기 전에 | 전략 분해가 빨라져 |
| `$autopilot` | 반복 실행을 길게 맡길 때 | 6단계 자율 실행 흐름을 바로 켜 |

이 중 핵심은 `$ralph`야. 단순히 "코드 짜는 스킬"이 아니라 "완료될 때까지 루프를 돌리는 운영자"에 가까워. 구현하고, 검증하고, 아키텍트 리뷰를 받고, deslop 하고, 남은 오류를 다시 잡고, 끝날 때까지 재시도하는 흐름이라서 막연한 태스크를 맡길수록 차이가 커.

### Tier 2: 품질 올릴 때 꺼내는 스킬

| 스킬 | 언제 켜냐 | 포인트 |
|------|-----------|--------|
| `$ai-slop-cleaner` | AI 냄새 나는 코드 정리할 때 | 불필요한 추상화, 장황한 헬퍼를 줄여 |
| `$deep-interview` | 요구사항이 애매할 때 | 소크라틱 질문으로 스펙을 좁혀 |
| `$visual-verdict` | UI 검토할 때 | 스크린샷 비교로 눈속임을 덜 타 |

실전 패턴은 더 중요해. 아래 네 개만 익혀도 OMX 체감이 확 올라가.

### 패턴 A: 막연한 아이디어를 코드로 바꿀 때

```text
$deep-interview -> $plan --consensus -> $autopilot
```

아이디어 단계에선 바로 구현보다 질문이 먼저여야 해. `$deep-interview`가 요구사항을 압축해 주고, `$plan --consensus`가 실행 전략을 합의 형태로 정리하고, `$autopilot`이 실제 작업을 밀어.

이 패턴은 기능 아이디어만 있고 스펙이 없을 때, 이해관계자가 여러 명일 때, 작업량이 커서 구현 전에 순서를 고정해야 할 때 특히 좋아.

### 패턴 B: 확실한 작업을 완료 보장으로 밀 때

```text
$ralph "API 라우트 전부 v2로 마이그레이션"
```

이건 "무엇을 만들지"보다 "반드시 끝까지 밀어야 해"가 중요한 작업에서 잘 먹혀. 대규모 린트 정리, API 버전 마이그레이션, 테스트 전면 보강 같은 태스크에 잘 맞고, 요구사항 탐색처럼 방향을 먼저 넓게 봐야 하는 작업엔 덜 맞아.

### 패턴 C: 병렬 팀 실행

```bash
$plan
omx team 3:executor "작업"
omx team status
omx team shutdown
```

여기서 초보자가 제일 많이 헷갈리는 포인트가 입력 채널이야.

| 입력 | 어디에 치냐 |
|------|-------------|
| `$ralph`, `$team`, `$plan` | Codex 채팅 안 |
| `omx team ...` | 터미널 CLI |

의도는 비슷해도 입력 위치가 달라. `$...`는 세션 안 명령, `omx ...`는 바깥 운영 명령이라고 기억하면 덜 헷갈려.

병렬 팀 모드는 강하지만 tmux 의존성이 커서 여기서 삐끗하는 경우가 제일 많아. 커뮤니티에서도 릴리스 절반이 tmux 핫픽스 같다는 농담이 괜히 나오는 게 아니야. 그래서 팀 모드를 쓸 땐 종료 습관이 중요해. 작업 끝나면 꼭 `omx team shutdown`으로 내려.

### 패턴 D: 조사와 분석

```text
$analyze
```

좋은 분석 출력은 `가설 -> 증거 -> file:line 레퍼런스 -> 다음 액션` 순서를 닮아. "느낌상 여기 문제 같아"가 아니라 "이 파일 몇 줄 근처가 왜 의심되는지"를 뽑아야 해.

### 세션 복구도 꽤 강해

OMX는 `.omx/` 디렉토리에 세션 상태를 꽤 많이 남겨. 대표적으로 `.omx/state/`, `.omx/project-memory.json`, `.omx/logs/` 같은 파일이 있어. 그래서 중단된 세션을 다시 붙일 때 완전히 처음부터 설명하지 않아도 돼.

```bash
omx resume
```

장시간 작업, 터미널 종료, tmux 충돌, 일시적 rate limit 뒤에 이어붙일 때 특히 좋아.

### 멀티 프로바이더 팀도 가능해

한 팀 안에 Codex, Claude, Gemini 워커를 섞는 패턴도 있어:

```bash
OMX_TEAM_WORKER_CLI_MAP=codex,claude,gemini omx team 3:executor "작업"
```

이런 조합은 "빠른 탐색은 Gemini, 긴 코드 수선은 Codex, 리뷰는 Claude"처럼 역할 배치를 다르게 줄 때 재미가 나. 다만 운영 난도는 확 올라가. 모델마다 맥락 유지 방식, 검증 습관, 파일 수정 스타일이 달라서 팀 리드 역할을 하는 사람이 공통 AGENTS.md를 더 빡빡하게 적는 편이 좋아.

### 알림 연동도 걸어두면 편해

장시간 작업이면 결과 기다리는 시간이 제일 아깝다. OMX는 Telegram, Discord, Slack 웹훅 연동도 지원해.

```bash
export OMX_TELEGRAM_BOT_TOKEN=xxx
export OMX_DISCORD_WEBHOOK_URL=xxx
```

이런 식으로 걸어두면 팀 작업 끝났을 때 바로 알림이 와. queue-and-review 패턴이 더 잘 굴러가.

### 함정도 몇 개 알고 가

| 함정 | 대응 |
|------|------|
| tmux 의존성 | 팀 모드 후 `omx team shutdown` 습관 들여 |
| Ralph 무한 루프 | 이상하면 `omx cancel`로 끊어 |
| macOS Intel 경고 | `syspolicyd` 팝업 가능성 감안해 |
| 메이저 업데이트 호환성 | 깨지면 `omx setup --force`로 재설치해 |

특히 `$ralph`는 강력한 만큼 루프가 길어질 수도 있어. 완료 기준이 애매하면 더 그래. 그래서 `$ralph`를 쓸 땐 "끝의 정의"를 꼭 같이 적어.

예를 들면:

```text
$ralph "worker의 공유 링크 삭제 API 추가. 완료 기준은 타입 체크와 통합 테스트 통과."
```

### oh-my-claudecode도 같은 계열이야

같은 저자가 만든 Claude Code 버전도 있어. 거긴 더 많은 에이전트와 스킬을 앞세우는 쪽으로 가고, 자동 재시작처럼 "rate limit가 풀릴 때까지 기다렸다가 다시 이어가는" 운영 기능도 강조해.

짧게 요약하면, OMX는 Codex를 대체하는 도구가 아니야. Codex 위에 팀 운영 레이어를 얹는 도구야. 혼자 한 세션을 오래 끄는 방식보다 병렬 큐, 역할 분리, 세션 복구, 알림 연동이 중요한 순간에 진가가 나.

## config.toml은 뭘 먼저 만져야 체감이 오냐?

제일 먼저 보는 건 모델과 프로필이야. 매번 세션 시작할 때 "이번엔 뭘로 돌리지?"를 다시 고민하면 운영이 느려져. 프로필을 나눠두면 이 판단을 미리 고정할 수 있어.

예를 들면:

```toml
[profiles.fast]
model = "gpt-5.4"
reasoning = "medium"

[profiles.deep]
model = "gpt-5.3-codex"
reasoning = "high"
```

그다음 호출은 이렇게 가:

```bash
codex --profile fast "빠른 리팩토링"
codex exec --profile deep --full-auto "아키텍처 영향 포함해서 마이그레이션"
```

프로필 설계는 보통 세 부류면 충분해:

| 프로필 | 용도 |
|--------|------|
| `fast` | 반복 수정, 문서 편집, 린트 정리 |
| `deep` | 구조 변경, 대규모 리팩터링, 애매한 설계 판단 |
| `review` | 코드 리뷰, diff 분석, 회귀 확인 |

비용 팁도 중요해. 루틴 작업까지 무조건 `high`로 올리면 비용만 늘고 체감은 생각보다 적어. 대충 이 기준이 무난해:

| 작업 | reasoning |
|------|-----------|
| 문서 수정 | `medium` |
| 테스트 보강 | `medium` |
| 린트 / 타입 에러 수선 | `medium` |
| 설계 갈림길이 많은 구조 변경 | `high` |
| 아키텍처 리뷰 | `high` |

즉, 복잡한 판단이 필요한 순간만 `high`, 나머진 `medium`이 실속 있어.

최신 CLI 기준으로는 CLI 플래그가 가장 먼저 이겨. 그다음 프로필, 그다음 `config.toml` 최상위, 마지막이 기본값이야.

| 우선순위 | 어디서 오냐 |
|----------|-------------|
| 1 | CLI 플래그 |
| 2 | `--profile`로 고른 프로필 값 |
| 3 | `config.toml` 최상위 기본 설정 |
| 4 | CLI 내장 기본값 |

이걸 알아두면 "왜 이번 세션은 설정이 다르게 먹지?" 같은 혼란이 확 줄어.

특히 `-c` 인라인 오버라이드는 실전에서 자주 써:

```bash
codex exec --profile fast -c 'model="gpt-5.3-codex"' "테스트 실패 원인만 분석해"
```

이건 "평소엔 fast 프로필 쓰는데, 이번 한 번만 모델을 바꾸고 싶다" 같은 순간에 깔끔해.

커스텀 모델 프로바이더를 붙이는 흐름도 있어. OpenAI 호환 API를 제공하는 프록시나 게이트웨이를 물려서 조직 내부 라우팅 정책을 태우는 방식이야. 이 패턴은 사내 비용 통제를 한 군데서 하고 싶거나, 로깅과 라우팅을 중앙화하고 싶거나, 모델 선택 정책을 레포별로 강제하고 싶은 팀에 잘 맞아.

MCP 서버 설정도 `config.toml` 쪽에 고정해 두면 편해. 세션마다 다시 연결하지 않아도 되고, 프로필별로 켜고 끌 수도 있어.

예를 들면 운영 감각은 이래:

| 시나리오 | config 전략 |
|----------|-------------|
| 일반 앱 개발 | `fast` 기본 프로필 |
| 분석 중심 세션 | `deep` + 검색/MCP 강화 |
| 리뷰 세션 | 읽기 위주 프로필 + review용 모델 |
| CI 배치 | `exec` 전용 프로필 + 엄격한 출력 스키마 |

로컬에서 바로 체감 오는 항목은 보통 네 개야. 기본 모델, reasoning 강도, 프로필 분리, 자주 쓰는 MCP 연결. 반대로 처음부터 너무 세세하게 만질 필요는 없어. 프로필 두세 개만 만들어도 운영 체감이 꽤 커.

실전 예시를 하나 더 적어볼게:

```toml
model = "gpt-5.4"
reasoning = "medium"

[profiles.fast]
model = "gpt-5.4"
reasoning = "medium"

[profiles.deep]
model = "gpt-5.3-codex"
reasoning = "high"

[profiles.review]
model = "gpt-5.4"
reasoning = "high"
approval_policy = "never"
```

이런 식으로 두면 평소엔 그냥 `codex`, 깊은 작업은 `codex --profile deep`, 배치 리뷰는 `codex exec --profile review`처럼 분기하기 쉬워.

결국 `config.toml`은 취향 파일이 아니라 운영 파일이야. 세션 시작할 때마다 같은 판단을 반복하지 않게 고정하는 데 의미가 있어.

## 한 줄 정리

Codex CLI는 "잘 말하면 잘 짜는 도구"라기보다, "룰을 AGENTS.md에 계층으로 깔고, 승인과 샌드박스를 상황별로 올리고, 프롬프트는 짧고 경계 중심으로 쓰고, 서브에이전트와 OMX는 역할 분리와 병렬 큐에 쓰고, 길어지기 전에 컨텍스트를 압축하고, config.toml로 운영 습관을 고정할 때" 제일 잘 굴러가.
