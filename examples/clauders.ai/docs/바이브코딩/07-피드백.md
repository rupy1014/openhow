**배포했으면 끝? 아니야. 피드백 받아야 진짜 시작이야.**

URL이 생겼어. 이제 사람들한테 보여줘야 해.

---

## 왜 피드백이야?

혼자 만들면 "이게 좋을 거야"라고 착각해.

실제로 보여주면 "이거 왜 이래?"가 나와.

6개월 만들고 아무도 안 쓰는 것보다, 1주 만에 틀린 방향을 아는 게 낫잖아.

---

## 5명이면 돼

```
❌ Google Analytics 설치 → 대시보드 분석
✅ 5명한테 보여주고 → 직접 물어보기
```

분석 도구 필요 없어. 5명한테 직접 물어보는 게 100배 정확해.

---

## 어떻게 물어봐?

:::steps

### URL을 보내

```
"나 이거 만들었어. 1분만 써보고 솔직하게 말해줘."
https://my-app.vercel.app
```

**"솔직하게"를 꼭 넣어.** 안 넣으면 "좋아~"만 나와.

친구들은 기분 안 상하게 하려고 칭찬만 해. "솔직하게"가 있어야 진짜가 나와.

### 구체적으로 물어봐

```
"써보니까 어땠어? 불편한 거 없었어?"
"중간에 그만두고 싶었던 적 있어?"
"돈 내고 쓸 것 같아? 솔직하게."
"뭐가 제일 불편했어?"
```

그냥 "어때?"라고 물으면 진짜 피드백이 안 나와.

### 기록하기

```
- 지현: "캐릭터 귀여웠어. 근데 결제 화면에서 나갔어"
- 민수: "입력이 좀 귀찮았어"
- 수진: "끝까지 했어. 근데 가격이 비싸"
```

Claude한테 정리시켜도 돼:

```text:Claude 프롬프트
이 피드백들 정리해줘.
공통된 불만이 뭔지, 다음에 뭘 고쳐야 하는지 포함해서.
코드는 짜지 마.
```

:::

---

## 가설이 맞았어?

아이디어 단계에서 세운 핵심 가정을 확인해.

```
✅ 맞았다 → 다음 기능 추가하거나 마케팅
❌ 틀렸다 → 방향 전환 (피벗)
🔶 애매하다 → 사람을 더 모아서 테스트
```

---

## 틀려도 괜찮아

방향 전환(피벗)은 실패가 아니야.

```
6개월 만들고 틀린 거 알면? → 6개월 날림
1주 만들고 틀린 거 알면? → 1주만 날리고 방향 수정
```

바이브코딩으로 만들었으니까 잃는 건 시간뿐이야.

그 시간에 배운 건 남아.

---

## 고치고 다시 돌려

피드백 반영해서 고치고, 다시 배포하고, 다시 물어봐.

```
만들기 → 배포 → 피드백 → 고치기 → 다시 배포 → 다시 피드백
```

이 사이클을 빨리 돌릴수록 좋은 서비스가 돼.

---

## 한 줄 정리

5명한테 "솔직하게" 물어보고, 맞으면 키우고, 틀리면 방향을 바꿔.
