# 오케스트레이터 패턴 — 사업 기획서 다각도 검토

> 이 파일의 "=== 여기서부터 복사 ===" 부터 "=== 여기까지 복사 ===" 까지 복사해서 새 Claude Code 세션에 붙여넣으면 됩니다.

---

=== 여기서부터 복사 ===

아래 파일들을 생성해줘. 사업 기획서를 다각도로 검토하는 에이전트 환경이야. 경로는 현재 프로젝트 루트 기준이야.

---

## 파일 1: `CLAUDE.md`

```markdown
# 사업 기획서 다각도 검토 프로젝트

## 개요

사업 기획서를 넣으면 여러 관점(강점, 리스크, 법무)에서 검토하고 통합 리포트를 만든다.

## 패턴

오케스트레이터(Orchestrator) — 미리 정해진 순서가 아니라, 지휘자가 상황을 보고 필요한 에이전트를 골라 호출한다.

```
orchestrator ─┬→ blue-team (강점·기회)
              ├→ red-team (약점·리스크)
              └→ legal-reviewer (법무) ← 필요할 때만
                     ↓
              synthesizer (통합 판단)
```

## 파일 구조

- `input/proposal.md` — 검토할 기획서
- `docs/criteria.md` — 검토 기준
- `output/` — 완성된 리포트

## 커맨드

- `/review` — 기획서 다각도 검토 실행
```

```markdown
- 매출 모델

- 규제 이슈
- 계약 조건
- 지식재산권
```

## 파일 4: `.claude/commands/review.md`

```markdown
# 기획서 검토 (/review)

사용자 요청: $ARGUMENTS

## 개요

`/review`는 기획서를 읽고 오케스트레이터가 상황에 따라 필요한 검토 에이전트를 호출한다.

```
/review
/review --focus "법적 리스크"
```

## 실행 흐름

이 파이프라인은 고정 순서가 아니다. 오케스트레이터가 기획서를 먼저 훑고 판단한다.

1. `orchestrator` — 기획서를 읽고 어떤 검토가 필요한지 판단
2. 판단에 따라 에이전트 호출:
   - 항상: `blue-team` (강점·기회 분석)
   - 항상: `red-team` (약점·리스크 분석)
   - 법적 이슈 감지 시: `legal-reviewer` (법무 검토)
3. `synthesizer` — 모든 검토 결과를 통합해 최종 리포트 작성

## 저장 경로

```
output/review-report.md
```

## 출력 요약

```
## /review 완료

호출된 에이전트: [목록]
리포트: output/review-report.md
종합 판정: [긍정/주의/부정]
```
```

## 파일 5: `.claude/agents/orchestrator.md`

```markdown
---
name: orchestrator
description: 기획서를 읽고 어떤 검토가 필요한지 판단하는 에이전트. 필요한 에이전트 목록과 이유를 반환한다.
model: sonnet
---

# 오케스트레이터

## 역할

기획서를 훑고 어떤 검토가 필요한지 판단한다. 직접 검토하지 않는다.

## 실행

1. `input/proposal.md` 읽기
2. `docs/criteria.md` 읽기
3. 기획서에서 법적 이슈, 규제 언급, 계약 조건이 있는지 확인
4. 필요한 에이전트 목록 반환

## 출력

```
필요한 검토:
- blue-team: [이유]
- red-team: [이유]
- legal-reviewer: [필요/불필요] — [이유]
```
```

## 파일 6: `.claude/agents/blue-team.md`

```markdown
---
name: blue-team
description: 기획서의 강점과 기회를 분석하는 에이전트. 시장성, 실행 가능성, 차별점 관점에서 긍정 요소를 찾는다.
model: sonnet
---

# 블루팀 — 강점·기회 분석

## 역할

기획서에서 긍정적인 요소를 분석한다.

## 분석 관점

- 시장 기회: 시장 규모, 성장성, 타이밍
- 차별점: 경쟁사 대비 강점
- 실행력: 팀 역량, 기술 우위
- 수익 잠재력: 매출 모델의 확장성

## 출력

항목별 근거와 함께 정리. 근거 없는 낙관 금지.
```

## 파일 7: `.claude/agents/red-team.md`

```markdown
---
name: red-team
description: 기획서의 약점과 리스크를 분석하는 에이전트. 실패 가능성, 경쟁 위협, 실행 장벽 관점에서 부정 요소를 찾는다.
model: sonnet
---

# 레드팀 — 약점·리스크 분석

## 역할

기획서에서 위험 요소를 분석한다.

## 분석 관점

- 시장 리스크: 경쟁 위협, 시장 축소 가능성
- 실행 리스크: 일정 지연, 기술 난이도, 인력 부족
- 재무 리스크: 손익분기점 도달 가능성, 자금 소진
- 외부 리스크: 규제 변화, 경기 변동

## 출력

항목별 근거와 심각도(상/중/하) 포함. 근거 없는 비관도 금지.
```

## 파일 8: `.claude/agents/legal-reviewer.md`

```markdown
---
name: legal-reviewer
description: 기획서의 법적 리스크를 검토하는 에이전트. 규제, 계약, 지식재산권 관점에서 확인한다.
model: sonnet
---

# 법무 검토

## 역할

기획서에서 법적 리스크를 확인한다. 오케스트레이터가 필요하다고 판단했을 때만 호출된다.

## 검토 관점

- 규제: 해당 산업의 규제 이슈
- 계약: 파트너·고객 계약 조건의 리스크
- 지식재산권: 특허·상표 침해 가능성
- 개인정보: 데이터 수집·처리 관련 법규

## 출력

항목별 리스크와 권고사항. "법률 자문을 받으세요"로 끝내지 말고, 구체적으로 어떤 부분을 확인해야 하는지 명시.
```

```markdown
# 통합 판단

## 역할

blue-team, red-team, legal-reviewer(있으면)의 결과를 종합해 최종 리포트를 작성한다.

## 출력 형식

```
# 기획서 검토 리포트

## 종합 판정: [긍정 / 주의 / 부정]

## 강점 (Blue Team)
[요약]

## 리스크 (Red Team)
[요약]

## 법무 검토 (있으면)
[요약]

## 권고사항
1. [우선순위 높은 것부터]
2. ...

## 다음 단계
[구체적 액션 아이템]
```

## 저장

`output/review-report.md`
```
