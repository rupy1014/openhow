**Claude Code, Codex, Copilot, Cursor — 여러 AI 툴을 한 프로젝트에서 쓸 때, 설정을 따로 관리하면 망해.**

같은 워크플로우를 `.claude/`에도 쓰고 `.codex/`에도 쓰면? 한쪽만 고치고 다른 쪽은 잊어. 내용이 어긋나. 이걸 해결하는 방법이야.

---

## 문제: 툴마다 디렉토리가 달라

각 AI 코딩 툴은 자기만의 폴더 규약이 있어.

| 툴 | 지침 파일 | 커맨드/워크플로우 | 에이전트 |
|----|----------|-----------------|---------|
| Claude Code | `CLAUDE.md` | `.claude/commands/` | `.claude/agents/` |
| OpenAI Codex | `AGENTS.md` | `.codex/commands/` | `.codex/agents/` |
| GitHub Copilot | `AGENTS.md` | — | — |
| Cursor | `AGENTS.md` | `.cursor/rules/` | — |
| Antigravity (Gemini) | `GEMINI.md` | `.agent/workflows/` | `.gemini/agents/` + `.agent/skills/` |

같은 워크플로우를 두세 군데에 복사하면? 수정할 때마다 전부 동기화해야 해. 사람이 이걸 실수 없이 할 수 있을까? 없어.

---

## 해결: `.agents/`에 한 벌만, 나머지는 심볼릭 링크

```
내-프로젝트/
├── .agents/                    ← 여기가 정본
│   ├── workflows/              ← 워크플로우 파일들
│   │   ├── 01-publish.md
│   │   ├── 02-review.md
│   │   └── 03-deploy.md
│   └── roles/                  ← 에이전트 역할 파일들
│       ├── reviewer.md
│       └── deployer.md
│
├── .claude/
│   ├── commands → ../.agents/workflows   ← 심볼릭 링크
│   └── agents   → ../.agents/roles       ← 심볼릭 링크
│
├── .codex/
│   ├── commands → ../.agents/workflows   ← 심볼릭 링크
│   └── agents   → ../.agents/roles       ← 심볼릭 링크
│
├── AGENTS.md                   ← 공용 지침 (대부분의 툴이 읽어)
└── CLAUDE.md                   ← Claude 전용 추가 지침
```

**핵심**: 파일은 `.agents/`에만 있어. `.claude/commands/`와 `.codex/commands/`는 같은 파일을 가리키는 바로가기야.

---

## 따라해봐

### Step 1. 정본 폴더 만들기

```bash:터미널
mkdir -p .agents/workflows .agents/roles
```

### Step 2. 기존 파일 옮기기

이미 `.claude/commands/`에 파일이 있으면:

```bash:터미널
# 기존 파일을 정본으로 이동
mv .claude/commands/*.md .agents/workflows/
mv .claude/agents/*.md .agents/roles/
```

### Step 3. 기존 폴더 지우고 심볼릭 링크 걸기

```bash:터미널
# Claude Code용
rm -rf .claude/commands .claude/agents
ln -s ../.agents/workflows .claude/commands
ln -s ../.agents/roles .claude/agents

# Codex용
mkdir -p .codex
ln -s ../.agents/workflows .codex/commands
ln -s ../.agents/roles .codex/agents
```

### Step 4. 확인

```bash:터미널
ls -la .claude/commands
# → ../.agents/workflows  (화살표가 보이면 성공)

ls .claude/commands/
# → .agents/workflows/ 안의 파일 목록이 그대로 나와야 해
```

이제 `.agents/workflows/01-publish.md`를 고치면 `.claude/commands/`에서도 `.codex/commands/`에서도 바로 반영돼.

## AGENTS.md — 크로스 툴 표준

`AGENTS.md`는 여러 AI 툴이 공통으로 읽는 지침 파일이야.

Linux Foundation이 주도하는 업계 표준이고, 지금 이 툴들이 지원해:

| 툴 | AGENTS.md 지원 |
|----|---------------|
| OpenAI Codex | 네이티브 (기본으로 읽어) |
| GitHub Copilot | 네이티브 |
| Cursor | 네이티브 |
| Windsurf | 네이티브 |
| Claude Code | Fallback (CLAUDE.md가 없으면 AGENTS.md를 읽어) |

그래서 **공용 규칙은 AGENTS.md에, Claude 전용 규칙은 CLAUDE.md에** 넣어.

### 어떻게 나눠?

| 넣을 곳 | 내용 |
|---------|------|
| `AGENTS.md` | 워크플로우 목록, 실행 조건, 인텐트 라우팅, 가드레일 |
| `CLAUDE.md` | 프로젝트 구조, 개발 명령어, 코딩 규칙, Claude만 아는 설정 |

`CLAUDE.md` 맨 위에 이렇게 적어두면 돼:

```markdown
> 워크플로우와 에이전트는 AGENTS.md 참조.
> 파일 원본은 .agents/workflows/와 .agents/roles/에 있다.
```

## 새 AI 툴 추가할 때

새 툴이 나와도 `.agents/` 파일은 건드릴 필요 없어. 심볼릭 링크만 하나 더 걸면 돼.

```bash:터미널
# 새 툴이 AGENTS.md를 읽으면 — 끝. 추가 작업 없어.

# 자체 폴더를 쓰면:
mkdir -p .새툴
ln -s ../.agents/workflows .새툴/commands
ln -s ../.agents/roles .새툴/agents
```

## Antigravity (Gemini) 설정

Antigravity는 다른 툴과 다르게 **3가지를 동시에 맞춰야** 워크플로우가 인식돼.

### 인식 조건 3가지

| 조건 | 없으면 |
|------|--------|
| `.gemini/settings.json`에 `enableAgents: true` | 에이전트/워크플로우 기능 자체가 꺼져 |
| 워크플로우 파일에 `description:` frontmatter | Customizations 패널에 안 뜸 |
| `.agent/skills/{이름}/SKILL.md` 파일 존재 | 스킬로 인식 안 됨 |

하나라도 빠지면 워크플로우가 보이지 않아. 심볼릭 링크만 걸었는데 안 된다면 이 세 가지를 확인해.

### Step 1. `.gemini/settings.json` 생성

```json
{
  "context": {
    "fileName": ["GEMINI.md", "CLAUDE.md"]
  },
  "experimental": {
    "enableAgents": true
  }
}
```

`context.fileName`은 Antigravity가 세션 시작 시 자동으로 읽는 파일 목록이야. `enableAgents`가 `true`여야 에이전트와 워크플로우 기능이 켜져.

### Step 2. `GEMINI.md` 생성

Antigravity는 `@./경로` 구문으로 다른 파일을 자동 임포트해.

```markdown
# GEMINI.md

## 규칙 (자동 임포트)

@./CLAUDE.md
@./AGENTS.md

## 파이프라인

(프로젝트별 파이프라인 요약)

## 에이전트 역할

(에이전트별 한 줄 설명)
```

`CLAUDE.md`나 `AGENTS.md`에 이미 있는 규칙을 복붙하지 않아도 돼. `@./` 임포트로 가져오면 돼.

### Step 3. 에이전트를 `.gemini/agents/`에 연결

```bash:터미널
mkdir -p .gemini/agents
ln -s ../.agents/roles .gemini/agents
# 또는 개별 심볼릭 링크:
# ln -sf ../../.agents/roles/reviewer.md .gemini/agents/reviewer.md
```

### Step 4. 워크플로우 frontmatter 추가

**이게 제일 많이 빠뜨리는 부분이야.** Antigravity는 워크플로우 파일 상단에 `description:` frontmatter가 있어야 인덱싱해.

```markdown
---
description: 코드 리뷰 워크플로우 - PR 검토 + 품질 체크
---

# Review Workflow

1. PR diff를 읽는다
2. ...
```

frontmatter 없이 바로 `# Title`로 시작하면? Customizations 패널에 안 뜸. 파일은 있는데 안 보이면 십중팔구 이거야.

### Step 5. `.agent/skills/` 스킬 등록

Antigravity는 `.agent/skills/{이름}/SKILL.md` 패턴으로 스킬을 인식해. 에이전트 정의 파일을 심볼릭 링크로 연결하면 단일 소스 원칙도 지켜져.

```bash:터미널
mkdir -p .agent/skills/reviewer
ln -sf ../../../.agents/roles/reviewer.md .agent/skills/reviewer/SKILL.md
```

여러 에이전트를 한번에 걸려면:

```bash:터미널
for agent in reviewer deployer builder; do
  mkdir -p .agent/skills/$agent
  ln -sf ../../../.agents/roles/$agent.md .agent/skills/$agent/SKILL.md
done
```

### 최종 구조

```
내-프로젝트/
├── .agents/                    ← 정본 (변경 없음)
│   ├── workflows/
│   └── roles/
├── .claude/
│   ├── commands → ../.agents/workflows
│   └── agents   → ../.agents/roles
├── .gemini/
│   ├── settings.json           ← enableAgents: true
│   └── agents → ../.agents/roles
├── .agent/
│   ├── workflows/              ← frontmatter 필수
│   │   ├── review-workflow.md
│   │   └── deploy-workflow.md
│   └── skills/                 ← SKILL.md 심볼릭 링크
│       ├── reviewer/
│       │   └── SKILL.md → ../../../.agents/roles/reviewer.md
│       └── deployer/
│           └── SKILL.md → ../../../.agents/roles/deployer.md
├── AGENTS.md
├── CLAUDE.md
└── GEMINI.md                   ← @./ 임포트
```

### 체크리스트

Antigravity에서 워크플로우가 안 보일 때 순서대로 확인해:

- [ ] `.gemini/settings.json` 존재하고 `enableAgents: true`인가
- [ ] `GEMINI.md`가 프로젝트 루트에 있는가
- [ ] 워크플로우 `.md` 파일에 `description:` frontmatter가 있는가
- [ ] `.agent/skills/{이름}/SKILL.md`가 존재하는가 (빈 폴더면 안 됨)
- [ ] Antigravity 워크스페이스를 다시 열었는가 (설정 변경 후 리로드 필요)

## 주의할 점

### 수정은 반드시 `.agents/`에서

`.claude/commands/01-publish.md`를 직접 열어서 수정하면? 그게 실은 `.agents/workflows/01-publish.md`야. 같은 파일이니까 상관없어.

하지만 **심볼릭 링크를 지우고 새 파일을 만들면** 연결이 끊겨. 조심해.

### Git에서 심볼릭 링크

```bash:터미널
git add .claude/commands .claude/agents .codex/commands .codex/agents
```

Git은 심볼릭 링크 자체를 추적해. 다른 사람이 clone해도 동일한 구조가 복원돼.

### Windows에서

Windows는 심볼릭 링크에 관리자 권한이 필요하거나 작동이 불안정할 수 있어. 그럴 땐 복사 스크립트를 만들어:

```bash
# scripts/sync-agent-configs.sh
cp -r .agents/workflows/* .claude/commands/
cp -r .agents/roles/* .claude/agents/
cp -r .agents/workflows/* .codex/commands/
cp -r .agents/roles/* .codex/agents/
```

## 실제 프로젝트 예시

디자인 시스템 프로젝트에서 6개 워크플로우 + 7개 에이전트를 이 구조로 운영한 결과:

```
.agents/
├── workflows/
│   ├── 01-publish-scene.md        ← 이미지 → 코드 퍼블리싱
│   ├── 02-review-unapproved.md    ← QA/품질 개선
│   ├── 03-promote-ds-assets.md    ← 디자인 시스템 자산 승격
│   ├── 04-group-screens.md        ← 씬 → 스크린 그룹핑
│   ├── 05-extract-patterns.md     ← 패턴 추출
│   └── 06-build-storybook.md      ← 인터랙티브 뷰어
└── roles/
    ├── scene-publisher.md         ← 구현 담당
    ├── scene-plan-reviewer.md     ← 계획 검토
    ├── scene-qa-reviewer.md       ← QA 심사
    ├── ds-asset-reviewer.md       ← 자산 승격 검토
    ├── screen-grouper.md          ← 스크린 그룹핑 담당
    ├── pattern-extractor.md       ← 패턴 추출 담당
    └── storybook-builder.md       ← 뷰어 구현 담당
```

Claude Code로 씬을 퍼블리싱하다가, Codex로 배치 작업을 돌려도 **같은 워크플로우, 같은 에이전트 정의**를 보게 돼. 한쪽에서 워크플로우를 개선하면 다른 쪽에도 즉시 반영.

## 한 줄 정리

**`.agents/`에 한 벌만 두고, 심볼릭 링크로 각 툴에 연결.** `AGENTS.md`에 공용 규칙, `CLAUDE.md`에 Claude 전용 규칙.
