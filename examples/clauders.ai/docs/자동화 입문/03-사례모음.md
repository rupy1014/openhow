**담당자를 조합하는 방식이 다섯 가지야. 체인, 분류, 동시처리, 지휘자, 반복개선.**

이 다섯 가지를 섞으면 웬만한 작업은 자동화돼.

---

## 체인은 언제 써?

앞 담당자의 결과를 다음 담당자가 이어받을 때. 순서가 중요해.

왜? 경력 분석 담당이 경력을 읽기 전에 글쓰기 담당이 먼저 쓰면? 근거 없는 이야기가 나와.

## 반복개선은 언제 써?

한 번에 완벽한 결과가 안 나올 때. 만들고 → 평가하고 → 고치고를 반복해.

왜? 글쓰기를 생각해봐. 초안이 바로 완성본인 적 없잖아. 피드백 받고 고치는 게 자연스러운 흐름이야.

**마케팅 문구 다듬기** — `이 문구 반복 개선해서 최종본 만들어줘.`

:::canvas-flow
{
  "nodes": [
    { "id": "writer", "label": "writer\n초안 작성", "col": 0, "row": 0, "type": "default" },
    { "id": "eval", "label": "evaluator\n평가 + 피드백", "col": 1, "row": 0, "type": "warning" },
    { "id": "final", "label": "통과!\n최종본 저장", "col": 2, "row": 0, "type": "success" }
  ],
  "edges": [
    { "from": "writer", "to": "eval" },
    { "from": "eval", "to": "writer", "label": "미달" },
    { "from": "eval", "to": "final", "label": "통과" }
  ],
  "direction": "LR",
  "cols": 3,
  "rows": 1
}
:::

1회차: 초안 → 평가자가 "너무 딱딱해" → 수정.
2회차: 수정본 → 평가자가 "혜택이 안 보여" → 수정.
3회차: 수정본 → 평가자가 "통과" → 끝.

평가 기준이 모호하면 영원히 돌아. 점수 기준이나 반복 횟수 상한을 정해둬야 해.

이전 글의 `/write`가 이 방식이야. 검수 담당(proofreader)이 80점 미만이면 되돌리는 구조.

## 어떤 패턴을 골라?

| 이런 상황이면 | 이 방식 |
|-------------|---------|
| 순서대로 처리해야 해 | 체인 |
| 종류별로 담당자가 달라 | 분류 |
| 동시에 여러 가지를 돌려도 돼 | 동시처리 |
| 상황 봐서 유연하게 대응해야 해 | 지휘자 |
| 품질을 반복해서 올려야 해 | 반복개선 |

하나만 쓰라는 법 없어. 시장조사(동시처리) → 통합 분석(체인) → 리포트 검수(반복개선)처럼 조합하면 돼.

---

## 한 줄 정리

체인은 순서, 분류는 종류별, 동시처리는 속도, 지휘자는 유연함, 반복개선은 품질. 다섯 방식을 조합하면 복잡한 작업도 자동화할 수 있어.
