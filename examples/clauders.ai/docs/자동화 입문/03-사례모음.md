**에이전트 패턴은 다섯 가지다. 체인, 라우팅, 병렬, 오케스트레이터, 평가-개선.**

조합하면 웬만한 작업은 자동화된다.

---

## 체인은 언제 써?

앞 에이전트 결과가 다음 에이전트 입력이 될 때. 순서가 중요하다.

왜? 프로파일러가 경력을 읽기 전에 스토리텔러가 먼저 쓰면? 근거 없는 이야기가 나와.

## 평가-개선은 언제 써?

한 번에 완벽한 결과가 안 나올 때. 만들고 → 평가하고 → 고치고를 반복한다.

왜? 글쓰기를 생각해봐. 초안이 바로 완성본인 적 없잖아. 피드백 받고 고치는 게 자연스러운 흐름이야.

**마케팅 문구 최적화** — `이 문구 반복 개선해서 최종본 만들어줘.`

:::canvas-flow
{
  "nodes": [
    { "id": "writer", "label": "writer\n초안 작성", "col": 0, "row": 0, "type": "default" },
    { "id": "eval", "label": "evaluator\n평가 + 피드백", "col": 1, "row": 0, "type": "warning" },
    { "id": "final", "label": "통과!\n최종본 저장", "col": 2, "row": 0, "type": "success" }
  ],
  "edges": [
    { "from": "writer", "to": "eval" },
    { "from": "eval", "to": "writer", "label": "미달" },
    { "from": "eval", "to": "final", "label": "통과" }
  ],
  "direction": "LR",
  "cols": 3,
  "rows": 1
}
:::

1회차: 초안 → 평가자가 "너무 딱딱해" → 수정.
2회차: 수정본 → 평가자가 "혜택이 안 보여" → 수정.
3회차: 수정본 → 평가자가 "통과" → 끝.

평가 기준이 모호하면 영원히 돈다. 점수 기준이나 반복 횟수 상한을 정해둬야 한다.

이전 글의 `/write`가 이 패턴이야. proofreader가 80점 미만이면 되돌리는 구조.

## 어떤 패턴을 골라?

| 이런 상황이면 | 이 패턴 |
|-------------|---------|
| 순서대로 처리해야 해 | 체인 |
| 유형별로 전문가가 달라 | 라우팅 |
| 동시에 여러 가지를 돌려도 돼 | 병렬 |
| 상황에 따라 유연하게 대응해야 해 | 오케스트레이터 |
| 품질을 반복해서 올려야 해 | 평가-개선 |

하나만 쓰라는 법 없다. 시장조사(병렬) → 통합 분석(체인) → 리포트 검수(평가-개선)처럼 조합하면 된다.

---

## 한 줄 정리

체인은 순서, 라우팅은 분류, 병렬은 속도, 오케스트레이터는 유연함, 평가-개선은 품질. 다섯 패턴을 조합하면 복잡한 작업도 에이전트로 설계할 수 있다.
