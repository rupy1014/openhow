# clauders.ai 4기 웨비나 — 발표 초안

> "오픈클로로 시작했다가 안 쓰고 새로 만든 카카오톡 비서 자동화 사례"
> clauders.ai 4기 홍보 목적 / 무료 강의 / 대상: Claude Code 사용자

---

## 발표 구성 (약 50분)

---

### 1. 훅: "카톡에 AI를 심었습니다" (3분)

> "고객이 카톡을 보내면, 2초 만에 AI가 답장합니다."

#### 데모 (화면 공유 또는 영상)

```
고객: "내일 오후 3시 예약 가능한가요?"
    ↓ 2초
냥팀장(AI): "내일 오후 3시 가능합니다! 성함 알려주시면 바로 잡아드릴게요 😊"
```

- 카카오톡 API가 없는데 어떻게? → **Android 알림을 읽고, 답장 버튼을 코드로 누른다**
- 사람이 알림 보고 답장하는 것과 똑같은 방식 — 카카오톡은 구분 못 함
- 1:1 채팅도, 단톡방도 동작
- 레이턴시 2~5초. 건당 비용 0.32원.

→ "어떻게 여기까지 왔는지, 처음부터 이야기해보겠습니다."

### 2. 왜 만들게 됐나 — 오픈클로에서 시작 (8분)

> "요즘 핫한 오픈클로, 써보니까 이런 생각이 들더라."

#### 오픈클로, 요즘 어떻게 쓰나

(청중에게 질문 던지기) "오픈클로 써보신 분? 어떻게 쓰고 계세요?"

보통 이런 패턴으로 쓰더라:

| 패턴 | 설명 | 예시 |
|------|------|------|
| **비서** | 텔레그램으로 대화하면서 잡무 처리 | "오늘 일정 알려줘", "이 글 요약해줘" |
| **Cron 스케줄러** | 주기적으로 에이전트가 알아서 돌아감 | 매일 아침 뉴스 브리핑, 모니터링 |
| **프로젝트 작업** | Claude Code처럼 코드 작성/수정 시키기 | "이 파일 리팩토링해줘" |

#### 나도 써봤다 — 맥미니 M4 사서

**맥미니 M4 16기가를 사서** 셋업하고 돌려봤다. 처음엔 기대가 컸다.

그런데 개발자 입장에서 솔직히 말하면, **PC 앞에서 Claude Code로 개발하는 게 더 생산적이었다.**

- Claude Code: 프로젝트 컨텍스트를 이미 알고, 파일 구조를 보면서 정확하게 수정
- 오픈클로 (Codex): 텔레그램 모바일에서 프롬프트를 치다 보니 맥락 전달이 어려움
- Codex로 코드 수정을 시켰는데, 결과가 마음에 안 드는 경우가 많았다
  - 프롬프트의 한계인지, 텔레그램 모바일 UI의 한계인지, Codex 자체의 한계인지 분리가 안 됨
  - 어쨌든 **결과물의 만족도가 낮았다**

자연스럽게 역할이 갈렸다:

```
오픈클로에 시키는 것: 자동화, 모니터링, 리뷰, 결과 보고
Claude Code에 시키는 것: 코드 작성, 버그 수정, 기능 구현
```

그런데 이러다 보니 의문이 생겼다 — **"이럴 바에 굳이 오픈클로를 써야 하나?"**

- cron으로 뉴스 브리핑? → **그냥 수집 프로젝트 만들어서 cron 돌리면 된다**
- 모니터링 결과 텔레그램 알림? → **GitHub Actions + 텔레그램 봇으로도 된다**
- **범용 AI 에이전트 플랫폼을 cron 러너로만 쓰고 있었다**

#### 뜯어봤다 — 오픈클로의 4가지 핵심

"대체 오픈클로가 뭘 하는 건지" 구조를 뜯어보니 핵심이 4가지더라:

| # | 기능 | 하는 일 |
|---|------|---------|
| 1 | **LLM Gateway** | 여러 LLM 모델 중 원하는 걸 골라서 돌릴 수 있는 관문 |
| 2 | **Tool Orchestration** | 웹 검색, 브라우저, 쉘 명령 등 도구를 LLM이 자연어로 판단해서 실행 |
| 3 | **Cron / Scheduling** | 주기적으로 에이전트를 깨워서 작업 (브리핑, 모니터링) |
| 4 | **Session / Multi-Agent** | 대화 세션 유지 + 여러 에이전트 라우팅 |

좋은 구조다. 그런데 내가 코드 작업에 안 쓰고 있으니, 진짜 쓰고 있는 건 3번(Cron)뿐이었다.

#### 깨달음: "직접 만들 수 있지 않나?"

이 4가지를 카카오톡 맥락에서 보니 **전부 대체 가능**했다:

| 오픈클로 | 대체한 방법 | 왜 더 나은가 |
|---------|------------|-------------|
| **LLM Gateway** | OpenRouter API 직접 호출 | 중간 레이어 제거, 레이턴시 15초→2초 |
| **Tool Orchestration** | LLM function calling 자체 구현 | 범용 도구 대신 비즈니스 맞춤 도구만 (예약, 입금확인, 전화) |
| **Cron / Scheduling** | Claude Code로 프로젝트 파서 작업 | cron + 에이전트보다 명확하고 토큰도 줄어듦 |
| **Session / Multi-Agent** | 방별 persona + room_souls DB | 카카오톡 방 구조에 맞춘 세션 관리가 자연스러움 |

> "오픈클로의 진짜 가치는 **'이렇게 만들면 된다'는 아키텍처 레퍼런스**. 아이디어를 차용하고, 직접 만들자."

특히 "카카오톡 단톡방에 AI를 심으면?" — 텔레그램이 아니라 **사람들이 이미 쓰는 곳에**.

### 3. 어떻게 만들었나 — 기술 + 바이브 코딩 (10분)

> "카카오톡에 봇을 심는다? API도 없는데? 방법이 있습니다."

#### 힌트: 메신저봇R

- 한국 안드로이드 커뮤니티에서 돌아다니는 카톡 자동응답 앱 **메신저봇R**
- 원리가 단순하다: **Android 알림을 읽고, 알림의 답장 버튼을 코드로 누른다**
- 카카오톡 API가 없어도 작동한다 — 알림 접근성 서비스는 Android 표준 기능
- 메신저봇R은 JavaScript 스크립트로 규칙을 짜는 방식 → 단순 자동응답에는 OK
- 하지만 LLM 연동, 컨텍스트 관리, 도구 실행까지 하려면 **직접 앱을 만드는 게 낫다**

→ 메신저봇R의 **작동 원리만 차용**하고, 앱을 직접 만들었다.

#### 핵심 기술: NotificationListener + RemoteInput

Android가 공식 제공하는 두 가지 API만으로 동작한다:

```
① 알림 읽기: NotificationListenerService (Android 4.3+)
   → 카카오톡 알림이 오면 onNotificationPosted() 콜백
   → sender, message, room 추출

② 알림 답장: RemoteInput (Android 7.0+)
   → 알림의 "답장" 버튼에 담긴 PendingIntent를 코드로 실행
   → 카카오톡이 답장을 받아서 메시지 전송
```

**이게 핵심이다** — 카카오톡 API 해킹이 아니라, Android의 정규 알림 시스템을 활용:
- 사람이 알림을 보고 → 답장 버튼을 누르는 행위를 코드가 대신 한다
- 카카오톡 입장에서는 사용자가 직접 답장한 것과 구분 불가

#### 아키텍처: Android가 두뇌다

```
고객이 카톡 보냄
    ↓
Android NotificationWatcher (알림 캡처)
    ↓ sender, message, room 추출
    ↓ debounce 1~3초 (연속 메시지 묶기)
    ↓
SoulEngine (두뇌)
    ├─ Worker API 병렬 호출 → 페르소나, 히스토리, 고객 메모리 수집 (~200ms)
    ├─ 의도 분류 + 프롬프트 조립
    ├─ OpenRouter API 호출 → Gemini Flash (~2초)
    ├─ function calling 시 → 도구 실행 (예약 조회, 입금확인 등)
    └─ 최종 답변 생성
    ↓
ReplyManager (답장 실행)
    ↓ RemoteInput으로 카카오톡 답장
    ↓
고객이 카톡에서 답장을 받음

총 레이턴시: 2~5초
```

**왜 Android가 두뇌인가:**
- 서버(Worker)가 두뇌이면: 알림 → 서버 전송 → LLM 호출 → 서버 응답 → 답장 (왕복 비용)
- Android가 두뇌이면: 알림 → **바로 LLM 호출** → 바로 답장 (왕복 없음)
- 서버가 죽어도 Android만 살아있으면 서비스는 돌아간다

#### 바이브 코딩: 한 사람이 풀스택을 다루는 법

**스택 (한 사람이 관리):**
- Android Kotlin (SoulEngine — 알림 캡처 + LLM 호출 + 답장)
- Cloudflare Worker TypeScript (API + DB + 결제)
- Vanilla JS SPA (관리자 대시보드)
- Nginx (프론트엔드 + API 프록시)
- Supabase (인증)

**한 사람이 이 스택을 다루는 게 가능한 이유:**
- Claude Code가 Android Kotlin도 치고, TypeScript도 치고, 배포 스크립트도 짠다
- "이거 어떻게 구현하지?"가 아니라 **"뭘 만들지?"**에 집중할 수 있다

**실제 작업 흐름 (라이브 시연 가능):**
1. "단톡방에서 내가 보낸 답장을 다시 캡처해서 무한루프 도는 문제 고쳐줘"
2. Claude Code가 코드 분석 → 원인 파악 → 수정 → 테스트까지
3. `git log` 한 줄이 남는다

**연동 아키텍처도 단순화:**
```
처음: 카톡 → Android → Worker → Cloudflare Tunnel → Mac Mini → OpenClaw Gateway → 응답
지금: 카톡 → Android → OpenRouter API 직접 호출 → 응답
```
중간 단계 4개 → 1개. Mac Mini 의존성 제거. 레이턴시 15~30초 → 2~5초.

### 4. 진짜 문제 — QA 케이스가 폭발한다 (10분)

> "바이브 코딩의 진짜 트러블은 코드를 못 짜는 게 아닙니다. QA 케이스가 폭발하는 겁니다."

#### 왜 QA가 폭발하는가

카카오톡 자동응답은 **edge case의 지뢰밭**:

| 문제 | 설명 |
|------|------|
| **에코 루프** | 내가 보낸 답장을 다시 알림으로 캡처 → 또 답장 → 무한루프 |
| **re-scan 중복** | 알림 재스캔 시 이미 답한 트리거에 다시 매칭 |
| **rapid-fire** | 3초 안에 메시지 5개 → 5개 다 답할 건지, 마지막만 답할 건지 |
| **[SKIP] 판단** | 단톡방에서 "이건 나한테 한 말 아닌데" 판단하기 |
| **unsolicited summary** | 아무도 안 물어봤는데 대화 요약을 던지는 문제 |
| **auto_reply=0인데 응답** | 자동응답 끈 방에서도 특정 조건에서 답하는 버그 |
| **그룹채팅 + 1:1 분기** | 같은 메시지라도 1:1이냐 단톡이냐에 따라 동작이 달라야 함 |

#### 바이브 코딩과 QA의 관계

- 기능 하나 만드는 건 빠르다 (Claude Code 덕분에)
- 하지만 **그 기능이 다른 기능과 충돌하는 케이스**는 AI가 미리 알려주지 않는다
- 결국 **직접 써보면서** 터지는 걸 하나씩 잡아야 한다
- 현재 **24개 도메인, 130개 시나리오** 자동 테스트로 방어

```
커밋 로그가 이야기해준다:
- "Fix echo/unsolicited summary/rapid-fire issues in group chat"
- "Fix re-scan matching already-responded triggers in group chats"
- "Fix auto_reply=0 group chat still responding"
```

→ **"만들고 → 써보고 → 터지고 → 고치고"의 반복**

#### 교훈

- 바이브 코딩으로 **빠르게 만드는 건 진짜 빠르다**
- 하지만 **품질을 올리는 건 결국 QA에 달려있다**
- AI가 코드를 짜주는 시대에, 개발자의 진짜 가치는 **"뭘 테스트해야 하는지 아는 것"**

### 5. 현실 — MVP 오버스펙, 단톡방, 그리고 오픈클로의 귀환 (8분)

> "솔직한 이야기를 하겠습니다."

#### 수익화를 생각하다

기술적으로 돌아가는 걸 확인하면, 자연스럽게 다음 단계에 접어든다:

```
리서치 → 기획 → 가설 → MVP → 구현 → QA → 피드백 → 반복
```

- **리서치**: 소상공인의 카톡 업무 패턴 조사
- **기획**: 네일샵, 미용실, 카페 — 카톡으로 예약, 문의, 안내하는 사람들
- **가설**: "고객 카톡에 자동 응답하면 사장님의 시간을 아껴줄 수 있다"
- **MVP 정의**: AI 자동응답 + 예약 관리 + 입금확인 + 관리자 대시보드
- **수익 모델**: 3-tier 정기결제 (5.9만~29.9만), 리셀러 파트너 프로그램까지 설계

#### 현실: MVP가 오버스펙이었다

솔직히 말하면 **MVP를 너무 크게 잡았다.**

- 자동응답 + 예약 + 입금확인 + 결제 + 파트너 → 한 사람이 감당하기엔 스코프가 너무 넓음
- 본업 + 다른 사이드 프로젝트에 치여서 **소상공인 상대로 제대로 된 QA를 못 하고 있다**
- 결제/파트너 시스템까지 만들어놨는데, 정작 고객에게 써보게 하는 단계에서 멈춰있음
- **런칭이 계속 딜레이되는 상황**

#### 단톡방이 테스트베드가 되다

개인적으로 매일 쓰다 보니, 자연스럽게 **단톡방 관리 쪽으로** 활용하게 됐다.

- 단톡방에 AI 비서("냥팀장" 캐릭터)를 넣고 돌리는 중
- QA 케이스의 80%가 **그룹채팅에서 발생** (에코, SKIP, rapid-fire 전부)
- 소상공인도 결국 **고객 단톡방**을 운영 → 단톡방 QA = 소상공인 QA
- "얘 뭐야 재밌다" → 자연스러운 입소문

**써보면서 발견한 것들:**
- 단톡방에서 AI가 "눈치 없이" 끼어들면 사람들이 불쾌해한다
- [SKIP] 판단이 **생각보다 훨씬 중요** — "이건 나한테 한 말이 아니다"를 정확히 판단해야
- 반대로, 적절한 타이밍에 한마디 던지면 반응이 폭발적

#### 오픈클로의 귀환

그리고 재밌는 건 — **QA 자동화를 오픈클로로 돌리고 있다.**

- 처음에 "굳이 오픈클로를 써야 하나?" 했는데, 결국 **자동 QA 모니터링 용도로 다시 쓰게 됨**
- cron으로 테스트 시나리오 돌리고, 결과를 텔레그램으로 보고받는 구조
- 오픈클로의 원래 강점(cron + 자동화)이 **여기서는 딱 맞았다**

```
원래 타겟: 소상공인 → 고객 응대 자동화 (런칭 딜레이 중)
실제 사용: 내 단톡방 → 단톡방 AI 비서 (냥팀장)
QA 자동화: 오픈클로 → cron으로 테스트 돌리고 결과 보고
```

→ 오픈클로를 안 쓴다고 했다가, 결국 **자기 자리를 찾아서 다시 쓰고 있는** 아이러니

### 6. 가져갈 것 + clauders.ai 4기 (5분)

#### 현재 구현 상태

**구현 완료:**
- AI 자동응답 (1:1 + 그룹채팅)
- 방별 페르소나 설정 (캐릭터, 톤, 규칙)
- 은행 입금 알림 자동 감지
- 예약 관리 (가용 시간 조회 + 예약 생성)
- 전화 걸기, 기기 제어 (알람, 볼륨 등)
- FAQ 자동응답, 환영 인사
- 관리자 대시보드 (웹)
- Bootpay 정기결제 + 리셀러/앰버서더 파트너 프로그램
- 24개 도메인 130개 시나리오 자동 테스트

**아키텍처 한 장:**
```
고객 (카카오톡)
    ↕ 메시지
사장님 카카오톡 (서브폰)
    ↕ 알림 캡처
Android SoulEngine
    ├─ 알림 → 의도 분류 → 프롬프트 조립
    ├─ OpenRouter API (Gemini Flash) 호출
    ├─ 도구 실행 (예약, 입금확인, 전화 등)
    └─ RemoteInput 자동 답장
    ↕ HTTPS
Cloudflare Worker + D1
    ├─ 상태 저장 (설정, 히스토리, 메모리)
    ├─ 결제/파트너 API
    └─ 관리자 대시보드 API
    ↕
Web Admin (tokdaeri.com)
```

#### 세 가지 테이크어웨이

**① 오픈클로 같은 도구는 "학습 재료"로 최고다**
- 직접 써보면 "AI 자동화가 실제로 뭘 하는지" 체감된다
- 4가지 핵심을 뜯어보면 **내 유스케이스에 맞게 재조합**할 수 있다
- 컨셉은 차용하되, 의존성은 걷어내는 것이 제품화의 시작

**② 바이브 코딩은 "만드는 속도"를 해결하지만, "품질"은 별개다**
- Claude Code로 Android + Worker + Frontend를 한 사람이 관리 가능
- 하지만 QA 케이스 폭발은 AI가 대신 해주지 않는다
- **테스트 시나리오를 설계하는 능력**이 바이브 코딩 시대의 핵심 스킬

**③ "자동화를 제품으로 만든다"는 관점**
- 데모에서 끝나는 자동화 vs 매일 돌아가는 자동화는 전혀 다른 문제
- 에러 처리, 멀티테넌시, 결제, 배포, 모니터링 — 이게 진짜 일

#### clauders.ai 4기
- "도구 사용법"이 아니라 **"도구로 실제 사례를 만들고 운영하는 사람들"**의 모임
- 아이디어 → 프로토타입 → 제품화 → QA → 운영까지의 전 과정
- 혼자 하면 막히는 지점을 함께 돌파하는 곳

## 예상 Q&A

| 질문 | 답변 포인트 |
|------|------------|
| 카카오톡 API가 없는데 어떻게? | Android 알림 접근성 서비스로 캡처. "사람이 알림 보는 것"과 같은 방식 |
| 오픈클로를 왜 완전히 안 쓰나? | 코드 작업은 Claude Code가 더 강력하고, 남는 건 cron뿐. 아키텍처 레퍼런스로서의 가치는 큼 |
| LLM 모델은? | OpenRouter 경유 Gemini Flash Lite. 건당 0.32원 |
| 단톡방에서 사람들 반응은? | "냥팀장" 캐릭터. 눈치 있으면 반응 폭발, 눈치 없으면 바로 불만 |
| 수익화는? | 3-tier 정기결제 (5.9만~29.9만). 리셀러 파트너도 준비 중 |
| 혼자 만든 건가? | 코드는 Claude Code와 페어. 기획/QA/운영은 사람의 몫 |

## 발표 톤

- **기술 자랑 X** → 경험담 위주
- "이렇게 만들었다" < **"이런 선택을 했고, 이런 걸 포기했다"**
- 오픈클로를 깎아내리는 게 아니라, **"좋은 레퍼런스였고, 뜯어서 배워서 직접 만들었다"**
- clauders.ai 4기 홍보는 마지막에 자연스럽게 — **"이런 과정을 같이 하는 곳"**
