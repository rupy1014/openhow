**"만들어줘" 한마디면 AI가 코드를 짜. 근데 그게 전부가 아니야.**

바이브코딩은 코드를 모르는 사람이 말로 서비스를 만드는 방법이야.

근데 "만들어줘"만 반복하면 금방 막혀.

---

## 바이브코딩이 뭐야?

2025년 2월, AI 연구자 Andrej Karpathy가 X에 올린 글에서 나온 말이야.

> "코드가 존재한다는 것도 잊고, 바이브에 완전히 맡긴다."

원하는 걸 말로 설명하고, 나온 결과를 보고, 다시 말하고. 이 반복이 전부야.

그해 Collins 영어 사전이 바이브코딩을 **올해의 단어**로 뽑았어.

---

## 왜 "만들어줘"만으로는 안 돼?

"TODO 앱 만들어줘"라고 하면, AI가 알아서 구조를 정해.

그러면 이런 일이 생겨:

- 왜 이 기술을 골랐는지 모르겠어
- 내가 원한 건 이게 아닌데
- 고치려고 했더니 다른 데가 망가져

이걸 **70/30 문제**라고 해.

AI가 70%까지는 순식간에 만들어줘. 근데 나머지 30%에서 막혀.

왜? 처음에 방향을 안 잡았으니까. 기초 공사 없이 건물 올린 거야.

---

## 바이브코딩 = AI가 다 해주는 거?

아니야.

AI가 코드를 짜줘. 근데 **뭘 만들지, 어떻게 만들지는 네가 결정해.**

| AI가 도와주는 것 | 네가 해야 하는 것 |
|----------------|-----------------|
| 코드 작성 | 뭘 만들지 결정 |
| 에러 수정 | 결과가 맞는지 확인 |
| 기술 추천 | 어떤 걸 선택할지 판단 |

AI는 도구야. 망치가 집을 지어주진 않잖아. 집을 어떻게 지을지 아는 사람이 망치를 쓰는 거야.

---

## 프로세스가 있으면 달라져

바이브코딩도 순서가 있어.

```
아이디어 → 기획 → 레퍼런스 → 점진적 빌드 → 디버깅 → 배포 → 피드백
```

| 단계 | 하는 일 |
|------|---------|
| 아이디어 | 뭘 만들지 찾기 (가설) |
| 기획 | 뭘 만들지 정리 (PRD) |
| 레퍼런스 | 스크린샷, 예시, 규칙 파일로 맥락 전달 |
| 점진적 빌드 | 한 기능씩 만들고 확인 |
| 디버깅 | 에러 복붙, 되돌리기, 다듬기 |
| 배포 | URL로 공개 |
| 피드백 | 사람들한테 물어보고 방향 잡기 |

"만들어줘" 한마디 vs 이 프로세스 — 결과물이 완전히 달라.

---

## 이 시리즈에서 뭘 배워?

이 시리즈는 **프로세스를 배워.** 특정 프로젝트를 따라 만드는 게 아니야.

너의 아이디어가 뭐든 — 블로그, 쇼핑몰, 대시보드, 게임 — 이 프로세스를 적용할 수 있어.

| 글 | 핵심 |
|----|------|
| 아이디어 찾기 | 네 귀찮음에서 시작하면 돼 |
| 바이브 기획 | "코드는 짜지 마"가 왜 중요한지 |
| PRD 작성법 | AI한테 기획서를 쓰게 하는 법 |
| 안 만들 것 | 기획의 절반은 빼는 거야 |
| 기술 선택 | 가장 단순한 걸 고르는 기준 |
| PRD 실전 예시 | 프로젝트별 PRD 템플릿 |
| 레퍼런스 주기 | AI한테 맥락을 잘 주는 법 |
| 한 입씩 만들기 | 한 번에 다 안 만들어 |
| 고치는 법 | 에러가 나면 어떻게 해 |
| 배포 | Vercel로 세상에 공개 |
| 피드백과 검증 | 5명한테 물어보고 방향 잡기 |
| 한계와 다음 | 언제 안 통하는지 |

---

## 한 줄 정리

바이브코딩은 "만들어줘"가 아니야. 아이디어 → 기획 → 빌드 → 피드백, 이 프로세스가 진짜야.
