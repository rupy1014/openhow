**바이브코딩은 만능이 아니야. 어디까지 되고, 어디서 안 되는지 알아야 해.**

---

## 어디에 잘 맞아?

| 잘 맞는 곳 | 예시 |
|-----------|------|
| 프로토타입 | "아이디어를 빨리 눈에 보이는 형태로" |
| 개인 프로젝트 | 포트폴리오, 블로그, 도구 |
| 아이디어 검증 | "이거 만들면 사람들이 쓸까?" |
| 1인 사이드 프로젝트 | 나만 쓰거나, 소수가 쓰는 서비스 |

바이브코딩의 강점은 **속도**야. 아이디어가 떠오르면 당일에 작동하는 걸 볼 수 있어.

---

## 어디에서 위험해?

| 위험한 곳 | 이유 |
|----------|------|
| 돈이 오가는 서비스 | 결제, 정산 로직 오류 → 실제 피해 |
| 개인정보를 다루는 곳 | AI가 보안 취약점을 못 잡아 |
| 여러 사람이 유지보수할 때 | 코드를 아무도 이해 못 해 |
| 규모가 큰 서비스 | 비슷한 문제를 다른 방식으로 풀어둬서 엉켜 |

AI가 만든 코드의 약 **45%에 보안 취약점**이 있다는 조사가 있어.

코드를 이해하지 못하면 취약점도 못 찾아.

---

## 안전하게 쓰려면?

**1. AI 코드를 신입 사원이 짠 코드처럼 봐.**

신입이 짠 코드를 그대로 배포하지 않잖아. 검토하고 고쳐.

바이브코딩도 마찬가지야. AI가 만든 결과를 그대로 배포하기 전에, 최소한 이건 확인해:

- 로그인이 우회되지 않아?
- 남의 데이터를 볼 수 없어?
- API 키가 코드에 노출되지 않아?

Claude한테 물어보면 돼:

```
이 코드에 보안 문제 없는지 확인해줘.
특히 API 키 노출, 인증 우회, 데이터 접근 부분.
```

**2. 내가 설명할 수 없는 코드는 배포하지 마.**

이건 Simon Willison(AI 분야 유명 개발자)의 원칙이야:

> "내가 다른 사람한테 이 코드가 뭘 하는지 설명 못 하면, 커밋하지 않는다."

바이브코딩으로 만든 프로토타입은 괜찮아. 근데 실제 서비스로 배포할 때는 코드를 한 번 살펴봐.

**3. 환경변수를 써.**

API 키, 비밀번호 같은 건 코드에 직접 적지 마.

```
API 키를 환경변수로 옮겨줘. 코드에 하드코딩하지 마.
```

## 다음 단계: 스펙 드리븐 개발

바이브코딩을 한 단계 올리면 **스펙 드리븐 개발**이야.

Andrej Karpathy(바이브코딩을 만든 사람)도 2026년에 이렇게 말했어:

> "바이브코딩은 이제 기본이야. 다음은 더 체계적인 방식."

스펙 드리븐 개발의 흐름:

```
1. 제품 스펙 → AI가 기획서를 작성
2. 설계 문서 → 스펙 기반으로 기술 설계
3. 구현 → 두 문서를 보고 코드 작성
4. 변경 → 스펙부터 수정 → 설계 수정 → 코드 수정
```

바이브코딩은 "말 → 코드"야.

스펙 드리븐은 "말 → 스펙 → 설계 → 코드"야.

단계가 하나 더 있지만, 결과물이 훨씬 단단해. 고칠 때도 스펙부터 고치면 코드가 따라오니까.

지금은 바이브코딩 프로세스를 먼저 익혀. 이 프로세스가 몸에 배면 스펙 드리븐은 자연스럽게 넘어가.

---

## 도구 고르기

바이브코딩 도구가 여러 가지야. 상황에 맞게 골라:

| 상황 | 추천 도구 |
|------|----------|
| 코드를 아예 모르겠어 | Lovable, Bolt |
| UI만 빠르게 만들고 싶어 | v0 (Vercel) |
| 좀 더 세밀하게 다루고 싶어 | Claude Code, Antigravity |
| 프로토타입 → 진짜 서비스로 | Claude Code + Antigravity |

하나로 정할 필요 없어. UI는 v0로 만들고, 기능은 Claude Code로 붙이는 조합도 좋아.

---

## 한 줄 정리

프로토타입엔 바이브코딩, 실서비스엔 검토 필수. 다음 단계는 스펙 드리븐이야.
