**코드부터 짜면 안 돼. 기획부터야.**

"AI 캐릭터 채팅 만들어줘"라고 하면 뭐가 나올까?

AI가 알아서 기술을 골라. 화면 구조도 정해. 기능도 넣어.

근데 그게 네가 원한 거야? 아마 아닐 거야.

---

## 제1원칙: "코드는 짜지 마"

바이브코딩에서 가장 중요한 말이야.

```
코드는 아직 짜지 마. 먼저 계획을 보여줘.
```

이 한 줄이면 AI가 코드 대신 계획을 내놓아.

왜 이게 중요해? AI한테 바로 "만들어줘"하면 뭐가 문제냐:

- AI가 복잡한 기술을 골라 (네가 필요 없는 걸)
- 화면 구조를 혼자 정해 (네 의도랑 다르게)
- 한번에 수백 줄을 쏟아내 (어디가 뭔지 모르겠어)

**"코드는 짜지 마"를 먼저 말하면,** AI가 90%의 과잉 설계를 안 해.

Peter Yang(바이브코딩 12원칙 저자)이 이렇게 말했어:

> "이 한마디가 90%의 과잉 설계를 막아준다."

계획을 보고, 마음에 드는 부분만 "이거 실행해줘"라고 하면 돼.

---

## 바이브 기획이 뭐야?

만들고 싶은 걸 AI한테 **먼저 정리해달라고** 하는 거야.

코드 말고 문서로.

이걸 **PRD**(Product Requirements Document)라고 해. 쉽게 말하면 "내가 뭘 만들고 싶은지 정리한 문서"야.

PRD가 있으면 AI가 이 문서를 기준으로 코드를 짜. 내 의도에서 벗어나지 않아.

---

## 어떻게 해?

**Step 1.** 만들고 싶은 걸 말로 설명해

```
나 AI 캐릭터 채팅 서비스 만들고 싶어.

- 캐릭터 5명이 있어
- 하나를 골라서 대화할 수 있어
- 한국어로 대화해

먼저 PRD.md로 정리해줘. 코드는 아직 짜지 마.
```

**Step 2.** AI가 PRD를 만들면 읽어봐

기술 스택, 화면 구성, 기능 목록이 정리돼 있어.

마음에 안 드는 게 있으면 말해.

```
캐릭터 선택 화면을 카드 형태로 해줘.
대화 화면은 카카오톡처럼 말풍선으로.
```

**Step 3.** 아키텍처 옵션을 물어봐

```
이 서비스를 만드는 방법이 여러 가지일 텐데,
가장 단순한 것부터 3가지 옵션을 보여줘.
아직 코드는 짜지 마.
```

AI가 옵션을 보여줘.

- 옵션 A: HTML 파일 하나로 다 해결 (가장 단순)
- 옵션 B: React + API 서버 (중간)
- 옵션 C: Next.js + DB + 인증 (풀스택)

**가장 단순한 걸 골라.** 복잡하게 시작하면 나중에 고치기 어려워.

---

## 좋은 PRD는 뭐가 달라?

PRD에 이 6가지가 있으면 돼:

| 항목 | 쉽게 말하면 |
|------|------------|
| 한 줄 설명 | "이 서비스는 뭐야?" |
| 누구를 위한 거야 | 쓰는 사람이 누군데 |
| 핵심 기능 | MVP에 꼭 필요한 것만 |
| 안 만들 것 | 이번에 안 할 것 (중요!) |
| 사용자 흐름 | 사용자가 어떤 순서로 쓰는지 |
| 성공 기준 | 어떻게 되면 완성인지 |

**"안 만들 것"이 제일 중요해.**

왜? AI는 시키지 않아도 기능을 추가해. 로그인, 다크모드, 설정 페이지 — 안 쓸 기능을 붙여.

```
안 만들 것:
- 로그인/회원가입
- 대화 저장
- 관리자 페이지
- 다국어 지원
```

이렇게 명시하면 AI가 딴짓을 안 해.

---

## 실전 예시: 포트폴리오 사이트

```
포트폴리오 웹사이트 만들고 싶어.

- 프로필 사진 + 한 줄 소개
- 프로젝트 3개를 카드로
- 연락처 섹션

안 만들 것:
- 블로그
- 로그인
- CMS(관리자 페이지)

가장 단순한 방법으로 PRD.md 정리해줘.
코드는 아직 짜지 마.
```

이렇게 하면 AI가 HTML+CSS 한 장짜리 PRD를 만들어.

React나 Next.js를 꺼내들지 않아.

---

## 한 줄 정리

코드 전에 PRD. "코드는 짜지 마"를 말하면 AI가 기획부터 해.
