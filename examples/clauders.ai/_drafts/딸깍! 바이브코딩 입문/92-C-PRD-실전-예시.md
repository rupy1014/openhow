> PRD는 길어서 좋은 게 아니야.
>
> 바로 만들 수 있어야 좋은 거야.

이 부록은 본문 13장의 연장선이다.

실제로 복사해서 바꿔 쓸 수 있는 형태로 세 가지 예시를 붙였다.

핵심은 항상 같다.

- 한 줄 설명
- 대상 사용자
- 핵심 기능
- 안 만들 것
- 사용자 흐름
- 성공 기준

---

## 예시 1. AI 캐릭터 채팅

```md
# PRD: AI 캐릭터 채팅

## 한 줄 설명
AI 캐릭터 중 하나를 골라 한국어로 대화하는 웹앱.

## 대상 사용자
AI 캐릭터 대화에 관심 있는 일반 사용자.

## 핵심 기능
1. 캐릭터 선택 화면
2. 채팅 화면
3. AI 응답 생성
4. 한국어 대화

## 안 만들 것
- 로그인/회원가입
- 대화 기록 저장
- 캐릭터 생성/편집
- 음성 대화
- 관리자 페이지

## 사용자 흐름
캐릭터 선택 → 대화 시작 → 메시지 입력 → 응답 확인

## 성공 기준
캐릭터를 고르고 10번 이상 대화할 수 있으면 완성.
```

포인트는 "안 만들 것"이 분명하다는 거다.

AI가 딴길로 새기 쉬운 기능을 처음부터 막아둔다.

---

## 예시 2. 개인 포트폴리오

```md
# PRD: 개인 포트폴리오

## 한 줄 설명
프로젝트 3개를 보여주는 1페이지 포트폴리오.

## 대상 사용자
채용 담당자, 협업 파트너.

## 핵심 기능
1. 프로필 섹션
2. 프로젝트 카드 3개
3. 연락처 섹션

## 안 만들 것
- 블로그
- CMS
- 방문자 통계
- 댓글
- 다국어
- 다크모드

## 사용자 흐름
페이지 접속 → 프로필 확인 → 프로젝트 클릭 → 연락처 확인

## 성공 기준
URL로 접속해서 프로젝트 3개가 보이면 완성.
```

포인트는 구조가 아주 단순하다는 것.

이런 건 괜히 무거운 기술 스택을 쓰지 않는 편이 낫다.

## 예시 3. 소규모 팀 TODO

```md
# PRD: 팀 TODO 리스트

## 한 줄 설명
팀원들이 할 일을 공유하는 간단한 웹앱.

## 대상 사용자
5명 이하 소규모 팀.

## 핵심 기능
1. 할 일 추가/완료/삭제
2. 팀원 이름 태그
3. 실시간 동기화

## 안 만들 것
- 로그인
- 마감일/캘린더
- 파일 첨부
- 알림
- 히스토리/되돌리기

## 사용자 흐름
URL 접속 → 할 일 추가 → 이름 태그 → 완료 체크

## 성공 기준
두 명이 동시에 접속해서 할 일을 추가하고 완료할 수 있으면 완성.
```

포인트는 핵심 경험만 남겼다는 거다.

실시간 동기화가 중요하다고 판단했기 때문에
그 외 편의 기능은 과감히 뺐다.

---

## PRD를 쓸 때 자주 하는 실수

### 핵심 기능이 너무 많다

다섯 개를 넘기기 시작하면
대체로 MVP가 아니라 희망사항 목록이 된다.

### 안 만들 것이 없다

이게 없으면 AI가 기능을 자꾸 더 붙인다.

### 성공 기준이 흐리다

"잘 되면" 같은 말 말고
행동으로 확인 가능한 문장을 써야 한다.

## 복사해서 시작하는 템플릿

```md
# PRD: [프로젝트 이름]

## 한 줄 설명

## 대상 사용자

## 핵심 기능
1.
2.
3.

## 안 만들 것
- 
- 
- 

## 사용자 흐름

## 성공 기준
```

---

## 한 줄 정리

좋은 PRD는 길지 않다.

핵심 기능은 적고,
안 만들 것은 분명하고,
성공 기준은 바로 확인 가능해야 한다.
