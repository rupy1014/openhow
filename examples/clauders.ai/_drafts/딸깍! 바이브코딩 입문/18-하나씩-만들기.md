> 한 번에 다 만들지 마.
>
> 한 입씩 만들어.

이 원칙 하나만 지켜도 성공 확률이 크게 올라간다.

입문자가 제일 많이 무너지는 지점이 여기거든.

"전체 서비스 만들어줘."

이 한 문장이 모든 사고의 시작일 때가 많다.

---

## 왜 한 번에 다 만들면 위험할까?

겉으론 빨라 보여.

근데 실제론 느리다.

왜냐면 네 가지가 동시에 커지거든.

- 코드 양
- 파일 수
- 에러 지점
- 판단 부담

그리고 제일 치명적인 건
어디서부터 잘못됐는지 모르게 된다는 거야.

한 번에 다 만들면 결과가 커서,
좋은지 나쁜지 판단도 흐려진다.

---

## "한 입씩"은 정확히 무슨 뜻이야?

기능 하나를
화면부터 동작까지
끝까지 만들어보고 확인하는 거야.

예를 들어 캐릭터 챗이라면 이런 식이다.

1. 캐릭터 선택 카드만 만든다
2. 입력창만 연결한다
3. 응답 표시만 붙인다
4. API 연결을 붙인다
5. 마지막으로 다듬는다

각 단계가 끝날 때마다 직접 보고 확인한다.

이게 중요하다.

확인 없는 구현은 그냥 누적이야.

---

## 캐릭터 챗을 한 입 크기로 자르면 어떻게 될까?

이게 실전 감각을 만드는 예시다.

나쁜 순서는 이렇다.

1. 캐릭터 목록
2. 채팅 UI
3. API 연결
4. 대화 저장
5. 모바일 대응
6. 설정 화면

이걸 한 번에 다 시키는 것.

좋은 순서는 이렇다.

1. 캐릭터 카드 한 개만 보이게 하기
2. 입력창과 말풍선 껍데기만 만들기
3. 응답 자리 표시 텍스트 넣기
4. 실제 API 연결 붙이기
5. 오류 문구 처리하기

좋은 순서는 매 단계마다 확인할 수 있다.

그래서 빨리 틀릴 수 있고,
빨리 고칠 수 있다.

---

## 왜 이 방식을 수직 슬라이싱이라고 할까?

이 이름을 외울 필요는 없지만 개념은 좋다.

기능 하나를 위아래로 끝낸다는 뜻이야.

예를 들어 버튼 하나를 만든다고 해도:

- 화면에 보이고
- 눌렀을 때 반응하고
- 필요한 데이터가 연결되는 것까지

그 한 조각을 완성한다.

전체 화면만 다 그려놓고,
나중에 기능 붙이고,
맨 끝에 데이터 붙이는 방식보다 낫다.

왜냐면 중간에 "이거 아닌데?"를 더 빨리 알 수 있기 때문이다.

---

## 적당한 한 입 크기는 어느 정도야?

내 기준은 이거야.

**10분 안에 확인 가능한 크기.**

너무 크면 전체 만들기와 다를 게 없고,
너무 작으면 진도가 안 나간다.

예를 들어:

- 적당함: "신청 버튼 눌렀을 때 완료 문구 보이게 해줘"
- 너무 큼: "신청 시스템 전체 만들어줘"
- 너무 작음: "버튼 글자 한 글자 바꿔줘"

이 감각은 몇 번 해보면 금방 생긴다.

---

## 기능 하나 끝나면 뭘 해야 해?

네 가지다.

1. 브라우저에서 확인
2. 이상하면 바로 수정
3. 저장
4. 지금 상태를 짧게 정리

정리가 중요한 이유는,
AI도 긴 대화 속에서 전에 한 걸 점점 흐리게 보기 때문이야.

짧은 정리 문서가 있으면 다음 기능으로 넘어갈 때 훨씬 안정적이다.

여기서 정리 문서는 길 필요 없다.

예를 들면 이렇게만 남겨도 충분하다.

- 오늘 만든 기능
- 바뀐 파일
- 아직 안 된 것
- 다음 순서

이 네 줄만 있어도 다음 대화가 훨씬 덜 흔들린다.

---

## 한 입씩 만들 때 제일 좋은 질문은 뭐야?

이거다.

> "지금 만들려는 이 조각이
> 10분 안에 확인 가능한가?"

아니면 아직 큰 거다.

더 잘라야 한다.

---

## 이 방식의 진짜 장점은 뭐야?

### 1. 에러가 좁아진다

방금 만든 기능 안에서 찾으면 되니까.

### 2. 되돌리기가 쉽다

저장 단위가 작아서.

### 3. 방향 수정이 쉽다

프로젝트 전체가 아니라 다음 한 조각만 틀면 되니까.

결국 "한 입씩"은 속도 전략이기도 하고,
안전 전략이기도 하다.

---

## 한 줄 정리

만들고,
확인하고,
저장하고,
다음으로 간다.

바이브 코딩은 이 리듬을 지키는 사람이 이긴다.
