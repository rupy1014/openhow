> 빨간 글씨는 적이 아닙니다. 대부분은 안내판입니다.

에러. 그런데 에러는 없앨 대상이 아니라 읽을 대상입니다.

이 감각이 생기면 AI도 훨씬 잘 쓸 수 있습니다.

---

## 완성 기준

- [ ] 에러 메시지에서 종류, 설명, 위치 세 가지를 찾을 수 있다
- [ ] 스택 트레이스에서 내 파일을 먼저 찾는 습관이 잡힌다
- [ ] AI에게 코드 + 에러 + 의도를 함께 전달할 수 있다
- [ ] "왜?"를 물어서 원인까지 이해할 수 있다

---

## STEP 1. 에러 메시지 읽기

### 1-1. 세 가지만 찾으세요

01 **에러 종류** — `TypeError`, `SyntaxError`, `ReferenceError` 같은 앞쪽 키워드를 확인합니다.

02 **설명 문장** — `cannot read properties of undefined` 같은 구체적 메시지를 읽습니다.

03 **파일과 줄 번호** — `page.tsx:14` 같은 위치 정보를 찾습니다.

예를 들어 이런 에러가 나왔다고 해보겠습니다.

```
TypeError: cannot read properties of undefined (reading 'name')
    at Dashboard (page.tsx:14)
```

이 세 줄만 봐도 "아, 14번째 줄에서 값이 비어 있구나" 하는 감이 옵니다.

> **확인**: 에러 메시지에서 종류, 설명, 위치를 바로 찾을 수 있으면 성공입니다.

<!-- IMG: [에러 메시지 구조 분석 — 종류/설명/위치가 화살표로 표시된 예시] -->

### 1-2. 스택 트레이스 읽기

스택 트레이스는 줄줄이 길게 나오니까 무서워 보입니다. 그런데 핵심은 보통 윗부분에 있습니다.

01 내 파일 이름과 줄 번호가 나오는 부분을 먼저 찾으세요.

02 전부 이해하려 하지 말고, 내 파일이 어디서 처음 등장하는지부터 보면 됩니다.

범인의 발자국 같은 겁니다. 첫 번째 내 파일이 가장 중요한 단서입니다.

> **확인**: 스택 트레이스에서 내 파일명을 1분 안에 찾을 수 있으면 충분합니다.

---

## STEP 2. AI에게 제대로 질문하기

### 2-1. 나쁜 질문과 좋은 질문

01 나쁜 질문을 먼저 보겠습니다.

> "안 돼. 고쳐줘."

이건 정보가 너무 적습니다. AI도 찍어야 하죠.

02 좋은 질문은 세 가지를 같이 줍니다.

- **관련 코드** — 문제가 생긴 파일의 해당 부분
- **에러 메시지** — 터미널이나 콘솔에 나온 전체 메시지
- **원래 의도** — 내가 하려던 게 뭐였는지

예시를 보세요.

> "`page.tsx` 14번째 줄에서 TypeError가 납니다. 로그인 직후 대시보드로 이동했고, 사용자 이름이 보여야 하는데 비어 있습니다."

이 차이가 정말 큽니다. AI 디버깅은 점쟁이 부르기가 아니라, 맥락을 주는 대화입니다.

> **확인**: 에러가 났을 때 코드 + 에러 + 의도를 한 번에 전달할 수 있으면 성공입니다.

### 2-2. "왜?"를 묻는 습관

01 AI가 고친 코드를 받았다고 바로 끝내면 안 됩니다.

02 왜 그렇게 고쳐야 하는지 물어보세요.

> "원래 뭐가 문제였어?"

> "이 수정은 왜 필요한 거야?"

03 이유까지 알면 다음엔 훨씬 빨라집니다. 그냥 복붙하면 다음 에러 때 또 처음부터 흔들립니다.

> **확인**: AI가 수정한 코드에 대해 "왜?"를 최소 한 번은 물어볼 수 있으면 좋습니다.

---

## STEP 3. 디버깅은 추리입니다

### 3-1. 디버깅의 흐름

01 단서를 모읍니다 — 에러 메시지, 콘솔 로그, 화면 상태를 확인합니다.

02 가설을 세웁니다 — "아마 이 변수가 비어 있어서 그런 것 같다"처럼 추측합니다.

03 확인합니다 — `console.log`를 찍거나, AI에게 "이 값이 undefined일 수 있나?"라고 물어봅니다.

04 맞으면 고치고, 아니면 다시 단서부터 봅니다.

AI는 여기서 좋은 조수가 될 수 있습니다. 대신 탐정은 여러분이어야 합니다.

어디가 의심스러운지, 무슨 상황에서 터졌는지, 무엇을 기대했는지. 그걸 말해주는 사람은 결국 여러분입니다.

디버깅은 단순한 수리가 아니라 상황 설명 능력과도 연결됩니다.

## 여기서 막혔나요?

**Q. 에러 메시지가 영어라서 읽기 어렵습니다.**

상황: 콘솔에 영어 에러가 길게 나와서 겁이 납니다.

원인:
- 영어 자체가 어려운 게 아니라, 어디를 봐야 할지 모르는 경우가 많습니다

해결:
1. 에러 메시지 전체를 복사해서 AI에게 "이 에러 한국어로 설명해줘"라고 요청하세요
2. 종류(TypeError 등) + 파일명 + 줄 번호, 이 세 가지만 먼저 찾는 연습을 하세요

**Q. AI가 고쳐줬는데 또 다른 에러가 납니다.**

상황: 하나 고쳤더니 다른 에러가 연쇄적으로 나옵니다.

원인:
- 원래 여러 개의 문제가 겹쳐 있었을 수 있습니다
- AI가 한 곳만 고치면서 다른 곳의 의존성이 깨진 경우도 있습니다

해결:
1. 새 에러도 동일한 방법으로 종류/설명/위치를 먼저 파악하세요
2. AI에게 "방금 수정 전후의 변경점과 새 에러를 함께 보여줄게"라고 맥락을 같이 주세요

**Q. 콘솔은 어디서 봐야 하나요?**

상황: "콘솔에서 확인하세요"라는 말이 나왔는데 어디를 열어야 할지 모릅니다.

원인:
- 브라우저 콘솔과 터미널 콘솔이 다른 곳입니다

해결:
1. **브라우저 콘솔**: Chrome에서 `F12` 또는 `Cmd+Option+I` → Console 탭
2. **터미널 콘솔**: `pnpm dev`나 `npm run dev`를 실행한 터미널 창을 확인하세요
3. 프론트엔드 에러는 브라우저, 서버 에러는 터미널에 나옵니다

<!-- IMG: [브라우저 개발자도구 Console 탭과 터미널 에러 메시지 비교 스크린샷] -->

**Q. console.log는 어떻게 찍나요?**

상황: AI가 "로그를 찍어보세요"라고 하는데 방법을 모릅니다.

원인:
- 디버깅의 가장 기본 도구인데, 처음엔 낯설 수 있습니다

해결:
1. 의심되는 줄 바로 위에 `console.log("여기 도착", 변수이름)` 을 추가하세요
2. 브라우저 콘솔이나 터미널에서 출력 결과를 확인하세요
3. 확인이 끝나면 `console.log`는 지워주세요

## 한 줄 정리

에러는 읽는 거고, 디버깅은 질문하는 겁니다. 관련 코드, 에러, 의도를 같이 주고 "왜?"를 묻는 순간 AI는 훨씬 좋은 디버깅 파트너가 됩니다.
