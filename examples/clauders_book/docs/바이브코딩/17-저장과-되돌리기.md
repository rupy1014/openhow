> 바이브 코딩에서 제일 무서운 건 에러가 아닙니다.
>
> 돌아갈 지점이 없는 겁니다.

왜냐하면 결국 한 번은 망가지거든요.

이건 저주가 아니라 사실입니다.

그래서 중요한 건 "망가지지 않는 법"이 아니라,
**망가졌을 때 돌아가는 법**입니다.

이번 장에서는 `git`으로 저장하고, 되돌리는 실전 흐름을 익혀보겠습니다.

---

## 완성 기준

- [ ] `git init`으로 프로젝트에 git을 연결할 수 있다
- [ ] 기능 하나를 완성한 뒤 `git add` + `git commit`으로 저장할 수 있다
- [ ] `git log`로 저장 이력을 확인할 수 있다
- [ ] `git checkout`으로 이전 시점으로 되돌릴 수 있다
- [ ] GitHub에 코드를 백업(`push`)할 수 있다

---

## STEP 1. git이 뭔지 이해하기

### 1-1. 가장 쉬운 비유

가장 쉬운 비유는 이겁니다.

**무한 세이브 슬롯.**

지금 상태를 저장해두고, 나중에 필요하면 그 지점으로 돌아갈 수 있게 해주는 도구죠.

워드에서 저장하면 이전 상태가 덮어씌워집니다. 그런데 `git`은 다릅니다. "어느 시점에 어떤 변경이 있었는지"가 전부 남습니다. 그래서 이력이 생기는 거죠.

이 이력은 AI랑 일할 때 특히 강력합니다. AI가 바꾼 것도 기록으로 남기기 쉽거든요.

### 1-2. 왜 저장이 그렇게 중요할까요?

AI는 빠릅니다. 좋은 뜻으로도 빠르고, 망가지는 것도 빠릅니다.

기능 하나를 고치다가 다른 데까지 건드릴 수 있고, 고치려다 더 꼬일 수도 있습니다.

이때 저장 지점이 없으면 정말 난감해집니다. 뭐가 바뀌었는지도 모르고, 어디까지 돌아가야 하는지도 모르거든요.

그래서 기능 하나가 끝날 때마다 저장해야 합니다. 게임에서 보스전 전에 세이브하듯이요.

## STEP 2. 저장하기 — add와 commit

### 2-1. 프로젝트에 git 연결하기

01 터미널에서 프로젝트 폴더로 이동합니다.

```bash
cd my-project
```

02 git을 초기화합니다.

```bash
git init
```

> **확인**: `Initialized empty Git repository`라는 메시지가 나오면 성공입니다.

### 2-2. 변경 사항 저장하기

기능 하나를 완성했다면, 이 두 명령어를 순서대로 입력하세요.

01 변경된 파일을 저장 대기열에 올립니다.

```bash
git add .
```

02 저장 메시지와 함께 확정합니다.

```bash
git commit -m "캐릭터 선택 화면 완성"
```

이게 한 번의 "세이브"입니다.

<!-- IMG: [터미널에서 git add, git commit 실행 화면] -->

### 2-3. 언제 저장해야 할까요?

기준은 단순합니다. **기능 하나 끝날 때마다.**

예를 들면:

- 캐릭터 선택 화면 완성
- 채팅 입력창 연결
- 응답 표시 완료

이런 단위마다 저장하세요.

세 기능을 몰아서 한 번에 저장하지 마세요. 나중에 되돌릴 때 너무 큰 덩어리가 되기 때문입니다.

저장은 자주, 메시지는 짧고 분명하게. 이게 좋은 습관입니다.

> **확인**: `git log`를 입력했을 때 방금 남긴 메시지가 보이면 성공입니다.

---

## STEP 3. 되돌리기 — 진짜 실력이 여기서 나옵니다

### 3-1. 되돌리기가 왜 중요할까요?

초보분들은 되돌리면 실패 같다고 느끼곤 합니다.

아닙니다. 잘 되돌리는 게 오히려 실력입니다.

왜냐하면 계속 꼬인 상태에서 버티는 게 더 느리거든요. 특히 같은 문제를 세 번쯤 고쳤는데도 점점 더 이상해진다면, 그건 밀어붙일 때가 아니라 돌아갈 때입니다.

예를 들어:

- 버튼 하나 고치다가 화면 전체가 깨졌다
- API 연결 손보다가 아까 되던 입력창까지 죽었다

이럴 땐 "조금만 더 보면 되겠지"보다 **"아까 되던 지점으로 돌아가자"**가 훨씬 낫습니다.

### 3-2. 이력 확인하기

01 저장 이력을 확인합니다.

```bash
git log --oneline
```

이렇게 하면 커밋 목록이 한 줄씩 나옵니다.

```
a1b2c3d 응답 표시 완료
e4f5g6h 채팅 입력창 연결
i7j8k9l 캐릭터 선택 화면 완성
```

<!-- IMG: [git log --oneline 실행 결과 화면] -->

### 3-3. 이전 시점으로 되돌리기

01 돌아가고 싶은 커밋의 해시(앞 7자리)를 복사합니다.

02 아래 명령어를 입력합니다.

```bash
git checkout e4f5g6h
```

이러면 "채팅 입력창 연결" 시점으로 돌아갑니다.

03 확인이 끝나면 다시 최신 상태로 돌아옵니다.

```bash
git checkout main
```

> **확인**: 돌아간 시점의 코드가 정상적으로 보이면 성공입니다.

## STEP 4. GitHub에 백업하기

### 4-1. GitHub은 왜 같이 쓸까요?

`git`이 내 컴퓨터 안의 세이브라면, GitHub은 그걸 인터넷에 올려두는 백업 공간입니다.

장점은 명확합니다:

- 컴퓨터가 바뀌어도 이어서 할 수 있습니다
- 배포 도구와 연결하기 쉽습니다
- 협업할 때 기준점이 생깁니다

### 4-2. GitHub에 올리기

01 GitHub에서 새 저장소(repository)를 만듭니다.

<!-- IMG: [GitHub에서 New Repository 버튼 클릭 화면] -->

02 터미널에서 원격 저장소를 연결합니다.

```bash
git remote add origin https://github.com/내아이디/my-project.git
```

03 코드를 올립니다.

```bash
git push -u origin main
```

> **확인**: GitHub 페이지를 새로고침했을 때 내 코드가 보이면 성공입니다.

<!-- IMG: [GitHub 저장소에 코드가 올라간 화면] -->

## STEP 5. 좋은 저장 습관 만들기

이 리듬이면 충분합니다.

1. 기능 하나 만든다
2. 직접 확인한다
3. `git add .` + `git commit -m "메시지"`로 저장한다
4. 다음 기능으로 넘어간다

이 리듬이 있으면 프로젝트가 갑자기 안정됩니다. 속도는 조금 느려 보일 수 있지만, 실제론 훨씬 빠릅니다. 되돌릴 수 있기 때문이죠.

그리고 커밋 메시지도 이렇게 남기면 좋습니다:

- 캐릭터 카드 UI 완성
- 뉴스 요약 필터 추가
- 카카오 발송 함수 연결

나중에 봐도 바로 이해되는 문장이 좋은 메시지입니다.

## 여기서 막혔나요?

**Q. `git init`을 했는데 아무 반응이 없어요.**

상황: 명령어를 입력했는데 에러도 안 나고 메시지도 안 나온다.

원인:
- 이미 git이 초기화된 폴더에서 다시 실행한 경우입니다
- `Reinitialized existing Git repository`라는 메시지가 나왔을 수 있습니다

해결:
1. `git status`를 입력해보세요. 정상적으로 상태가 나오면 이미 연결된 겁니다
2. 문제없이 `add`와 `commit`을 진행하시면 됩니다

**Q. `git commit`을 했는데 "nothing to commit" 이라고 나와요.**

상황: 분명히 파일을 수정했는데 커밋이 안 된다.

원인:
- `git add .`를 먼저 하지 않은 경우입니다
- git은 "저장 대기열에 올린 것"만 커밋합니다

해결:
1. `git add .`를 먼저 실행하세요
2. 그다음 `git commit -m "메시지"`를 다시 시도하세요

**Q. 되돌렸는데 "detached HEAD" 라는 경고가 떠요.**

상황: `git checkout`으로 이전 커밋으로 갔더니 무서운 메시지가 나온다.

원인:
- 정상입니다. "지금 특정 시점을 구경하고 있다"는 뜻이에요
- 브랜치가 아닌 커밋을 직접 체크아웃하면 항상 나오는 메시지입니다

해결:
1. 구경만 할 거라면 그냥 확인하시면 됩니다
2. 다시 최신으로 돌아가려면 `git checkout main`을 입력하세요
3. 이 시점에서 새로 작업하고 싶다면 `git checkout -b 새브랜치이름`으로 브랜치를 만드세요

**Q. 커밋 메시지를 잘못 적었어요. 고칠 수 있나요?**

상황: "asdf"라고 커밋해버렸다.

원인:
- 급하게 저장하다 보면 자주 생기는 일입니다

해결:
1. 바로 직전 커밋이라면 `git commit --amend -m "올바른 메시지"`로 수정할 수 있습니다
2. 이전 커밋이라면 그냥 두셔도 됩니다. 다음부터 잘 적으면 충분합니다

## 한 줄 정리

저장은 귀찮은 절차가 아니라 보험입니다. 기능 하나 끝날 때마다 세이브 포인트를 찍으세요. 그래야 망가져도 계속 갈 수 있습니다.
