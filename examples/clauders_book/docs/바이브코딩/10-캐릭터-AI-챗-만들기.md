> 첫 앱은 거창할 필요 없습니다.
>
> "대화가 된다"까지만 가면 됩니다.

이 장에서 만들 첫 프로젝트는 **캐릭터 AI 챗**입니다.

이걸 첫 프로젝트로 고른 이유는 단순합니다. **성과가 빨리 보이는데, 배울 것도 많기 때문**이죠.

입력창이 있고, 응답이 오고, 캐릭터 성격이 있고, 화면도 있고, 배포까지 연결하기 좋습니다.

---

## 완성 기준

- [ ] 캐릭터 한 명의 소개가 화면에 표시된다
- [ ] 사용자가 메시지를 입력하고 전송할 수 있다
- [ ] AI가 캐릭터 말투로 응답을 돌려준다
- [ ] API 키가 프론트엔드 코드에 노출되지 않는다
- [ ] 한 화면에서 대화가 오간다 (페이지 이동 없음)

처음부터 캐릭터 20명, 대화 기록 저장, 로그인, 결제, 음성 대화 같은 걸 꿈꾸면 안 됩니다. 이건 나중 얘기입니다.

이 장의 목표는 훨씬 작습니다. **캐릭터 하나를 고르고, 메시지를 보내고, 답을 받는다.** 거기까지 가면 성공입니다.

첫 프로젝트는 성공 기준이 작아야 합니다. 그래야 진짜 끝까지 갈 수 있거든요.

---

## 왜 하필 캐릭터 챗일까?

이 프로젝트는 바이브 코딩의 핵심을 다 보여줍니다.

### 기획이 중요하다는 걸 바로 느끼게 해줍니다

캐릭터 성격을 어떻게 잡느냐에 따라 결과가 완전히 달라집니다. 기술보다 설정이 더 중요하다는 걸 여기서 바로 체감하게 되죠.

### "안 만들 것"이 얼마나 중요한지 드러납니다

로그인 하나만 붙어도 난이도가 확 올라갑니다. 그래서 MVP 자르기 훈련에 딱 좋습니다.

### 눈에 보이는 결과가 빨리 나옵니다

대화가 오가기 시작하면 만족감이 큽니다. 초반엔 이 만족감이 정말 중요합니다.

---

## STEP 1. 프로젝트 기획 잡기

### 1-1. 최소 구성 요소 확인하기

정말 최소로 줄이면 이 다섯 가지면 됩니다.

1. 캐릭터 소개 한 줄
2. 대화 화면 한 장
3. 사용자 입력창
4. AI 응답 영역
5. API 호출 연결

이것만 돌아가면 이미 앱입니다. 대부분 여기서 놀랍니다. "이 정도만 있어도 진짜 서비스처럼 느껴지네?" 맞습니다. 처음엔 그 정도면 충분합니다.

### 1-2. 첫 프롬프트 작성하기

많은 분이 여기서 바로 이렇게 시작합니다.

> "AI 캐릭터 채팅 서비스 만들어줘."

그러면 보통 과해집니다. 로그인도 붙고, 캐릭터 관리 화면도 붙고, 쓸데없이 많은 파일이 생기죠.

처음엔 이렇게 가는 게 낫습니다.

```
AI 캐릭터 채팅 MVP를 만들고 싶어.

캐릭터는 한 명만.
한 화면에서 대화만 되면 돼.
로그인, 저장, 관리자 기능은 빼.

먼저 PRD 초안부터 정리해줘. 코드는 아직 짜지 마.
```

이 프롬프트가 좋은 이유는 세 가지입니다.

01 **MVP라고 못을 박습니다** — AI가 범위를 넓히지 않습니다.

02 **"한 명만"이라고 범위를 자릅니다** — 캐릭터가 여러 명이면 테스트 복잡도가 같이 늘어나거든요.

03 **안 만들 것을 먼저 말합니다** — 이게 가장 중요합니다. "빼줘"라고 명시하지 않으면 AI는 다 넣으려고 합니다.

> **확인**: Codex가 PRD 초안을 정리해서 돌려줬다면 이 단계는 성공입니다.

## STEP 2. 캐릭터 설정 잡기

### 2-1. 캐릭터를 한 명만 정하기

다섯 명 만들지 마세요. 한 명만 정하세요.

말투, 성격, 금지사항도 한 명 기준으로 먼저 잡습니다. 캐릭터가 여러 명이 되면 테스트 복잡도도 같이 늘어납니다.

### 2-2. 캐릭터 설명문 작성하기

캐릭터 챗에서 제일 큰 차이는 모델이 아니라 **캐릭터 설명문**에서 나는 경우가 많습니다.

**나쁜 예:**

> "귀여운 캐릭터로 해줘."

이건 너무 흐립니다.

**더 좋은 예:**

```
20대 초반 느낌의 밝은 캐릭터야.
반말을 쓰고,
너무 길게 설명하지 않고,
한 번에 3문장 이상 말하지 마.
위로는 해주되 교훈적인 말투는 피하고,
한국 메신저 대화처럼 짧고 자연스럽게 답해.
```

이 정도면 결과가 확 달라집니다. 특히 아래 항목이 중요합니다.

- **말투** — 반말? 존댓말? 이모티콘 쓸까?
- **문장 길이** — 한 번에 몇 문장까지?
- **감정 온도** — 쿨한 톤? 다정한 톤?
- **하지 말아야 할 태도** — "설교하지 마", "질문 반복하지 마" 등
- **세계관** — 있으면 한 줄로

캐릭터는 프롬프트 디자인이 곧 제품 품질입니다.

> **확인**: 캐릭터 설명문이 최소 5줄 이상이면 이 단계는 성공입니다.

## STEP 3. 한 화면으로 만들기

### 3-1. 화면을 한 장만 만들기

목록 페이지, 상세 페이지, 설정 페이지를 만들지 마세요. 한 화면에서 다 되게 합니다.

초반엔 화면 수가 적을수록 좋습니다.

### 3-2. 대화 저장은 빼기

새로고침하면 사라져도 됩니다. 지금은 핵심 경험 확인이 먼저입니다.

### 3-3. API 키는 화면 코드에 직접 넣지 않기

이건 꼭 기억하세요. AI 응답을 받으려면 외부 API를 쓰게 되는데, 비밀 키가 노출되면 안 됩니다.

그래서 이 프로젝트는 아주 단순한 구조라도 **화면과 서버 역할을 나눠야** 합니다. 이게 뒤의 기술 선택 장과도 연결됩니다.

> **확인**: 한 화면에서 메시지 입력 → AI 응답이 표시되면 이 단계는 성공입니다.

## STEP 4. 톤 수정 루프 돌리기

### 4-1. 첫 완성 뒤에 바로 말투 다듬기

기능이 돌아가면 끝이 아닙니다. 바로 톤 수정 루프로 들어가야 합니다.

예를 들면 이런 식으로 Codex에게 요청하세요.

```
답이 너무 설명체야. 더 짧게 해줘.
```

```
위로는 좋은데 너무 교훈적이야. 덜 가르치듯 말해줘.
```

```
반말은 맞는데 너무 공격적이야. 더 부드럽게.
```

이 수정이 제품의 인상을 바꿉니다. 기능은 되는데 대화가 그냥 일반 챗봇처럼 느껴진다면, 대체로 캐릭터 정의가 너무 약한 경우입니다.

> **확인**: 캐릭터 톤을 2~3번 수정해서 "이 캐릭터답다"는 느낌이 들면 성공입니다.

## 이 프로젝트에서 배울 진짜 포인트

코드보다 더 중요한 건 네 가지입니다.

### 1. 기획이 결과를 바꿉니다

같은 모델을 써도 캐릭터 정의가 다르면 완전히 다른 앱이 됩니다.

### 2. 핵심 경험만 남기면 빨라집니다

대화 하나가 돌아가면 나머지는 나중 문제입니다.

### 3. 보이는 결과가 나오면 피드백이 가능해집니다

사람은 문서보다 실제 화면을 보고 더 잘 말합니다. "이 캐릭터는 너무 딱딱해.", "답이 너무 길어.", "입력창이 너무 작아." 이런 피드백은 앱이 떠야만 나옵니다.

### 4. 재미도 품질 항목입니다

캐릭터 챗은 기능만 맞다고 끝나지 않습니다. 사람이 다시 열고 싶어야 하죠. 즉, 몰입감도 품질입니다. 이 장에서 그 감각을 처음 배우게 됩니다.

---

## 첫 실전에서 제일 많이 하는 실수

### 캐릭터보다 시스템을 먼저 크게 잡는다

로그인, 결제, 저장, 관리 화면부터 생각하면 바로 무거워집니다.

### 한 번에 여러 캐릭터를 넣는다

비교는 좋지만 초반엔 복잡도만 늘립니다.

### 결과가 이상한데도 구조를 계속 쌓는다

한 번 대화해보고 캐릭터 톤이 틀렸으면, 그때 바로 고쳐야 합니다. 안 그러면 나중에 전부 다시 손보게 됩니다.

### 응답은 오는데 캐릭터가 없다

기능은 되는데 대화가 그냥 일반 챗봇처럼 느껴지는 거죠. 이건 대체로 캐릭터 정의가 너무 약한 경우입니다.

## 여기서 막혔나요?

**Q. Codex가 PRD 대신 바로 코드를 짜기 시작합니다.**

상황: "PRD 초안부터 정리해줘"라고 했는데 코드부터 나옵니다.

원인:
- 프롬프트 앞부분에 "만들고 싶어"가 있으면 Codex가 구현으로 넘어가는 경향이 있습니다.

해결:
1. 프롬프트 마지막에 **"코드는 아직 짜지 마."**를 반드시 넣으세요.
2. 그래도 코드가 나오면 "아직 코드 단계 아니야. PRD 문서만 정리해줘."라고 한 번 더 말하세요.

---

**Q. API 키를 어디에 넣어야 할지 모르겠습니다.**

상황: OpenAI나 Anthropic API 키가 필요한데 코드 어디에 넣어야 할지 모릅니다.

원인:
- 프론트엔드 코드에 직접 넣으면 브라우저에서 키가 노출됩니다.

해결:
1. `.env` 파일에 `API_KEY=sk-xxx` 형태로 저장하세요.
2. 서버 쪽 코드(백엔드/API 라우트)에서만 이 키를 읽도록 하세요.
3. Codex에게 "API 키가 프론트엔드에 노출되지 않게 해줘"라고 명시하면 알아서 분리해줍니다.

---

**Q. 캐릭터가 설정한 말투를 안 따릅니다.**

상황: "반말로 짧게 답해"라고 했는데 존댓말에 긴 설명이 나옵니다.

원인:
- 시스템 프롬프트가 너무 짧거나 모호합니다.
- "짧게"만 쓰면 모델마다 해석이 다릅니다.

해결:
1. "한 번에 2문장 이내"처럼 **숫자로 제한**하세요.
2. "~하지 마" 형태의 **금지 조건**을 명확히 넣으세요.
3. 예시 대화를 2~3개 포함하면 말투 준수율이 크게 올라갑니다.

---

**Q. 대화가 오가긴 하는데 화면이 안 예쁩니다.**

상황: 기능은 되는데 채팅 UI가 엉성합니다.

원인:
- 초기 MVP에서는 정상입니다. 걱정하지 마세요.

해결:
1. 먼저 대화 흐름이 자연스러운지 확인하세요. 기능이 우선입니다.
2. 톤 수정까지 끝난 뒤에 "채팅 UI를 카카오톡 스타일로 예쁘게 바꿔줘"라고 요청하세요.
3. CSS는 가장 마지막에 다듬는 게 효율적입니다.

---

## 이 장 다음엔?

바로 여기서 자연스럽게 질문이 나옵니다.

> "이걸 만들긴 하겠는데, 그 전에 뭘 먼저 정리해야 덜 망하지?"

그 질문의 답이 다음 장들입니다. 70%는 기획이고, 아이디어는 가설이어야 하고, PRD는 한 장이어야 하고, "안 만들 것"이 분명해야 하고, 기술은 가장 단순한 걸 골라야 합니다.

이 장은 첫 실전의 맛보기이고, 다음 장들은 여러분이 실제로 끝까지 갈 수 있게 만드는 안전장치입니다.

---

## 한 줄 정리

첫 앱의 목표는 멋진 서비스가 아닙니다. 캐릭터 하나와 대화가 된다 — 거기까지 가면, 진짜 시작할 수 있습니다.
