> 한 번에 다 만들지 마세요. 한 입씩 만드세요.

이 원칙 하나만 지켜도 성공 확률이 크게 올라갑니다.

입문자가 제일 많이 무너지는 지점이 바로 여기거든요.

"전체 서비스 만들어줘."

이 한 문장이 모든 사고의 시작일 때가 많습니다. 이번 장에서는 "한 입씩 만들기"를 실제로 어떻게 적용하는지, 캐릭터 챗 예시를 통해 직접 체감해보겠습니다.

---

## 완성 기준

- [ ] "한 번에 다 만들기"가 왜 위험한지 설명할 수 있다
- [ ] 기능 하나를 "한 입 크기"로 쪼개서 프롬프트를 작성할 수 있다
- [ ] 매 단계마다 확인 → 저장 → 정리 루틴을 실행할 수 있다
- [ ] 10분 안에 확인 가능한 크기인지 스스로 판단할 수 있다

---

## STEP 1. 왜 한 번에 다 만들면 위험한지 이해하기

### 1-1. 동시에 커지는 네 가지

겉으로는 빨라 보입니다. 그런데 실제로는 느립니다.

왜냐하면 네 가지가 동시에 커지거든요.

- 코드 양
- 파일 수
- 에러 지점
- 판단 부담

그리고 제일 치명적인 건, 어디서부터 잘못됐는지 모르게 된다는 겁니다.

### 1-2. 판단이 흐려지는 순간

한 번에 다 만들면 결과가 너무 커서, 좋은지 나쁜지 판단도 흐려집니다. "뭔가 이상한데... 어디가 이상한 거지?"라는 상태가 되면, 사실상 처음부터 다시 보는 것과 같습니다.

> **확인**: "한 번에 다 만들기"의 위험성을 팀원에게 설명할 수 있으면 이 단계는 통과입니다.

---

## STEP 2. "한 입 크기"로 쪼개기

### 2-1. 한 입씩이란 무슨 뜻인가요?

기능 하나를, 화면부터 동작까지, 끝까지 만들어보고 확인하는 겁니다. 이걸 소프트웨어 업계에서는 "수직 슬라이싱"이라고 부릅니다. 이름은 외울 필요 없지만, 개념은 아주 좋습니다.

기능 하나를 위아래로 끝낸다는 뜻이죠. 예를 들어 버튼 하나를 만든다고 해도:

- 화면에 보이고
- 눌렀을 때 반응하고
- 필요한 데이터가 연결되는 것까지

그 한 조각을 완성합니다.

전체 화면만 다 그려놓고, 나중에 기능 붙이고, 맨 끝에 데이터 붙이는 방식보다 낫습니다. 왜냐하면 중간에 "이거 아닌데?"를 더 빨리 알 수 있기 때문이죠.

### 2-2. 캐릭터 챗을 예시로 쪼개보기

나쁜 순서 -- 한 번에 다 시키는 경우:

```
"캐릭터 목록, 채팅 UI, API 연결, 대화 저장, 모바일 대응, 설정 화면 다 만들어줘"
```

좋은 순서 -- 한 입씩 나누는 경우:

01 캐릭터 카드 한 개만 보이게 하기

02 입력창과 말풍선 껍데기만 만들기

03 응답 자리 표시 텍스트 넣기

04 실제 API 연결 붙이기

05 오류 문구 처리하기

좋은 순서는 매 단계마다 확인할 수 있습니다. 그래서 빨리 틀릴 수 있고, 빨리 고칠 수 있죠.

<!-- IMG: [나쁜 순서 vs 좋은 순서 비교 다이어그램 — 수평 슬라이싱(화면 전체 → 기능 전체 → 데이터 전체) vs 수직 슬라이싱(기능 하나를 화면+기능+데이터까지 완성)] -->

> **확인**: 여러분이 만들고 싶은 프로젝트의 첫 번째 기능을 5단계 이하로 쪼갤 수 있으면 이 단계는 통과입니다.

---

## STEP 3. 적당한 크기 감각 익히기

### 3-1. 10분 룰

적당한 한 입 크기의 기준은 이겁니다.

**10분 안에 확인 가능한 크기.**

너무 크면 전체 만들기와 다를 게 없고, 너무 작으면 진도가 안 나갑니다.

| 크기 | 예시 프롬프트 | 판정 |
|------|-------------|------|
| 적당함 | "신청 버튼 눌렀을 때 완료 문구 보이게 해줘" | O |
| 너무 큼 | "신청 시스템 전체 만들어줘" | X |
| 너무 작음 | "버튼 글자 한 글자 바꿔줘" | X |

이 감각은 몇 번 해보면 금방 생깁니다.

### 3-2. 스스로 점검하는 질문

다음 프롬프트를 보내기 전에, 이 질문을 스스로에게 던져보세요.

> "지금 만들려는 이 조각이 10분 안에 확인 가능한가?"

아니라면 아직 큰 겁니다. 더 잘라야 합니다.

> **확인**: 프롬프트를 보내기 전에 "10분 안에 확인 가능한가?"를 스스로 물어보는 습관이 생기면 이 단계는 통과입니다.

---

## STEP 4. 확인 → 저장 → 정리 루틴 만들기

### 4-1. 기능 하나가 끝나면 할 일 네 가지

01 브라우저에서 확인합니다

02 이상하면 바로 수정합니다

03 저장합니다

04 지금 상태를 짧게 정리합니다

### 4-2. 왜 정리가 중요한가요?

AI도 긴 대화 속에서 이전에 한 일을 점점 흐리게 보기 때문입니다. 짧은 정리 문서가 있으면 다음 기능으로 넘어갈 때 훨씬 안정적이죠.

정리 문서는 길 필요 없습니다. 예를 들면 이렇게만 남겨도 충분합니다:

```markdown
## 진행 상황 메모
- 오늘 만든 기능: 캐릭터 카드 UI
- 바뀐 파일: CharacterCard.tsx, App.tsx
- 아직 안 된 것: API 연결
- 다음 순서: 입력창 연결
```

이 네 줄만 있어도 다음 대화가 훨씬 덜 흔들립니다.

> **확인**: 기능 하나를 완성한 뒤 정리 메모를 작성할 수 있으면 이 단계는 통과입니다.

---

## STEP 5. 한 입씩의 진짜 장점 체감하기

이 방식의 장점은 세 가지입니다.

### 5-1. 에러가 좁아집니다

방금 만든 기능 안에서 찾으면 되니까요. 전체 코드를 뒤질 필요가 없습니다.

### 5-2. 되돌리기가 쉽습니다

저장 단위가 작아서, 문제가 생기면 마지막 저장 지점으로 돌아가면 됩니다.

### 5-3. 방향 수정이 쉽습니다

프로젝트 전체가 아니라 다음 한 조각만 틀면 되니까요.

결국 "한 입씩"은 속도 전략이기도 하고, 안전 전략이기도 합니다.

> **확인**: 세 가지 장점을 자신의 프로젝트에 대입해서 설명할 수 있으면 이 단계는 통과입니다.

## 여기서 막혔나요?

**Q. 어디까지가 "한 입"인지 감이 안 잡혀요.**

상황: 기능을 쪼개야 한다는 건 알겠는데, 어느 수준까지 쪼개야 할지 모르겠습니다.

원인:
- 처음에는 적정 크기에 대한 감각이 없는 게 정상입니다
- 프로젝트마다 적정 크기가 다릅니다

해결:
1. "10분 안에 브라우저에서 확인할 수 있는가?"를 기준으로 삼으세요
2. 확신이 없으면 일단 더 잘게 쪼개세요 -- 너무 작은 건 합치면 되지만, 너무 큰 건 되돌리기 어렵습니다
3. 3~4번 반복하면 자연스럽게 감이 옵니다

**Q. AI한테 한 입씩 시키면 너무 느린 거 아닌가요?**

상황: 한 번에 전체를 시키면 금방 나오는데, 굳이 쪼개는 게 비효율적으로 느껴집니다.

원인:
- 한 번에 만든 결과물은 빠르게 나오지만, 수정에 걸리는 시간이 훨씬 깁니다
- 어디가 잘못됐는지 찾는 시간이 누적되면 전체적으로 더 느려집니다

해결:
1. 한 번에 만든 결과물의 "수정 시간"까지 포함해서 비교해보세요
2. 한 입씩 만들 때 각 단계 완료 시간을 재보면, 총합이 오히려 짧다는 걸 체감하게 됩니다

**Q. 정리 메모를 매번 쓰기 귀찮아요.**

상황: 기능 하나 끝날 때마다 메모를 남기라는데, 귀찮아서 안 하게 됩니다.

원인:
- 정리를 "문서 작성"으로 생각하면 부담이 큽니다

해결:
1. 네 줄(만든 기능 / 바뀐 파일 / 안 된 것 / 다음 순서)만 적으세요 -- 30초면 됩니다
2. AI에게 "지금까지 한 거 정리해줘"라고 시키는 것도 좋은 방법입니다
3. 정리 없이 다음 기능으로 넘어가면 AI가 맥락을 잃어서 결국 더 많은 시간이 걸립니다

**Q. 수직 슬라이싱이라는 개념이 잘 이해가 안 돼요.**

상황: 수직/수평이 뭔지는 알겠는데 실제로 어떻게 적용하는지 모르겠습니다.

원인:
- 추상적인 개념이라 실제 예시 없이는 와닿기 어렵습니다

해결:
1. 수평 = "화면 전체 먼저 → 기능 전체 → 데이터 전체" 순으로 층(layer)을 쌓는 방식입니다
2. 수직 = "기능 하나를 화면+기능+데이터까지 한 번에 끝내는" 방식입니다
3. 캐릭터 챗 예시를 떠올려보세요 -- 카드 UI만 그리고 끝내는 게 아니라, 카드 하나가 클릭되고 반응까지 되는 걸 먼저 만드는 거죠
