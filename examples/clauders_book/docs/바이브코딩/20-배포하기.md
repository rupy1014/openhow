> 배포 전까지는 아직 반쪽입니다. URL이 나와야 진짜 시작이거든요.

로컬에서만 돌아가는 앱은 아직 여러분만의 실험입니다.
배포가 되면 그때부터 다른 사람도 볼 수 있고, 피드백이 생기죠.

즉, 배포는 마지막 단계가 아니라 **검증의 시작점**입니다.

이 장에서는 처음으로 내 앱을 인터넷에 올려보겠습니다.
Vercel을 사용해서 GitHub 저장소를 연결하고, 배포 URL을 받아 직접 확인하는 것까지가 목표입니다.

---

## 완성 기준

- [ ] GitHub에 프로젝트 코드가 올라가 있다
- [ ] Vercel에 가입하고 GitHub 저장소를 연결했다
- [ ] 배포 URL이 생성되어 브라우저에서 열린다
- [ ] 모바일에서도 배포된 앱을 확인했다
- [ ] 코드 수정 후 재배포가 자동 반영되는 것을 확인했다

---

## STEP 1. 배포가 뭔지 이해하기

### 1-1. 배포 = 내 앱에 인터넷 주소를 붙이는 것

`localhost`는 여러분만 볼 수 있습니다.
배포하면 링크가 생기고, 그 링크를 보내면 누구든 쓸 수 있게 됩니다.

이 차이가 정말 큽니다.
문서로 설명하는 것보다, 링크 하나 보내는 게 훨씬 강력하거든요.

<!-- IMG: [localhost vs 배포 URL 비교 — 내 컴퓨터에서만 열리는 localhost:3000과 누구나 접속 가능한 배포 URL의 차이를 보여주는 다이어그램] -->

### 1-2. 왜 배포를 빨리 해보는 게 좋을까?

초보일수록 배포를 나중으로 미루고 싶어 합니다.
그런데 그게 오히려 손해입니다.

배포를 빨리 해보면 바로 알게 되거든요.

- 환경 변수는 제대로 넣었는지
- 내 컴퓨터에서만 되던 건 아닌지
- 다른 사람의 폰에서도 보이는지
- 정말로 공유 가능한 상태인지

배포는 기술 과시가 아니라 **현실 점검**입니다.

## STEP 2. 배포 전 준비사항 점검하기

### 2-1. 코드가 GitHub에 올라가 있는지 확인하기

01 요즘 배포 흐름은 GitHub 기준으로 이어지는 경우가 대부분입니다.

02 이전 장에서 GitHub에 코드를 올려두었다면 이미 준비된 상태입니다.

03 아직 안 했다면, 터미널에서 아래 명령어로 확인하세요.

```bash
git remote -v
```

> **확인**: `origin`에 GitHub URL이 표시되면 OK입니다.

### 2-2. 비밀 키가 코드에 들어있지 않은지 확인하기

01 이건 정말 중요합니다.

02 API 키를 코드에 직접 넣으면 인터넷에 그대로 노출될 수 있습니다.

03 비밀 값은 반드시 환경 변수(`.env` 파일)로 분리해야 합니다.

```bash
# .env 파일이 .gitignore에 포함되어 있는지 확인
cat .gitignore | grep .env
```

> **확인**: `.env`가 `.gitignore`에 있으면 OK. 없다면 추가하세요.

### 2-3. 로컬에서 핵심 기능 한 번 확인하기

01 배포 전에 기본 동작은 꼭 확인해야 합니다. 안 되는 걸 배포한다고 저절로 되진 않습니다.

02 확인은 그냥 화면 보기만이 아닙니다. 아래를 한 번씩 해보세요.

- 버튼 눌러보기
- 폼 제출해보기
- 에러 상황 한 번 만들어보기
- 모바일 화면 크기로 보기

> **확인**: 핵심 흐름을 끝까지 한 번 가봤다면 OK입니다.

## STEP 3. Vercel로 배포하기

### 3-1. 왜 Vercel인가?

입문자 입장에서 이유는 단순합니다.

- GitHub와 연결이 쉽고
- 웹앱 배포 흐름이 단순하고
- 무료 범위로도 시작 가능하고
- 코드를 수정하면 자동으로 다시 배포됩니다

중요한 건 "최고의 플랫폼"이 아니라 **"처음 배포하기 쉬운가"**입니다.

### 3-2. Vercel 가입 및 프로젝트 연결

01 [vercel.com](https://vercel.com)에 접속합니다.

02 **Continue with GitHub** 버튼을 클릭해서 GitHub 계정으로 가입합니다.

<!-- IMG: [Vercel 가입 화면 — Continue with GitHub 버튼이 보이는 로그인 페이지] -->

03 **Add New Project** 버튼을 클릭합니다.

04 배포할 GitHub 저장소를 선택합니다.

<!-- IMG: [Vercel Import 화면 — GitHub 저장소 목록에서 프로젝트를 선택하는 화면] -->

05 Framework Preset이 자동 감지되었는지 확인합니다. (Next.js, Vite 등)

06 환경 변수가 필요하다면 **Environment Variables** 섹션에 `.env` 파일의 값을 하나씩 입력합니다.

<!-- IMG: [Vercel 환경 변수 설정 화면 — KEY와 VALUE를 입력하는 폼] -->

07 **Deploy** 버튼을 클릭합니다.

> **확인**: 빌드 로그가 나타나고, 잠시 후 `Congratulations!` 화면이 보이면 성공입니다.

<!-- IMG: [Vercel 배포 완료 화면 — Congratulations! 메시지와 함께 미리보기가 표시된 화면] -->

### 3-3. 배포 URL 확인하기

01 배포가 완료되면 `https://프로젝트명.vercel.app` 형태의 URL이 생깁니다.

02 이 URL을 브라우저에 입력하고, 앱이 정상적으로 열리는지 확인하세요.

```
https://my-project.vercel.app
```

> **확인**: 브라우저에서 앱이 열리면 첫 배포 성공입니다!

## STEP 4. 배포 후 점검하기

### 4-1. 배포 직후 체크리스트

01 먼저 여러분이 직접 링크를 열어보세요. 의외로 이걸 안 하는 분이 많습니다.

02 노트북에서는 괜찮았는데 폰에선 줄바꿈이 깨지거나, 버튼이 안 보이거나, 스크롤이 이상한 경우가 꽤 있거든요.

아래 항목을 하나씩 확인하세요.

- 데스크톱 브라우저에서 열기
- **모바일 브라우저에서 열기** (필수!)
- 핵심 기능 1회 실행
- 공유 링크가 제대로 열리는지 확인

<!-- IMG: [데스크톱과 모바일에서 동일한 배포 URL을 열어 비교하는 화면] -->

### 4-2. 수정 → 재배포 흐름 익히기

01 배포 뒤의 좋은 작업 흐름은 이렇습니다.

1. 로컬에서 코드 수정
2. 동작 확인
3. `git add` → `git commit` → `git push`
4. Vercel이 자동으로 재배포
5. 배포 링크에서 다시 확인

02 이렇게 되면 작업이 **프로젝트**처럼 느껴집니다. 즉흥적인 파일 수정이 아니라, 작은 릴리스 단위로 움직이게 되죠.

```bash
# 수정 후 재배포 명령어
git add .
git commit -m "버튼 색상 변경"
git push origin main
# → Vercel이 자동으로 감지해서 재배포합니다
```

> **확인**: push 후 Vercel 대시보드에서 새 빌드가 시작되면 성공입니다.

## 여기서 막혔나요?

**Q. 배포는 됐는데 화면이 하얗게 나와요**

상황: Vercel에서 배포 성공이라고 뜨는데, URL을 열면 빈 화면만 보입니다.

원인:
- 빌드 결과물의 경로가 맞지 않는 경우가 많습니다
- `build` 명령어가 설정되지 않았거나, Output Directory가 다를 수 있습니다

해결:
1. Vercel 프로젝트 Settings > General에서 **Build & Development Settings**를 확인하세요
2. Framework Preset이 올바르게 감지되었는지 확인하세요
3. Output Directory가 맞는지 확인하세요 (예: Vite는 `dist`, Next.js는 `.next`)

---

**Q. 환경 변수를 넣었는데 API가 안 돼요**

상황: 로컬에서는 잘 되던 API 호출이 배포 후에 실패합니다.

원인:
- Vercel에 환경 변수를 입력하지 않았거나, 이름이 다릅니다
- 프론트엔드에서 쓰는 환경 변수는 `NEXT_PUBLIC_` 또는 `VITE_` 접두사가 필요합니다

해결:
1. Vercel 대시보드 > Settings > Environment Variables에서 값이 들어있는지 확인하세요
2. 변수 이름이 `.env` 파일과 정확히 같은지 확인하세요
3. 변수를 추가한 뒤에는 **Redeploy**를 해야 반영됩니다

---

**Q. GitHub에 push했는데 자동 배포가 안 돼요**

상황: 코드를 수정하고 push했는데 Vercel에서 새 빌드가 시작되지 않습니다.

원인:
- Vercel과 GitHub 연결이 끊어졌을 수 있습니다
- push한 브랜치가 배포 대상 브랜치(보통 `main`)가 아닐 수 있습니다

해결:
1. Vercel 대시보드에서 Git 연결 상태를 확인하세요
2. `git branch`로 현재 브랜치를 확인하고, `main` 브랜치로 push했는지 확인하세요
3. 그래도 안 되면 Vercel 대시보드에서 수동으로 **Redeploy** 버튼을 누르세요

---

**Q. 모바일에서 레이아웃이 깨져요**

상황: PC에서는 잘 보이는데 모바일에서 열면 글자가 넘치거나 버튼이 안 보입니다.

원인:
- 반응형(responsive) CSS가 적용되지 않았을 가능성이 높습니다

해결:
1. Codex에게 "모바일에서 레이아웃이 깨지는데, 반응형으로 수정해줘"라고 요청하세요
2. 브라우저 개발자 도구(F12)에서 모바일 뷰로 미리 확인하는 습관을 들이세요
3. 수정 후 다시 push하면 자동으로 재배포됩니다

---

## 한 줄 정리

배포는 끝이 아니라 검증의 시작입니다. GitHub에 올리고, 환경 변수 챙기고, 배포된 링크로 직접 끝까지 써보세요.
