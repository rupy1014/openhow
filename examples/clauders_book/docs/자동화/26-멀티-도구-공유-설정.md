> AI 도구는 여러 개 써도 됩니다. 대신 규칙이 따로 놀면 안 됩니다.

Codex를 메인으로 쓰더라도, 나중엔 다른 도구를 같이 쓰게 되는 경우가 많습니다. 그런데 문제는 여기서 생기죠.

한쪽에서 고친 규칙이 다른 쪽엔 반영 안 되고, 워크플로우가 도구마다 다르게 흩어지기 시작합니다.

이 장에서는 여러 AI 도구를 쓸 때 **규칙 원본을 한 벌로 관리하는 방법**을 실습합니다.

---

## 완성 기준

- [ ] 공용 규칙 파일과 도구 전용 규칙 파일을 분리할 수 있다
- [ ] 규칙 원본을 한 곳에 두고 여러 도구에서 참조하는 구조를 이해했다
- [ ] "어느 파일이 원본인지" 팀원에게 설명할 수 있다

---

## STEP 1. 왜 여러 도구를 같이 쓰게 되는지 이해하기

### 1-1. 현실적인 이유

01 어떤 도구는 터미널 작업이 편합니다

02 어떤 도구는 에디터 안에서 보기 좋습니다

03 어떤 도구는 특정 작업이 더 잘 맞습니다

04 어떤 도구는 팀원이 이미 쓰고 있습니다

하나로 평생 통일되지 않을 수 있고, 그건 자연스러운 일입니다.

문제는 도구가 여러 개인 게 아니라, **규칙 원본이 여러 군데 생기는 것**입니다.

> **확인**: "도구가 여럿인 것 자체는 문제가 아니다"라는 점을 이해했으면 다음으로 넘어가세요.

---

## STEP 2. 문제 상황 파악하기

### 2-1. 규칙이 흩어지면 생기는 일

01 프로젝트 규칙은 한 파일에 있고, 커맨드 정의는 또 다른 폴더에 있습니다

02 담당자 역할 정의도 도구마다 따로 있습니다

03 한쪽만 고치고 다른 쪽은 잊어버립니다

시간이 갈수록 어긋나게 됩니다. 처음엔 별일 아닌 것 같아도, 나중엔 "왜 이 도구는 이렇게 행동하지?"가 되죠. 원인은 대체로 **동기화 실패**입니다.

예를 들어 한쪽 도구에선 "모바일 우선" 규칙이 들어 있고, 다른 쪽 도구에선 빠져 있다고 해보세요. 같은 프로젝트인데 결과가 달라질 수 있습니다. 이건 도구 성능 문제가 아니라 **정본 관리 문제**입니다.

---

## STEP 3. 원본 한 벌 원칙 적용하기

### 3-1. 핵심 원칙

딱 하나입니다. **원본은 한 벌만 두세요.** 그리고 나머지는 그 원본을 바라보게 하면 됩니다.

도구가 여러 개여도 정본은 하나. 그래야 수정할 때도 한 번만 고치면 되거든요.

### 3-2. 공용 규칙과 도구 전용 규칙 분리

모든 도구에 공통인 건 따로, 특정 도구에서만 필요한 건 따로 두는 게 좋습니다.

**공용 규칙** (한 곳에 원본으로 관리):

```markdown
# project-rules.md (공용 원본)

## 프로젝트 목적
- 이 프로젝트는 ~를 위한 서비스입니다

## 코딩 스타일
- TypeScript 사용
- 함수명은 camelCase

## 금지 사항
- console.log를 프로덕션에 남기지 않습니다

## 워크플로우
- PR 전 반드시 lint 통과
```

**Codex 전용 설정** (공용 규칙과 분리):

```toml
# .codex/config.toml (Codex 전용)

model = "gpt-5.4"
approval_policy = "never"
sandbox_mode = "workspace-write"
model_reasoning_effort = "medium"

[sandbox_workspace_write]
network_access = false

[profiles.cautious]
approval_policy = "on-request"
sandbox_mode = "workspace-write"
```

**공용 지침 파일** (도구가 함께 보는 기준축):

```markdown
# AGENTS.md

이 프로젝트의 공용 규칙은 project-rules.md를 따릅니다.

- 작업 전 먼저 구조를 읽습니다
- 큰 변경은 계획부터 제안합니다
- 완료 후 수정 파일을 요약합니다
```

> **확인**: 공용 규칙 파일 하나를 만들고, 각 도구의 설정에서 그 파일을 참조하는 구조가 잡혔으면 성공입니다.

---

## STEP 4. 정리 시점 판단하기

### 4-1. 이런 신호가 보이면 바로 정리하세요

01 같은 규칙을 두 군데 이상에 복붙하고 있습니다

02 한 도구에선 되고 다른 도구에선 결과가 다릅니다

03 팀원이 "어느 파일이 진짜 원본이야?"라고 묻기 시작합니다

처음엔 꼭 안 해도 됩니다. 그런데 조짐이 보이면 빨리 정리하는 게 좋습니다. 나중에 엉킨 뒤에 정리하면 훨씬 힘들거든요.

> **확인**: 위 세 가지 신호 중 하나라도 해당되면 지금 바로 정리를 시작하세요.

## 여기서 막혔나요?

**Q. 공용 규칙 파일을 어디에 두는 게 좋을까요?**

상황: 프로젝트 루트에 둘지, 별도 폴더에 둘지 모르겠는 경우입니다.

원인:
- 정해진 표준은 없지만, 찾기 쉬운 곳이 좋습니다

해결:
1. 프로젝트 루트에 `project-rules.md` 또는 `RULES.md`로 두는 것이 가장 직관적입니다
2. 규칙이 많아지면 `rules/` 폴더를 만들어 분리하세요
3. README에 "규칙 원본 위치"를 명시해두면 팀원도 헷갈리지 않습니다

**Q. 도구마다 설정 파일 형식이 다른데 어떻게 통일하나요?**

상황: Codex는 TOML과 마크다운을 같이 쓰고, 다른 도구는 JSON이나 YAML을 쓰는 경우입니다.

원인:
- 도구마다 설정 포맷이 다른 건 자연스럽습니다

해결:
1. 공용 규칙은 마크다운으로 작성합니다 (사람이 읽기 가장 편합니다)
2. 각 도구의 설정 파일에서 "공용 규칙은 project-rules.md 또는 AGENTS.md를 참조"라고 명시합니다
3. 도구 전용 문법(슬래시 커맨드, 특수 설정 등)만 해당 도구 설정에 넣습니다

Codex 쪽은 여기서 한 가지를 더 기억하면 좋습니다.

- 기본 실행 성격은 `.codex/config.toml`
- 공용 행동 규칙은 `AGENTS.md`
- 범용 기능은 built-in skill
- 프로젝트 특화 작업만 repo skill

**Q. 혼자 쓰는데도 이렇게 나눠야 하나요?**

상황: 팀이 아니라 개인 프로젝트인 경우입니다.

원인:
- 혼자여도 도구가 두 개 이상이면 같은 문제가 생깁니다

해결:
1. 도구 하나만 쓸 때는 굳이 나눌 필요 없습니다
2. 두 번째 도구를 쓰기 시작하면 그때 분리하세요
3. 미래의 자신도 "팀원"이라고 생각하면 정리할 동기가 생깁니다

## 한 줄 정리

AI 도구를 여러 개 써도 괜찮습니다. **대신 규칙 원본은 한 벌만 두고, 나머지는 그걸 함께 보게 만드세요.**
