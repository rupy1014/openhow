> AI가 내 컴퓨터 안 파일만 만지면 반쪽입니다. 브라우저와 외부 서비스까지 닿아야 진짜 확장되죠.

자동화가 재밌어지는 순간이 바로 여기입니다.

웹페이지를 직접 열고, 클릭하고, 스크린샷 찍고, 슬랙이나 노션 같은 바깥 서비스와 연결하는 순간. 그때부터 AI는 채팅 도구가 아니라 **작업 허브**가 됩니다.

이 장에서는 브라우저 자동화의 기본 개념과 MCP를 통한 외부 서비스 연결을 직접 체험해봅니다.

---

## 완성 기준

- [ ] 브라우저 자동화로 웹페이지 스크린샷을 찍을 수 있다
- [ ] MCP의 개념과 역할을 설명할 수 있다
- [ ] MCP 서버를 하나 이상 연결하고 동작을 확인했다

---

## STEP 1. 브라우저 자동화 이해하기

### 1-1. 브라우저 자동화가 강한 이유

사람이 웹에서 하는 반복 작업은 생각보다 많습니다.

- 페이지 열기, 로그인
- 버튼 클릭, 폼 입력
- 표 읽기, 스크린샷 찍기

이걸 손으로 하면 귀찮고, AI에게 시킬 수 있으면 엄청 편해지죠. 예를 들어 이런 작업들입니다.

- 내 서비스 첫 화면 캡처하기
- 경쟁사 페이지 구조 확인하기
- 로컬에서 버튼이 제대로 눌리는지 체크하기

<!-- IMG: [브라우저 자동화로 웹페이지를 캡처하는 과정 스크린샷] -->

### 1-2. 두 가지 방식 비교

브라우저 자동화는 크게 두 방식으로 나뉩니다.

**대화형 조작** — 한 단계씩 보고 다음 행동을 정하는 방식입니다. 눈으로 확인하며 진행하기 좋지만, 느리고 비용이 올라갈 수 있습니다.

**스크립트형 조작** — 한 번에 절차를 적고 실행하는 방식입니다. 빠르고 안정적이지만, 처음에 절차를 정리해야 합니다.

대부분의 반복 자동화에는 스크립트형이 더 잘 맞습니다.

> **확인**: 두 방식의 차이를 이해했다면 다음 단계로 넘어가세요.

## STEP 2. MCP 개념 잡기

### 2-1. MCP란 무엇인가요?

아주 쉽게 말하면, **AI용 USB-C 규격**입니다.

예전에는 서비스마다 따로 연결 방식을 만들어야 했습니다. 그런데 MCP라는 표준이 있으면 노션, 슬랙, 드라이브, 깃허브, 데이터베이스 같은 바깥 도구를 비슷한 방식으로 연결할 수 있죠.

왜 중요할까요? AI가 더 이상 "내 컴퓨터 안"에만 갇히지 않게 되기 때문입니다.

<!-- IMG: [MCP 연결 구조 다이어그램 — AI ↔ MCP ↔ 노션/슬랙/깃허브/DB] -->

### 2-2. MCP가 열어주는 것들

예를 들면 이런 것들이 가능해집니다.

- 노션에 회의록 정리하기
- 슬랙 채널 요약하기
- 구글 드라이브 문서 읽기
- 데이터베이스 조회하기
- 깃허브 이슈 정리하기

파일 읽기/쓰기와 터미널 실행에서 끝나던 AI가, **외부 서비스까지 손을 뻗게 되는 것**이죠.

> **확인**: MCP가 "AI와 외부 서비스를 연결하는 표준 규격"이라는 점을 이해했으면 충분합니다.

## STEP 3. MCP 서버 연결해보기

### 3-1. Codex에서 MCP 설정 확인

01 Codex 터미널에서 현재 연결된 MCP 서버 목록을 확인합니다.

```bash
codex mcp list
```

02 아직 연결된 서버가 없다면, 파일시스템 MCP 서버를 예시로 추가해봅니다.

```bash
codex mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/dir
```

03 다시 목록을 확인해서 추가된 서버가 보이는지 확인합니다.

```bash
codex mcp list
```

<!-- IMG: [codex mcp list 실행 결과 — 연결된 MCP 서버 목록 터미널 스크린샷] -->

> **확인**: `codex mcp list`에서 추가한 서버가 표시되면 성공입니다.

### 3-2. 입문자가 알면 충분한 범위

처음부터 다 연결할 필요는 없습니다. 이 정도만 기억하세요.

1. 브라우저 자동화는 반복 웹 작업에 강합니다
2. MCP는 외부 서비스 연결 규격입니다
3. 지금 당장 필요할 때 하나씩 붙이면 됩니다

슬랙이 필요하면 슬랙부터, 노션이 필요하면 노션부터, 브라우저 캡처가 필요하면 거기부터 붙이면 됩니다.

## STEP 4. 버려야 할 환상 한 가지

> "다 연결하면 완전 무인 자동화가 되겠지."

아닙니다.

연결은 시작일 뿐입니다. 어떤 조건에서, 무슨 데이터를 읽고, 어떻게 판단하고, 어디까지 자동으로 보내고, 어디서 사람이 확인해야 하는지 — 이 **설계**가 더 중요합니다.

자동화는 연결보다 **판단 설계**가 더 중요합니다.

## 여기서 막혔나요?

**Q. MCP 서버를 추가했는데 Codex에서 인식을 못합니다**

상황: `codex mcp add` 명령은 성공했는데, 실제 대화에서 해당 도구가 나타나지 않는 경우입니다.

원인:
- Codex 세션을 새로 시작하지 않았을 수 있습니다
- MCP 서버에 필요한 패키지가 설치되지 않았을 수 있습니다

해결:
1. Codex를 종료했다가 다시 시작합니다
2. `codex mcp list`로 서버 상태를 확인합니다
3. 필요한 npm 패키지가 글로벌로 설치되어 있는지 확인합니다

**Q. 브라우저 자동화가 로컬에서 동작하지 않습니다**

상황: 스크립트를 실행했는데 브라우저가 열리지 않거나 타임아웃이 발생합니다.

원인:
- Playwright나 Puppeteer가 설치되지 않았을 수 있습니다
- headless 모드 설정이 누락되었을 수 있습니다

해결:
1. `npx playwright install` 로 브라우저 바이너리를 설치합니다
2. 스크립트에서 `headless: true` 옵션이 설정되어 있는지 확인합니다

**Q. MCP로 연결할 수 있는 서비스 목록은 어디서 볼 수 있나요?**

상황: 노션, 슬랙 외에 어떤 서비스가 지원되는지 궁금한 경우입니다.

원인:
- MCP는 오픈 생태계라서 계속 새로운 서버가 추가되고 있습니다

해결:
1. [MCP 공식 서버 목록](https://github.com/modelcontextprotocol/servers)에서 확인합니다
2. `awesome-mcp-servers` 같은 커뮤니티 목록도 참고하세요
3. 필요한 서비스가 없으면 직접 만들 수도 있습니다 (고급)

**Q. 대화형 조작과 스크립트형 조작 중 어떤 걸 먼저 해야 하나요?**

상황: 두 방식 중 어떤 것부터 시작해야 할지 모르겠는 경우입니다.

원인:
- 작업 성격에 따라 적합한 방식이 다릅니다

해결:
1. 처음 탐색할 때는 대화형으로 한 단계씩 확인합니다
2. 같은 작업을 반복하게 되면 스크립트형으로 전환합니다
3. 대부분의 실무 자동화는 스크립트형이 더 안정적입니다

---

## 한 줄 정리

브라우저 자동화는 웹 반복 작업을 줄이고, MCP는 외부 서비스를 AI 작업 흐름에 연결합니다. **둘을 알면 AI가 채팅창 밖으로 나옵니다.**
