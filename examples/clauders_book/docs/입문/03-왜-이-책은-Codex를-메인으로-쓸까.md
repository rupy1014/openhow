> 좋은 AI 도구를 고르는 일은
>
> 성능표보다 작업 방식에 더 가깝습니다.

이 장에서는 **왜 이 책이 Codex를 메인 도구로 잡았는지** 설명드리겠습니다.

중요한 건 "Codex가 무조건 최고다"가 아닙니다.

어떤 도구는 별다른 설정 없이도 잘 굴러갑니다.

반대로 어떤 도구는 설정을 쌓을수록 훨씬 강해집니다.

Codex는 후자에 가깝습니다.

---

## Claude는 왜 처음부터 잘하는 것처럼 느껴질까요?

많은 분이 Claude를 처음 써보면 이런 느낌을 받습니다.

"설명 안 많이 해도 알아서 척척 하네?"

실제로 그런 장점이 있습니다.

대화만으로도 꽤 자연스럽게 코딩을 이어가는 편이거든요.

그래서 첫인상은 아주 좋습니다.

초반 진입 장벽도 낮고요.

그런데 프로젝트가 길어지고,
반복 작업이 생기고,
리서치와 검수와 수정이 나뉘기 시작하면
다른 감각이 필요해집니다.

이때부터는 그냥 똑똑한 채팅 상대보다
**설정 가능한 작업 환경**이 중요해집니다.

---

## Codex는 어디서 강해질까요?

OpenAI 공식 문서가 권하는 흐름은 꽤 분명합니다.

Codex를 일회성 도우미가 아니라,
**설정하고 길들이는 팀원처럼 쓰라**는 겁니다.

대표적인 축이 다섯 가지입니다.

### 1. `AGENTS.md`

이 프로젝트에서 꼭 지켜야 할 규칙을 적어두는 파일입니다.

무슨 폴더를 우선 봐야 하는지,
어떤 테스트를 돌려야 하는지,
건드리면 안 되는 건 뭔지
이런 걸 여기에 적습니다.

### 2. `config.toml`

모델,
추론 강도,
승인 방식,
샌드박스 범위를 잡는 설정 파일입니다.

즉, Codex의 기본 성격을 여기서 정합니다.

### 3. MCP

문서,
브라우저,
외부 도구,
사내 시스템을 연결하는 표준 통로입니다.

단순 채팅이 아니라,
실제 작업 파이프라인으로 넘어갈 때 중요해집니다.

### 4. Skills

반복해서 쓰는 작업 방식을 재사용 가능한 패턴으로 묶는 겁니다.

예를 들면 이런 식입니다.

- 챕터 초안 검수하기
- 릴리스 전 체크리스트 돌리기
- 특정 폴더 전용 UI 다듬기

여기서 중요한 포인트가 하나 더 있습니다.

Codex에는 OpenAI가 기본으로 넣어둔 built-in skill도 있습니다.

예를 들면 공식 문서에 `skill-creator`, plan skills, `skill-installer`가 나옵니다.

즉, 모든 걸 repo 안에 다시 만들 필요는 없습니다.

### 5. Automations

사람이 매번 붙어 있지 않아도 되는 작업은
`codex exec` 같은 방식으로 자동화할 수 있습니다.

즉, 인터랙티브 대화에서 끝나지 않습니다.

---

## 그래서 이 책은 왜 Codex를 메인으로 잡을까요?

이 책 뒤쪽에서 하려는 일이 단순한 코딩 보조가 아니기 때문입니다.

우리는 여기까지 가려고 합니다.

- 프로젝트 폴더 전체 이해하기
- 규칙 파일로 작업 방식 고정하기
- 역할을 나눠 검수와 리서치를 분리하기
- 반복 작업을 skills로 묶기
- 나중에는 자동화까지 연결하기

이 흐름에는 Codex가 아주 잘 맞습니다.

특히 "한 번 잘 시키는 것"보다
"같은 방식으로 계속 잘 굴리는 것"에 강합니다.

즉, 이 책은 단순히 AI에게 한두 번 코드를 받아내는 법보다,
**AI 작업 환경을 설계하는 법**까지 다루려는 겁니다.

그래서 Codex를 메인으로 잡았습니다.

---

## Codex는 기본값만으로 쓰면 안 될까요?

써도 됩니다.

처음엔 오히려 그게 맞습니다.

그런데 Codex의 진짜 장점은
기본값보다 **누적 설정**에서 나옵니다.

예를 들어 이런 식입니다.

- 이 프로젝트 규칙은 `AGENTS.md`에 적어둔다
- 이 작업은 `workspace-write` 안에서 자동으로 움직인다
- 리서치용 skill과 리뷰용 skill을 따로 둔다
- 일반 planning은 built-in skill에 맡기고, repo에는 도메인 특화 skill만 둔다
- 큰 작업은 계획부터 받고, 독립 작업만 병렬로 나눈다

이런 구조를 잡아두면
매번 처음부터 설명하지 않아도 됩니다.

한마디로 말하면,
Claude는 "처음 대화할 때 편한 도구"에 가깝고,
Codex는 "환경을 세팅할수록 팀처럼 변하는 도구"에 가깝습니다.

---

## 공식 권장 설정은 어떻게 생각하면 쉬울까요?

OpenAI 공식 문서 기준으로 아주 간단히 줄이면 이렇습니다.

### 1. 모델은 `gpt-5.4`부터 시작합니다

대부분의 작업은 여기서 시작하면 됩니다.

가벼운 작업이나 하위 작업에는 `gpt-5.4-mini`를 붙이는 방식이 권장됩니다.

### 2. 승인과 샌드박스는 같이 봐야 합니다

공식 문서는 `on-request` 같은 보수적 흐름도 설명하고,
`full-auto` 같은 자동 실행 흐름도 함께 설명합니다.

이 책 예제는 그중에서 실전 체감이 빠른 쪽을 택했습니다.

- 승인 없음
- `workspace-write`
- 네트워크 꺼둠

즉, **자동으로 진행하되 울타리는 둔다**는 방식입니다.

### 3. 규칙은 채팅창이 아니라 파일에 둡니다

그게 `AGENTS.md`입니다.

매번 길게 설명하는 것보다,
자주 쓰는 규칙은 파일로 고정하는 게 훨씬 낫습니다.

### 4. 반복 작업은 prompt가 아니라 skill로 올립니다

같은 일을 세 번째부터는 skill 후보라고 보면 됩니다.

다만 범용 planning까지 repo에 다 넣기보다,
기본 제공 skill은 기본 제공 skill대로 쓰고
repo에는 그 프로젝트에만 필요한 skill만 두는 편이 더 깔끔합니다.

### 5. 병렬 분업은 정말 독립적인 일일 때만 씁니다

앞 단계 결과를 받아야 하는 체인은
억지로 병렬화하면 품질이 흔들립니다.

---

## 이 책의 예제 프로젝트는 뭘 보여주나요?

이번에 같이 넣은 `codex-starter` 예제는
"Codex를 어떻게 메인 도구로 세팅하는가"를 보여주기 위한 작은 워크스페이스입니다.

들어 있는 건 거창하지 않습니다.

- 작은 정적 웹앱
- 프로젝트 규칙용 `AGENTS.md`
- 기본 동작을 잡는 `.codex/config.toml`
- repo 공용 `.agents/skills/`
- `site/` 폴더 전용 `.agents/skills/`
- 공식 built-in skill 메모 파일

즉, 결과물 자체보다
**작업 환경 구조**를 보여주는 예제라고 보시면 됩니다.

책 앞부분에서 이 구조를 먼저 보고 가면,
뒤쪽의 오케스트레이션과 멀티 에이전트 챕터가 훨씬 덜 추상적으로 느껴집니다.

---

## 그럼 다른 도구는 안 써도 될까요?

그럴 필요는 없습니다.

오히려 나중에는 같이 쓰게 될 가능성이 높습니다.

예를 들면 이렇습니다.

- 빠른 대화 초안은 Claude
- 프로젝트 중심 작업은 Codex
- 에디터 안 즉시 수정은 Cursor 계열

이런 식의 역할 분담은 충분히 현실적입니다.

다만 책은 기준축이 하나 있어야 합니다.

그래서 메인을 Codex로 잡는 겁니다.

---

## 실무에 어떻게 쓸까?

**기획자/PM**: 화면 하나 만드는 데서 끝내지 말고, `AGENTS.md`와 skill까지 같이 보세요. 그래야 같은 방식으로 두 번째, 세 번째 결과물을 만들 수 있습니다.

**마케터/운영**: 보고서 초안만 뽑는 데 쓸 수도 있지만, 진짜 차이는 반복 작업을 자동화할 때 납니다. Codex는 그 다리 역할을 해줍니다.

**개발자/창업팀**: 개인 생산성 도구로만 쓰지 말고, 팀 규칙과 검수 흐름까지 Codex에 얹어보세요. 그때부터 작업 품질이 더 안정됩니다.

---

## 한 줄 정리

Codex를 메인으로 쓴다는 건
그냥 더 똑똑한 AI를 고른다는 뜻이 아닙니다.

AI가 계속 같은 방식으로 잘 일하게 만드는
작업 환경을 고른다는 뜻입니다.
