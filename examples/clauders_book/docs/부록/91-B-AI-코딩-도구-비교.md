> 어떤 도구가 최고냐보다
>
> 지금 여러분에게 뭐가 맞느냐가 더 중요합니다.

세부 기능은 자주 바뀝니다.

그래서 이 부록은 최신 스펙 경쟁표보다
**도구를 고르는 기준**에 집중합니다.

---

## 먼저 큰 분류부터 보겠습니다

AI 코딩 도구는 보통 이렇게 나뉩니다.

### 터미널 중심

검은 화면에서 AI와 일하는 방식입니다.

프로젝트 전체를 보고,
파일과 명령 실행까지 묶기 좋습니다.

### 에디터 중심

코드 편집기 안에서 옆에 붙어 일하는 방식입니다.

파일을 눈으로 보면서 바로 수정하기 좋죠.

### 브라우저 중심

설치 부담이 적고,
가볍게 시작하기 좋습니다.

대신 프로젝트 전체 작업과 자동화는 상대적으로 약할 수 있습니다.

## 도구별 감각 차이

| 도구 | 기본 감각 | 잘 맞는 사람 | 주의할 점 |
| --- | --- | --- | --- |
| Codex CLI | 터미널 중심, 실행/분업/오케스트레이션에 강함 | 여러 작업 흐름을 정리하고 자동화까지 묶고 싶은 사람 | 구조 없이 쓰면 거칠게 느껴질 수 있음 |
| Claude Code | 터미널 중심, 파일/명령 실행까지 연결 | 앱 만들기부터 자동화까지 길게 가려는 사람 | 초반엔 검은 화면이 낯설 수 있음 |
| Cursor | 에디터 중심, 코드 보면서 대화하기 편함 | 에디터 안에서 바로 수정하는 흐름이 편한 사람 | 채팅창 위주로만 쓰면 큰 흐름을 놓칠 수 있음 |
| Copilot 계열 | 에디터 보조에 강함 | 기존 개발 흐름에 AI를 얹고 싶은 사람 | 처음 배우는 입문자에겐 전체 흐름이 덜 보일 수 있음 |
| Windsurf 계열 | 에디터 중심, 자동 제안과 흐름 연결 | 빠른 편집과 보조를 선호하는 사람 | 프로젝트 구조를 스스로 챙겨야 함 |

이 표는 우열표가 아닙니다.

**작업 스타일 표**에 가깝습니다.

## 어떤 기준으로 고르면 될까요?

### 1. 결과물만 빨리 보고 싶은가?

그렇다면 브라우저형이나 에디터형이 편할 수 있습니다.

### 2. 앱 만들기에서 자동화까지 길게 갈 건가?

그렇다면 터미널형이 유리합니다.

프로젝트 전체와 실행 흐름이 더 잘 보이기 때문입니다.

### 3. 나는 어디서 덜 겁먹는가?

이 기준도 중요합니다.

검은 화면이 너무 막막하면 에디터형이 시작은 쉬울 수 있습니다.

반대로 구조를 빨리 익히고 싶으면 터미널형이 더 잘 맞습니다.

## 이 책이 Codex를 메인으로 잡은 이유

이 책은 단순한 코딩 보조가 아니라

- 앱 만들기
- 배포
- 자동화
- 브라우저 조작
- MCP 연결
- 맥미니 자동화

여기까지 한 흐름으로 가기 때문입니다.

이 범위에는 터미널형이 더 잘 맞습니다.

그리고 무엇보다
AI가 지금 뭘 하고 있는지가 더 잘 보입니다.

그게 입문자에겐 꽤 중요하거든요.

특히 Codex는

- `AGENTS.md`
- `config.toml`
- skills
- 자동화

같은 구조를 쌓아가며
작업 환경을 설계하기 좋습니다.

그래서 이 책은 Codex를 기준축으로 잡고 설명합니다.

## 좋은 선택 순서는 뭘까요?

추천 순서는 이렇습니다.

1. 하나를 먼저 고릅니다
2. 그걸로 끝까지 가봅니다
3. 그다음 다른 도구를 비교합니다

처음부터 도구 비교만 하면
대체로 아무것도 안 만듭니다.

경험이 쌓인 뒤에 비교해야
차이가 진짜로 보입니다.
