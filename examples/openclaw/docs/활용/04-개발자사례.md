## 2026-02 유튜브 실전 공통 패턴

2026년 2월 18일~24일 공개된 OpenClaw 실사용 영상 8개를 보면, 잘 되는 팀은 거의 같은 운영 패턴을 써.

1. **컨텍스트 분리**
- 텔레그램은 그룹 + 토픽, 디스코드는 워크플로우별 채널로 분리
- 한 스레드에 모든 작업을 넣지 않음 (맥락 오염 방지)

2. **모델 라우팅**
- 리서치/설계만 고성능 모델
- 하트비트/요약/분류는 저비용 모델
- 채널 단위로 모델을 다르게 둬서 비용과 속도를 동시에 잡음

3. **Markdown-first 메모리**
- 중요한 내용은 채팅에만 두지 않고 `.md` 파일로 축적
- 야간 크론으로 대화 내용을 정리/통합해 다음 날 컨텍스트로 재활용

4. **크론 + 하트비트로 선제 대응**
- 단순 알림이 아니라, 문제 탐지 후 초안/리포트까지 자동화
- 이메일은 `draft only`, 파괴적 작업은 반드시 승인 후 실행

5. **장시간 작업은 감시 루프 필수**
- "코딩 맡기고 끝"이 아니라 heartbeat에서 진행 상태를 재확인
- 장시간 세션은 `/tmp` 대신 유지되는 작업 디렉터리 사용

6. **노이즈 관리**
- 모든 알림을 실시간으로 받지 않고 우선순위별 배치 전송
- "긴급만 즉시", 나머지는 시간 단위 digest로 받음

## 1. 세컨드 브레인 (개인 지식 관리)

텔레그램에 아무거나 보내. 메모, 아이디어, 링크, 할 일.
나중에 "지난주에 봤던 그 기사 뭐였지?" 하면 찾아줘.

**메모리 3단계 구조:**

| 단계 | 파일 | 역할 |
|------|------|------|
| 항상 로드 | `MEMORY.md` | 핵심 사실 (~100줄) |
| 일별 기록 | `memory/YYYY-MM-DD.md` | 그날의 작업 노트 |
| 주제별 폴더 | `memory/people/`, `projects/` | 시맨틱 검색용 |

한 개발자는 Next.js로 대시보드를 만들어서 Cmd+K로 전체 메모리를 검색해.

**팁**: `MEMORY.md`는 100줄 이내로 유지해. 길어지면 토큰 낭비야.

## 2. 스마트홈 제어 (Home Assistant 연동)

한 개발자(Dan Malone)는 **"Claudette"**라는 너구리 캐릭터 에이전트를 만들었어.

**셋업:**
- Proxmox 서버, Home Assistant VM
- Ubuntu LXC 컨테이너에 OpenClaw (RAM 2GB, CPU 2코어)
- ha-mcp로 Home Assistant REST API 연결
- ElevenLabs "Lily" 음성 → Google Home 스피커

**텔레그램에서 이렇게 써:**

```
"아기 낮잠 모드 켜줘"
→ 모든 조명 1% 밝기, 2000K 색온도
→ 주말 낮 시간에만 자동 활성화

"거실 불 켜줘"
→ Home Assistant API 호출 → 즉시 실행
```

**효과**: Home Assistant UI에서 60분 걸리던 자동화 설정이 텔레그램으로 **5분**.

**비용**: Claude Max 구독 $200/월 (업무용이라 이미 쓰고 있었음)

## 3. 밤새 코딩 시키기 ("킬러 피처")

자기 전에 이렇게 시켜:

```
이 GitHub 이슈 분석하고, 코드 읽고, 수정해서 PR 올려놔.
```

아침에 일어나면 PR이 올라와 있어. 리뷰만 하면 돼.

**셋업:**
- opencode-controller 스킬
- 모델: Claude Opus (복잡한 코드) or Sonnet (단순 수정)
- 타임아웃 넉넉하게: `timeoutMs: 600000` (10분)

**주의**: 명확한 요구사항이 있을 때만 써. "뭔가 좋은 거 만들어줘"는 안 돼.

## 4. 제로 직원 회사 (멀티 에이전트)

한 창업자가 에이전트 6개로 **1주일간 회사를 혼자 운영**했어.

| 에이전트 | 역할 |
|---------|------|
| Morgan | 총괄 코디네이터 |
| Atlas | IT 인프라 |
| Ada | 운영 |
| Linus | 개발 |
| James | 리서치 |
| (본인) | 최종 의사결정만 |

**모델 전략 (비용 최소화):**
- 메인: Kimi K2.5 (무료, 128K 컨텍스트)
- 폴백: GLM 4.7 (무료)
- 로컬: Ollama qwen2.5:7b (무료)
- 하트비트: Ollama (무료)

**월 비용: ~$50** (대부분 로컬 추론)

**배운 것들:**
1. **모델 폴백은 필수**. 클라우드 API는 자주 타임아웃 남
2. **에이전트당 Gateway 하나**. 공유하면 연쇄 장애
3. **메모리 영속성 필수**. 재시작하면 맥락 사라져
4. **하트비트가 장애의 90%를 잡아줘**. 로컬 모델로 비용 $0

## 5. GitHub 이슈 자동 처리 (Slack 연동)

Slack에서 GitHub 이슈 알림이 오면:
1. OpenClaw가 이슈 분석
2. 로컬 코드 확인
3. 수정 구현
4. 이슈에 업데이트 코멘트

도구 전환(IDE ↔ 브라우저 ↔ 터미널) 없이 Slack에서 끝나.

**PR 모니터링 크론:**

```bash
openclaw cron add --name "PR체크" --cron "0 9 * * 1-5" \
  --session isolated --announce --channel slack \
  --message "열린 PR 중 머지 가능한 거 확인해. 충돌 없고, CI 통과, 승인 완료된 것만"
```

## 바로 복붙할 운영 템플릿

### 1) 토픽/채널 분리 규칙

```text
앞으로 작업을 주제별로 분리해서 운영해.
- 채널/토픽 1개 = 워크플로우 1개
- 서로 다른 주제는 같은 세션에 섞지 마
- 새 주제가 나오면 새 채널(또는 토픽) 제안해
```

### 2) 모델 라우팅 규칙

```text
모델 사용을 작업 난이도 기준으로 분리해.
- Deep research/설계: 고성능 모델
- 요약/분류/하트비트: 저비용 모델
- 매주 비용 리포트와 라우팅 최적화안을 제출해
```

### 3) 야간 메모리 통합

```text
매일 새벽 2시에 오늘 대화를 정리해.
1) 중요한 결정/할 일/배운 점 추출
2) 관련 markdown 파일 업데이트
3) 중복/오래된 내용 정리
4) 내일 바로 쓸 수 있는 브리핑 생성
```

### 4) 안전 실행 정책

```text
다음 정책을 기본값으로 적용해.
- 이메일은 초안만 작성, 발송은 승인 필요
- 외부 결제/삭제/배포는 승인 없으면 실행 금지
- 실행 가능한 명령은 allowlist 중심으로 관리
```

### 5) 장시간 코딩 작업 감시

```text
장시간 코딩 작업은 시작 시 작업 로그를 남기고 heartbeat로 추적해.
- 세션이 종료/실패하면 원인 기록 후 1회 자동 재시도
- 성공 시 결과 링크 + 변경 요약 + 테스트 결과 보고
```

## 잘 되는 것 vs 안 되는 것

| 잘 되는 것 | 안 되는 것 |
|-----------|-----------|
| 정형화된 반복 작업 | 모호한 탐색적 작업 |
| 이메일 정리, 파일 분류 | 캘린더 관리 (불안정) |
| 모닝 브리핑, 예약 보고서 | 복잡한 UI 브라우저 자동화 (깨지기 쉬워) |
| 밤새 코딩 | 확인 없이 실행 (위험) |
| 세컨드 브레인 | 무한 루프 (가끔 빠짐) |
| 스마트홈 제어 | 긴 침묵 (생각 중 몇 분) |
