*검증 기준일: 2026-03-03 (소스 코드·공식 문서 기준)*

**AI 비서를 여러 명 고용해서 각자 담당 업무를 맡겨.**

---

## 왜 한 명이면 부족해?

비서가 하나면 뭐든 다 시켜야 해.
이메일도, 코딩도, 서버 감시도 한 놈이 해.

그러면 이런 문제가 생겨:

- 코딩하다가 이메일 얘기가 섞이면 맥락이 꼬여
- 복잡한 분석 중에 단순 알림이 끼어들어
- 비싼 모델로 단순 작업까지 돌려서 비용 낭비

**해결법: 담당자를 나눠.**

---

## 담당자 구성 예시

| 비서 ID | 담당 | AI 모델 | 채널 |
|---------|------|---------|------|
| `secretary` | 이메일·일정 | Claude Sonnet | 텔레그램 |
| `developer` | 코딩·PR 관리 | Claude Opus | 슬랙 |
| `monitor` | 서버 감시 | GPT-4o-mini | 디스코드 |

각 담당자는 완전히 독립이야.
**자기만의 기억, 성격, 도구, AI 모델**을 갖고 있어.

하나가 꺼져도 나머지는 멀쩡해.

---

## 어떻게 만들어?

### Step 1. 담당자 목록을 등록해

설정 파일(`~/.openclaw/openclaw.json`)에 `agents.list`를 적어.

```json5
{
  agents: {
    list: [
      {
        id: "secretary",
        name: "비서",
        model: "anthropic/claude-sonnet-4-5",
        identity: { name: "비서", emoji: "📋" }
      },
      {
        id: "developer",
        name: "개발자",
        model: "anthropic/claude-opus-4-6",
        identity: { name: "개발자", emoji: "💻" }
      },
      {
        id: "monitor",
        name: "감시봇",
        model: "openai/gpt-4o-mini",
        identity: { name: "감시봇", emoji: "🔍" }
      }
    ]
  }
}
```

각 담당자는 자동으로 자기만의 작업 폴더를 갖게 돼:

```
~/.openclaw/agents/secretary/workspace-secretary/
~/.openclaw/agents/developer/workspace-developer/
~/.openclaw/agents/monitor/workspace-monitor/
```

성격(SOUL.md), 규칙(AGENTS.md), 기억(MEMORY.md)이 전부 따로야.

### Step 2. 어떤 채널이 어떤 담당자를 부를지 정해

```json5
{
  bindings: [
    // 텔레그램 → 비서가 받아
    { agentId: "secretary", match: { channel: "telegram" } },

    // 슬랙 → 개발자가 받아
    { agentId: "developer", match: { channel: "slack" } },

    // 디스코드 → 감시봇이 받아
    { agentId: "monitor", match: { channel: "discord" } }
  ]
}
```

이걸 **바인딩**이라고 해.
"이 채널 메시지는 이 담당자한테 보내" 라는 배정 규칙이야.

### Step 3. 게이트웨이(중간 다리)를 재시작해

```bash
openclaw gateway restart
```

끝. 이제 채널마다 다른 담당자가 대응해.

## 더 세밀하게 배정할 수 있어?

바인딩은 채널 단위보다 더 세밀하게 설정할 수 있어.

```json5
{
  bindings: [
    // 디스코드 서버 중 "research" 서버만 → 리서치 담당
    {
      agentId: "researcher",
      match: { channel: "discord", guildId: "research-server" }
    },

    // 디스코드 "researchers" 역할을 가진 사람만 → 리서치 담당
    {
      agentId: "researcher",
      match: { channel: "discord", roles: ["researchers"] }
    },

    // 텔레그램인데 특정 사람한테만 → 개발자가 받아
    {
      agentId: "developer",
      match: {
        channel: "telegram",
        peer: { kind: "direct", id: "특정-chat-id" }
      }
    }
  ]
}
```

채널, 서버, 역할, 특정 사람 — 원하는 단위로 배정할 수 있어.

## 담당자별 AI 모델 다르게 쓰기

이게 비용 절약의 핵심이야.

```json5
{
  agents: {
    list: [
      {
        id: "developer",
        model: {
          primary: "anthropic/claude-opus-4-6",     // 메인: 강력한 모델
          fallbacks: ["anthropic/claude-sonnet-4-5"] // 백업: 저렴한 모델
        }
      },
      {
        id: "monitor",
        model: "openai/gpt-4o-mini"   // 단순 감시는 싸구려로
      },
      {
        id: "secretary",
        model: "anthropic/claude-sonnet-4-5"  // 중간 성능
      }
    ]
  }
}
```

복잡한 코딩 → Opus (비싸지만 똑똑해)
서버 감시 → GPT-4o-mini (싸고 빨라)
이메일 정리 → Sonnet (가성비)

이렇게 나누면 **비용이 60~80% 줄어**.

## 담당자끼리 협업시키기

여기가 진짜 강력한 부분이야.

담당자들이 서로 메시지를 주고받을 수 있어.

### 켜는 법

```json5
{
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["secretary", "developer", "monitor"]
    }
  },
  session: {
    agentToAgent: {
      maxPingPongTurns: 3   // 최대 3번 주고받기 (무한 루프 방지)
    }
  }
}
```

### 어떻게 돌아가?

예를 들어 비서한테 "서버 상태 알려줘"라고 하면:

```
너 → 비서: "서버 상태 알려줘"
      ↓
비서 → 감시봇: "지금 서버 상태 보내줘" (자동으로 물어봄)
      ↓
감시봇 → 비서: "CPU 45%, 디스크 72%, 정상"
      ↓
비서 → 너: "서버 정상이야. CPU 45%, 디스크 72%."
```

비서가 알아서 적절한 담당자한테 물어보고 답을 정리해서 줘.

### 주의사항

- `maxPingPongTurns`를 꼭 설정해. 안 하면 무한 대화할 수 있어
- `allow` 목록에 없는 담당자끼리는 대화 불가 (보안)
- 기본은 꺼져 있어. 명시적으로 켜야 해

## 하위 담당자(서브에이전트) 만들기

담당자가 혼자 감당 못 하는 일이면 **임시 하위 담당자를 만들어서 시킬 수 있어.**

```json5
{
  agents: {
    list: [
      {
        id: "developer",
        subagents: {
          allowAgents: ["developer"],   // 자기 밑에 서브 생성 허용
          model: "anthropic/claude-sonnet-4-5"  // 서브는 저렴한 모델로
        }
      }
    ]
  }
}
```

개발자 담당에게 "이 3개 파일 각각 리뷰해줘"라고 하면:
- 서브에이전트 3개를 만들어서 동시에 리뷰
- 결과를 모아서 보고

**제한이 있어:**
- 깊이 제한: 기본 1단계 (서브의 서브는 안 돼)
- 동시 개수: 기본 5개까지
- 끝나면 자동으로 정리돼

## 담당자마다 하트비트(자동 점검) 다르게

```json5
{
  agents: {
    list: [
      {
        id: "monitor",
        heartbeat: {
          every: "10m",                    // 10분마다
          activeHours: { start: "00:00", end: "23:59" },  // 24시간
          model: "openai/gpt-4o-mini",     // 싼 모델로
          target: "discord"                // 결과는 디스코드로
        }
      },
      {
        id: "secretary",
        heartbeat: {
          every: "30m",                    // 30분마다
          activeHours: { start: "08:00", end: "18:00" },  // 업무시간만
          lightContext: true               // HEARTBEAT.md만 읽어 (가벼움)
        }
      }
    ]
  }
}
```

감시봇은 24시간 10분마다, 비서는 업무시간에만 30분마다.

`lightContext: true`로 하면 HEARTBEAT.md만 읽어서 토큰을 아껴.

## 담당자별 샌드박스(격리 실행)

각 담당자마다 보안 수준을 다르게 설정할 수 있어.

```json5
{
  agents: {
    list: [
      {
        id: "developer",
        sandbox: {
          mode: "run",        // Docker 안에서 실행
          scope: "agent",     // 이 담당자 전용 컨테이너
          docker: {
            cpus: "2",
            memory: "4g"
          }
        }
      },
      {
        id: "secretary",
        sandbox: { mode: "off" }   // 격리 없이 호스트에서 실행
      }
    ]
  }
}
```

코딩하는 담당자는 Docker 안에서 안전하게.
이메일 담당은 호스트에서 바로.

## 실전 예시: 1인 회사 운영

한 창업자가 담당자 6명으로 1주일간 회사를 혼자 운영했어.

| 담당자 | 역할 | 모델 |
|--------|------|------|
| Morgan | 총괄. 다른 담당자한테 일 배분 | Kimi K2.5 (무료) |
| Atlas | 서버 관리 | Ollama (무료) |
| Ada | 운영 (이메일, 일정) | Kimi K2.5 |
| Linus | 코딩 | Claude Opus |
| James | 리서치 | GLM 4.7 (무료) |

**월 비용: ~$50.** 무료 모델을 적극 활용해서.

비결:
- 복잡한 코딩만 비싼 모델 (Opus)
- 단순 체크는 로컬 모델 (Ollama, 비용 $0)
- 담당자끼리 협업 켜서 자동으로 일 넘기기

## 한 줄 정리

`agents.list`에 담당자를 등록하고, `bindings`로 채널을 배정하고, `agentToAgent`로 협업을 켜면 AI 팀이 완성돼.
