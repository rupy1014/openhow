# 구독/정기결제 기획하려는데 뭘 알아야 하나요?

> 한 줄 요약: 이 글을 읽으면 구독 서비스의 과금 모델, 플랜 구조, 결제 실패 대응, 환불 정책까지 — 결제 연동 전에 기획자가 먼저 결정해야 할 것들을 정리할 수 있다

## TL;DR

- 구독 서비스 구축의 80%는 기획 의사결정이고, 결제 연동은 나머지 20%에 불과하다
- 빌링키(카드정보를 암호화한 결제용 키)를 발급받아 반복 청구하는 구조인데, 대부분의 PG사는 스케줄링을 제공하지 않아 직접 구현해야 한다
- 기획자가 먼저 정해야 하는 5가지: 과금 모델, 플랜 구조, 무료체험 설계, 업/다운그레이드 방식, 해지 흐름
- 결제 실패로 인한 비자발적 이탈이 전체 이탈의 25%를 차지한다 — 재시도 정책과 유예 기간 설계가 필수
- 환불 정책은 CS가 터지기 전에 미리 정해두지 않으면, 담당자마다 다른 기준으로 처리하게 된다

## 1. 구독 결제, 일반 결제와 뭐가 다른가

일반 결제는 고객이 결제 버튼을 누를 때마다 카드번호를 입력하고 본인인증을 거치는 구조다. 구독 결제는 다르다. 처음 한 번만 카드를 등록하면, 이후에는 고객 개입 없이 자동으로 결제가 반복된다.

이걸 가능하게 하는 게 **빌링키**다.

**빌링키**: 고객의 카드번호, 유효기간, CVC를 암호화해서 만든 결제용 키. 카드정보를 직접 저장하는 건 법적으로 불가능하기 때문에, 카드사가 이 암호화된 키를 대신 발급해준다. 이후 이 키로 결제를 요청하면 카드번호 입력 없이 결제가 된다.

### 인증 결제 vs 비인증 결제

|  | 인증 결제 | 비인증 결제 |
| --- | --- | --- |
| 본인인증 | 매 결제마다 필요 | 최초 1회만 (빌링키 발급 시) |
| 사용 예 | 일반 쇼핑몰 결제 | 구독, 정기결제, 자동 충전 |
| 고객 경험 | 결제할 때마다 인증 팝업 | 한 번 등록하면 자동 결제 |
| 기술적 차이 | PG 결제창 호출 | 빌링키로 서버에서 직접 결제 요청 |

구독 서비스를 만들려면 비인증 결제를 사용하게 된다. 여기서 첫 번째 의사결정이 필요하다.

### 빌링키 발급 방식: PG 결제창 vs API 직접 호출

| 방식 | 장점 | 단점 | 추천 상황 |
| --- | --- | --- | --- |
| PG 결제창 | 카드정보를 직접 다루지 않아 보안 부담 적음. 구현 간단 | UX 커스텀 제한. PG사 결제창 디자인에 종속 | 초기 서비스, 보안 인증 없는 팀 |
| API 직접 호출 | 결제 화면 자유 설계. 더 매끄러운 UX | 카드정보를 서버에서 다루므로 보안 요건 높음. 지원 PG 제한적 | UX가 중요한 SaaS, 충분한 개발 리소스 |

> **실무 팁**: PG 결제창 방식으로 시작하는 게 안전하다. API 직접 호출 방식은 카드정보를 서버에서 다뤄야 해서 PCI DSS 같은 보안 인증이 필요할 수 있다. 초기 서비스라면 PG 결제창으로 빌링키를 발급받고, 이후 UX 개선이 필요해지면 API 방식으로 전환해도 늦지 않다.

### PG사별 정기결제 지원 비교

구독 서비스에서 PG를 고를 때 흔히 기대하는 게 **스케줄링**이다. "매월 15일에 9,900원을 청구해줘"라고 PG에 맡기는 기능인데, 결론부터 말하면 **대부분의 PG사는 스케줄링을 제공하지 않는다.** 빌링키를 발급해주는 것까지가 PG의 역할이고, 언제·얼마를 청구할지는 서비스가 직접 구현해야 한다.

| PG사 | 빌링키 발급 | 자동 스케줄링 | 구독 라이프사이클 관리 | 비고 |
| --- | --- | --- | --- | --- |
| 토스페이먼츠 | O | **X** | **X** | 개발자 문서 잘 되어 있지만, 스케줄링·상태관리 모두 직접 구현 |
| 나이스페이먼츠 | O | **X** | **X** | API/결제창 모두 지원. 빌링키 발급까지만 제공 |
| NHN KCP | O | **X** | **X** | 안정적이고 레거시 풍부. 빌링키 발급까지만 제공 |
| KG이니시스 | O | **X** | **X** | 시장점유율 높음. 빌링키 발급까지만 제공 |
| 부트페이 커머스 | O | **O** | **O** | 구독 계약·자동 과금·재시도·일시정지·해지까지 솔루션 제공 |

대부분의 PG사는 빌링키 발급까지만 해주기 때문에, 나머지를 전부 직접 만들어야 한다:

- **스케줄러**: 결제 예정일에 맞춰 빌링키로 결제를 요청하는 배치/크론 작업
- **상태 관리**: 결제 성공/실패에 따른 구독 상태 전환 로직 (활성·일시정지·해지 등)
- **재시도 큐**: 실패한 결제를 언제, 몇 번 재시도할지 관리하는 로직
- **라이프사이클**: 플랜 변경, 일시정지/재개, 해지 요청 처리, 과금 주기별 이력 관리

부트페이 커머스는 이 전체를 구독관리 솔루션으로 제공한다. 구독 계약을 생성하면 과금 주기에 맞춰 자동으로 결제가 실행되고, 실패 시 재시도까지 처리된다. 일시정지·재개·중도 해지·플랜 변경도 API 하나로 가능하기 때문에, 개발팀이 스케줄러나 상태 관리 로직을 직접 만들 필요가 없다.

> **실무 팁**: "PG만 연동하면 구독 결제가 되겠지"라고 생각하고 시작하면, 스케줄링·재시도·상태 관리를 직접 만들어야 한다는 걸 개발 중반에 알게 된다. 이 부분의 개발 공수가 2\~4주 추가되는 경우가 많다. 구독관리 솔루션을 제공하는 결제 서비스를 선택하면 이 공수를 줄일 수 있고, 그렇지 않다면 결제 연동 전에 스케줄러 설계부터 해야 한다.

## 2. 기획자가 먼저 결정해야 하는 5가지

결제 연동 코드를 짜기 전에, 기획자가 먼저 답을 내려야 하는 질문들이 있다. 이걸 안 정하고 개발부터 시작하면, 나중에 DB 구조부터 다시 뜯어고치게 된다.

### (1) 과금 모델: 어떤 기준으로 돈을 받을 것인가

| 모델 | 설명 | 대표 서비스 | 적합한 경우 |
| --- | --- | --- | --- |
| 정액제 | 매월 고정 금액 | 넷플릭스, 밀리의 서재 | 콘텐츠 구독, 사용량 차이가 적은 서비스 |
| 종량제 | 사용한 만큼 과금 | AWS, Twilio | API 호출, 메시지 발송 등 사용량 편차가 큰 서비스 |
| 시트당(유저당) | 사용자 수 x 단가 | Slack, Notion 팀플랜 | B2B SaaS, 팀 협업 도구 |
| 하이브리드 | 기본료 + 초과 사용료 | HubSpot, Zoom | 기본 기능은 정액 + 추가 기능은 종량 |

**어떤 모델을 고를지 판단 기준:**

- 모든 고객이 비슷하게 쓰는가? --&gt; 정액제
- 고객마다 사용량 편차가 큰가? --> 종량제 또는 하이브리드
- 팀 단위로 쓰는 B2B인가? --&gt; 시트당 과금
- 매출 예측이 중요한가? --&gt; 정액제 (MRR 예측이 쉬움)

> **실무 팁**: 과금 모델에 따라 빌링키 사용 방식, DB 설계, 청구 로직이 완전히 달라진다. 정액제는 "매월 같은 금액을 청구"하면 되지만, 종량제는 "이번 달 사용량을 집계한 뒤 청구"해야 한다. 이 결정을 미루고 결제 연동부터 시작하면, 나중에 과금 모델을 바꿀 때 DB 마이그레이션과 기존 고객 전환 이슈가 동시에 터진다.

### (2) 플랜 구조: 몇 개를 어떻게 나눌 것인가

플랜 개수와 구조도 사전에 설계해야 한다. 실제 서비스들이 어떻게 하고 있는지 보자.

| 서비스 | 플랜 수 | 무료 플랜 | 월 가격 | 연 가격 (월 환산) | 연 할인율 | 구조 특징 |
| --- | --- | --- | --- | --- | --- | --- |
| 넷플릭스 | 3개 | X | 5,500~17,000원 | 월 결제만 | - | 화질 + 동시접속 수로 등급 구분 |
| 노션 | 4개 | O (개인 무료) | $10/시트 | $8/시트 | 20% | 무료 → 유료 전환 유도. 팀 플랜은 시트당 |
| Slack | 3개 | O (기능 제한) | $8.75/시트 | $7.25/시트 | 17% | 무료에서 메시지 검색 제한으로 유료 전환 유도 |
| Figma | 3개 | O (2파일 제한) | $15/시트 | $12/시트 | 20% | 무료에서 파일 수 제한. 팀 기능으로 유료 전환 |
| ChatGPT | 3개 | O (기능 제한) | $20 | $200/연 ($16.7) | 17% | 무료에서 모델/사용량 제한으로 유료 전환 |
| 밀리의 서재 | 2개 | X | 9,900원 | 99,000원 (8,250원) | 17% | 단순 구조. 플랜 구분 없이 전체 열람 |

**플랜 설계 판단 기준:**

- 플랜 수는 3~4개가 적정. 2개는 선택지가 부족하고, 5개 이상은 고민만 늘린다
- 무료 플랜은 "바이럴이 필요한 서비스"에 효과적. 대신 무료 → 유료 전환 트리거를 명확히 설계해야 한다 (Slack의 메시지 검색 제한, Figma의 파일 수 제한처럼 "딱 하나 부족한 것"을 만들어야 한다)

### 월 결제 vs 연 결제: 가격만이 아니라 정책이 다르다

연 결제 시 15~25% 할인이 일반적이다. 이건 단순한 할인이 아니라 결제 실패 리스크를 줄이고 현금흐름을 안정시키는 효과가 있다. 단, **연/월 결제는 가격뿐 아니라 취소·환불 정책도 달라야 한다.** 이걸 안 정해두면 CS에서 혼란이 생긴다.

| 항목 | 월 결제 | 연 결제 |
| --- | --- | --- |
| 가격 | 정가 | 15~25% 할인 (2개월 무료 효과) |
| 해지 시점 | 즉시 or 현재 주기 끝 | 현재 연 주기 끝 (일반적) |
| 중도 해지 환불 | 보통 없음 (남은 기간 이용 가능) | **반드시 정책 명시** — 아래 참고 |
| 결제 실패 리스크 | 매월 발생 가능 | 연 1회만 (비자발적 이탈 ↓) |
| 캐시플로우 | 매월 소액 유입 | 연초 대금 확보 (예측 가능) |

**연 결제 중도 해지가 가장 분쟁이 많다.** 예를 들어 연 100,000원(월 환산 8,333원)으로 결제한 고객이 3개월 후 해지하면:

- **방법 A**: 남은 9개월분 환불 → 100,000 - (8,333 × 3) = 75,001원 환불
- **방법 B**: 월 정상가(12,000원) 기준 차감 → 100,000 - (12,000 × 3) = 64,000원 환불
- **방법 C**: 환불 없음 — 연 주기 끝까지 이용 가능

대부분의 SaaS는 **방법 B**(월 정상가 기준 차감)를 쓴다. 연 할인은 "1년 약정"의 대가이므로, 중도 해지 시 할인 혜택을 소급 적용하는 게 합리적이라는 논리다. 어떤 방법을 선택하든 **약관과 결제 화면에 반드시 명시**해야 한다. 안 그러면 CS마다 다른 기준으로 처리하게 된다.

> **실무 팁**: 연 결제를 도입할 때 "할인율을 얼마로 할까"만 고민하고 중도 해지 환불 정책을 안 정하는 경우가 많다. 그러다 첫 환불 요청이 들어오면 "어떻게 계산하지?"부터 시작하게 된다. 연 결제를 만들기로 했다면 할인율과 환불 정책을 세트로 결정하라.

### (3) 무료체험: 언제 카드를 받고, 어떻게 전환할 것인가

무료체험 설계에서 가장 큰 결정은 "카드 등록 시점"이다.

| 방식 | 카드 등록 시점 | 전환율 | 대표 사례 |
| --- | --- | --- | --- |
| 카드 선등록 | 무료체험 시작 시 | 높음 (업계 평균 60\~80%) | Netflix, Spotify |
| 카드 후등록 | 무료체험 종료 시 | 낮음 (업계 평균 15\~30%) | Notion, Canva |

카드 선등록은 전환율이 높지만, 법적 주의가 필요하다.

> **실무 팁**: 다크패턴 규제에 주의해야 한다. 무료체험 종료 후 자동으로 유료 전환할 때, 사전 동의 없이 과금하면 전자상거래법 위반이다. 구체적으로: (1) 무료체험 등록 시 "체험 종료 후 자동 결제됩니다"를 명확히 고지, (2) 결제 7일 전 안내 메시지 발송 의무, (3) 안내에 해지 방법을 구체적으로 포함해야 한다. 실제로 쿠팡, 콘텐츠웨이브, NHN벅스, 스포티파이 등이 다크패턴 규정 위반으로 과태료를 부과받았다.

### (4) 업/다운그레이드: 프로레이션을 어떻게 처리할 것인가

프로레이션(proration): 플랜 변경 시 남은 기간에 대한 요금을 일할 계산하는 것. 예를 들어 월 1만원 플랜에서 월 3만원 플랜으로 월 중간에 바꾸면, 남은 15일분의 차액(1만원)을 어떻게 처리할 것인가.

| 방식 | 설명 | 고객 경험 | 구현 복잡도 |
| --- | --- | --- | --- |
| 프로레이션 + 즉시 청구 | 남은 기간 차액을 바로 결제 | 즉시 상위 플랜 이용. 청구 내역이 복잡해질 수 있음 | 높음 |
| 프로레이션 + 다음 결제일 반영 | 차액을 다음 청구에 합산 | 자연스러움. 가장 일반적 | 중간 |
| 다음 결제일부터 변경 | 현재 주기 끝나면 새 플랜 적용 | 단순하지만, 고객이 "바로 안 바뀌네?" 불만 가능 | 낮음 |
| 즉시 새 플랜 전액 결제 | 기존 잔여분 무시, 새 플랜으로 새 주기 시작 | 고객에게 불리. 불만 소지 | 낮음 |

**판단 기준:** 초기 서비스라면 "다음 결제일부터 변경" 방식이 구현도 간단하고 분쟁도 적다. 서비스가 커지면 "프로레이션 + 다음 결제일 반영" 방식으로 전환하는 게 일반적이다.

참고로 해외 서비스(Paddle)는 분 단위까지 정밀하게 프로레이션을 계산하지만, 한국 PG는 프로레이션 자동 계산을 지원하지 않는다. 어떤 방식을 선택하든 직접 구현해야 한다.

### (5) 해지 흐름: 쉽게 해지할 수 있어야 한다

"해지를 어렵게 만들면 이탈이 줄 거야"라고 생각할 수 있다. 결론부터 말하면, 법적으로도 안 되고 효과도 없다.

**다크패턴 규제 핵심:**

- 가입보다 해지가 더 복잡하면 위반이다
- 웹/앱에서 가입했으면, 동일 경로에서 해지가 가능해야 한다
- 해지 전 불필요한 단계(전화 연결, 팝업 반복)를 넣으면 과태료 대상이다
- 실제 부과 사례: 쿠팡 250만원, 콘텐츠웨이브 400만원, NHN벅스 300만원

**대신 이렇게 설계하는 게 낫다:**

- 해지 사유를 선택지로 수집 (강제 X, 선택형 O)
- 해지 확인 화면에서 "남은 혜택"을 보여주기 (예: "현재 플랜에서 이번 달 남은 크레딧 15개")
- 해지 완료 후 윈백 이메일 시퀀스 (7일 후, 30일 후)
- 즉시 해지 vs 기간 만료 해지 선택 제공

## 3. 결제 실패 — 가장 많이 놓치는 설계

구독 서비스를 기획할 때 과금 모델이나 플랜 구조는 고민하면서, 결제 실패 시나리오는 빠뜨리는 경우가 많다. 하지만 Stripe 데이터에 따르면, **전체 구독 이탈의 25%가 결제 실패로 인한 비자발적 이탈**이다. 고객이 서비스를 싫어서 나간 게 아니라, 카드 잔액 부족이나 만료 같은 이유로 결제가 안 돼서 자동으로 빠져나간 것이다.

반대로, 결제 실패 후 복구에 성공한 구독자는 평균 7개월 더 서비스를 유지한다. 재시도 정책을 잘 설계하는 것만으로 매출을 지킬 수 있다.

### 재시도 정책(Dunning) 설계

Dunning: 결제 실패 시 재시도하고, 고객에게 알림을 보내는 전체 프로세스.

**업계 표준 재시도 정책:**

| 단계 | 시점 | 액션 |
| --- | --- | --- |
| 1차 재시도 | 실패 후 3일 | 자동 재시도 + 고객에게 실패 안내 이메일/알림 |
| 2차 재시도 | 실패 후 6일 | 자동 재시도 + "결제수단을 확인해주세요" 알림 |
| 3차 재시도 | 실패 후 9일 | 마지막 재시도 + "서비스가 중단될 수 있습니다" 경고 |
| 서비스 중단 | 실패 후 10\~14일 | 서비스 접근 차단 + 데이터 보존 (30\~90일) |

> **실무 팁**: 결제가 실패했다고 서비스를 즉시 차단하면 안 된다. 일시적인 잔액 부족, 카드사 점검 등 일시적 사유가 대부분이다. 유예 기간(Grace Period) 없이 즉시 차단하면 고객 이탈과 CS 폭증이 동시에 온다. 최소 3\~7일은 서비스를 유지하면서 재시도하는 게 업계 표준이다.

### 카드 만료 대응

카드 유효기간은 보통 5년이다. 서비스를 2\~3년 운영하면 초기 가입자들의 카드가 대량으로 만료되기 시작한다.

**대응 플로우:**

1. **만료 2개월 전**: "등록하신 카드가 곧 만료됩니다" 이메일 발송
2. **만료 1개월 전**: 앱 내 배너 + 이메일 재발송
3. **만료 시점**: 원클릭으로 새 카드를 등록할 수 있는 페이지 링크 제공
4. **만료 후 결제 실패**: 위의 dunning 프로세스 진입

일부 카드사는 갱신카드가 발급되면 빌링키를 자동으로 연결해주지만, 모든 카드사가 그런 건 아니다. PG사별로 정확한 정책을 확인해야 한다.

### 고객 알림 타이밍

결제 실패 시 고객에게 보내는 알림은 최소 3회가 필요하다.

| 시점 | 메시지 톤 | 포함 내용 |
| --- | --- | --- |
| 실패 즉시 | 사실 전달 ("결제가 처리되지 않았어요") | 결제수단 변경 링크 |
| 재시도 전 | 부드러운 독촉 ("서비스를 계속 이용하시려면") | 결제수단 변경 링크 + 잔여 기간 안내 |
| 서비스 중단 전 | 긴급 톤 ("내일 서비스가 중단됩니다") | 결제수단 변경 링크 + 데이터 보존 기간 안내 |

## 4. 환불 정책: 공정위 기준과 실무 사이

구독 환불은 단건 결제 환불보다 복잡하다. "3만원짜리 월 구독을 15일째에 해지하면 얼마를 돌려주는가?"에 대한 답을 미리 정해두지 않으면, CS 담당자마다 다른 기준을 적용하게 되고, 고객 불만과 법적 리스크가 동시에 쌓인다.

### 법적 기준

- **기본 원칙**: 디지털콘텐츠는 구매 후 7일 이내 청약철회 가능
- **예외**: 콘텐츠 제공이 이미 시작된 경우 철회 불가
- **구독의 특수성**: 구독은 "가분적 콘텐츠"에 해당할 수 있다. 즉, 아직 제공되지 않은 나머지 기간분은 환불 가능하다는 게 공정위 해석이다

### 환불 시나리오별 처리

| 시나리오 | 처리 방식 | 예시 |
| --- | --- | --- |
| 가입 직후 해지 (7일 이내) | 전액 환불 | 월 9,900원 결제 → 3일 후 해지 → 9,900원 환불 |
| 월 중간 해지 | 이용일수 기준 일할 계산 환불 | 월 30,000원, 10일 이용 → 20,000원 환불 |
| 연 결제 중도 해지 | 이용월수 기준 + 월 단가 차이 정산 | 위 섹션 2-(2)의 "연 결제 중도 해지" 계산 방식 참고 |

### 환불 정책 설계 체크리스트

- [ ] 가입 후 며칠 이내는 전액 환불인가?

- [ ] 일할 계산 기준은 달력일 기준인가, 영업일 기준인가?

- [ ] 연 결제 중도 해지 시 월 단가 차이를 어떻게 정산하는가?

- [ ] 부분 환불 금액의 소수점은 올림/반올림/버림 중 어떤 방식인가?

- [ ] 환불 처리 기한은 며칠인가? (카드사 부분취소 처리 시간 포함)

- [ ] 환불 정책이 약관과 결제 화면에 명시되어 있는가?

## 5. 우리 서비스에 맞는 구독 구조는?

서비스 유형에 따라 구독 구조가 달라진다. 아래 의사결정 맵을 참고하자.

### 콘텐츠 구독 (넷플릭스형)

- **과금 모델**: 정액제
- **플랜 구분 기준**: 화질, 동시접속 수, 광고 유무
- **무료체험**: 7\~30일, 카드 선등록 방식이 일반적
- **해지 흐름**: 즉시 해지 + 결제 주기 끝까지 이용 가능
- **주의점**: 콘텐츠 양이 적으면 무료체험 기간에 다 소비하고 이탈한다. 체험 기간을 짧게 가져가거나, 체험 중 접근 가능한 콘텐츠를 제한하는 방법을 고려하자

### SaaS 도구 (노션형)

- **과금 모델**: 시트당 과금 또는 정액제 + 용량 제한
- **플랜 구분 기준**: 기능 차이 + 사용자 수
- **무료체험**: 14일이 일반적, 카드 후등록 방식 많음 (B2B는 결재 프로세스 때문)
- **연/월 차이**: 연 결제 시 15\~25% 할인 필수 (B2B는 연 결제 비율이 높음)
- **주의점**: 무료 플랜이 너무 넉넉하면 유료 전환이 안 된다. 무료에서 "딱 하나 부족한 것"을 만들어야 한다 (Slack의 메시지 검색 제한, Notion의 블록 수 제한 등)

### 정기배송 (마켓컬리형)

- **과금 모델**: 정액제 (박스당 고정가) 또는 멤버십 (배송비 무료 등 혜택)
- **플랜 구분 기준**: 배송 주기 (주 1회, 격주, 월 1회)
- **특수 기능**: 건너뛰기, 배송일 변경, 일시정지
- **해지 흐름**: "다음 배송 건너뛰기"를 해지 대안으로 제공하면 해지율이 낮아진다
- **주의점**: 실물 배송이라 환불 정책이 더 복잡하다. 배송 전/후, 상품 하자 등 시나리오별 환불 기준을 상세하게 정해야 한다

### 어떤 유형인지 모르겠다면

| 질문 | 답변 | 추천 구조 |
| --- | --- | --- |
| 고객이 주로 개인인가, 팀인가? | 개인 | 정액제 + 등급별 플랜 |
|  | 팀 | 시트당 과금 |
| 사용량 차이가 큰가? | 크다 | 종량제 또는 하이브리드 |
|  | 비슷하다 | 정액제 |
| 물리적 배송이 있는가? | 있다 | 정기배송형 (건너뛰기 필수) |
|  | 없다 | 콘텐츠/SaaS형 |

## 체크리스트: 구독 결제 기획 전 확인사항

### 기획 단계

- [ ] 과금 모델 결정 (정액 / 종량 / 시트당 / 하이브리드)

- [ ] 플랜 수와 구분 기준 결정

- [ ] 각 플랜의 월/연 가격 설계

- [ ] 무료 플랜 포함 여부 결정

- [ ] 무료체험 기간과 카드 등록 시점 결정

- [ ] 무료체험 → 유료 전환 시 사전 동의 플로우 설계

- [ ] 업/다운그레이드 시 프로레이션 방식 결정

- [ ] 해지 흐름 설계 (다크패턴 규제 준수 확인)

- [ ] 해지 사유 수집 방식 결정

### 결제/운영 단계

- [ ] PG사 선택 (스케줄링 지원 여부 확인)

- [ ] 빌링키 발급 방식 결정 (결제창 vs API)

- [ ] 결제 실패 재시도 정책 수립 (횟수, 간격)

- [ ] Grace period(유예 기간) 설정

- [ ] 카드 만료 사전 안내 플로우 구축

- [ ] 결제 실패 시 고객 알림 시나리오 작성

### 환불/법적 대응

- [ ] 환불 정책 수립 (일할 계산 기준, 연 결제 중도 해지)

- [ ] 환불 정책을 약관과 결제 화면에 명시

- [ ] 다크패턴 규제 6가지 금지 유형 확인

- [ ] 결제 7일 전 안내 발송 시스템 구축

- [ ] 해지 경로가 가입 경로와 동일한지 확인
