# 결제 기획 시 고려해야 할 사항

> 한 줄 요약: 이 글을 읽으면 사업자등록부터 첫 결제가 일어나기까지의 전체 과정을 이해하고, 내 상황에 맞는 PG사를 골라 심사를 통과할 수 있다.

## TL;DR

- 결제가 붙기까지 최소 2~3주. 사업자등록 + PG 심사가 병목이고, 개발 연동은 그 사이에 병렬로 진행할 수 있다
- PG사 선택에서 **수수료가 가장 중요하다**. 다만 수수료 협상은 국세청 매출신고 기준 중소3(연매출 10\~30억) 이상부터 가능하고, 그 이하는 카드사 우대수수료가 자동 적용된다
- 심사에서 떨어지는 3대 사유: 사이트 미구축, 위험업종, 서류 미비. 미리 알면 다 피할 수 있다
- 영세/중소 사업자는 카드사 우대수수료로 실제 비용이 훨씬 낮아진다. 신규 사업자도 소급 적용 가능
- 글 끝에 체크리스트가 있으니, 팀에 공유해서 하나씩 체크하면 된다

## 1. 결제가 붙기까지의 전체 그림

처음 결제를 도입하려면 "코드 먼저 짜야 하나?"라고 생각하기 쉬운데, 실제 병목은 코드가 아니라 **서류와 심사**에 있다.

```
[사업자등록] → [통신판매업 신고] → [PG사 가입 신청] → [심사 (7~10영업일)]
                                        ↕ (병렬 진행 가능)
                                  [개발: 결제 연동 + 테스트]
                                        ↓
                                  [심사 통과] → [실결제 전환] → [라이브]
```

| 단계 | 소요 시간 | 비고 |
|------|----------|------|
| 사업자등록 | 1~3일 | 세무서 방문 당일 또는 온라인 신청 시 3일 |
| 통신판매업 신고 | 1~5일 | 구청 또는 정부24에서 신고. 간이과세자는 직전 반기 50건 미만이면 면제. 현재는 카드사 심사에 필수가 아님 |
| PG사 가입 신청 | 당일 | 서류 준비만 되면 바로 가능 |
| PG 심사 | 7~10영업일 | 반려되면 보완 후 재심사로 14일+ |
| 결제 연동 (통합 SDK) | 1~3일 | 직접 연동이면 1~2주. 통합결제 SDK를 쓰면 훨씬 빠르다 |
| 테스트 → 실결제 전환 | 1일 | 키 교체, 웹훅 URL 확인, HTTPS 설정 |

**핵심: 심사와 개발을 동시에 진행해라.** PG사 대부분이 테스트 키를 먼저 발급해준다. 심사 결과를 기다리는 동안 테스트 환경에서 연동을 완료하면, 심사 통과 후 키만 교체하고 바로 라이브할 수 있다.

> **실무 팁**: 테스트 환경에서 실결제로 전환할 때 웹훅(결제 결과를 서버로 받는 콜백) URL, HTTPS 인증서, 도메인이 모두 운영 환경과 일치하는지 확인하세요. 테스트에서는 잘 되다가 라이브에서 결제 확인이 안 되는 원인의 대부분이 여기에 있습니다.

## 2. "나는 어떤 케이스인가" 분기표

결제 도입은 상황마다 준비물과 선택지가 다르다. 아래 분기표에서 자기 상황을 찾아보면 된다.

### 사업자 유형별

| 구분 | 개인사업자 | 법인 |
|------|----------|------|
| 필수 서류 | 사업자등록증, 정산계좌, 대표자 신분증, 인감증명서 | + 법인등기부등본, 법인인감증명서, 주주명부 |
| 보증보험 | 일부 PG사 면제 가능 | 대부분 필수 |
| 심사 난이도 | 상대적으로 간단 | 서류가 많지만 심사 자체는 비슷 |
| 정산 계좌 | 대표자 명의 계좌 | 법인 명의 계좌 |

### 상품 유형별

| 상품 유형 | 주의 사항 | 추가 준비 |
|----------|----------|----------|
| 실물 상품 (쇼핑몰) | 가장 무난한 케이스 | 배송 정책, 교환/반품 규정 |
| 디지털 콘텐츠 | 환불 정책을 명확히 해야 심사 통과 | 이용약관에 디지털 콘텐츠 환불 조건 명시 |
| SaaS/구독 서비스 | "조건부 가능" 업종 — 서비스 설명 상세 필요 | 과금 모델, 해지 정책 문서화 |
| 포인트/크레딧 충전 | **고위험 업종** — 사전 확인 필수 | 취급 가능한 PG사 확인 필요 |
| 크라우드펀딩 | **고위험 업종** — 일부 PG사 거절 | 금융위 등록 여부 + 취급 가능 PG사 확인 |

고위험 업종이나 조건부 업종은 PG사마다 심사 기준이 다르다. PG사에 하나하나 문의하는 대신, [부트페이에 문의](https://bootpay.co.kr)하면 내 업종에 맞는 PG사를 찾아준다. 여러 PG사의 심사 기준을 알고 있기 때문에, 직접 알아보는 것보다 빠르다.

### 판매 대상별

| 구분 | 국내만 | 해외 포함 |
|------|-------|----------|
| 결제수단 | 신용카드 + 간편결제면 충분 | 해외카드 + PayPal 등 추가 필요 |
| PG 선택 | 대부분의 PG사 가능 | 해외카드 수수료(3.5~3.8%) 별도 확인 |
| 추가 고려 | - | 환율, 해외 정산, 부가세 처리 |

**판단 기준 요약**: 실물 상품 + 국내 + 개인사업자라면 가장 단순한 케이스다. 여기서 벗어나는 조건이 하나씩 추가될 때마다 준비물과 확인 사항이 늘어난다고 보면 된다.

## 3. PG사 선택: 결국 수수료가 제일 중요하다

PG사(Payment Gateway)는 가맹점과 카드사 사이에서 결제를 중계해주는 회사다. "상황에 맞게 선택하라"는 말을 많이 하지만, 솔직히 **수수료가 가장 중요하다.** 결제 건마다 빠져나가는 돈이고, 매출이 커질수록 차이가 기하급수적으로 벌어진다.

### 수수료 구조부터 이해하자

PG 수수료에는 카드사 수수료가 포함되어 있다. 즉, 가맹점이 PG에 내는 수수료 안에 카드사 몫이 들어 있고, 나머지가 PG 마진이다.

```
가맹점이 내는 수수료 = 카드사 수수료 + PG 마진 + 부가세
```

여기서 **카드사 수수료는 국세청 매출신고 기준으로 등급이 정해지고, 등급별로 자동 적용**된다. 내가 협상할 수 있는 건 PG 마진 부분뿐인데, 이것도 일정 규모가 되어야 가능하다.

### 카드사 우대수수료 등급 — 국세청 매출신고 기준

카드사 수수료는 PG사가 아니라 **국세청에 신고한 연매출 기준**으로 등급이 결정된다. 매년 1월과 7월에 재산정되고, 모든 카드사에 동일하게 적용된다.

| 등급 | 연매출 기준 (국세청 매출신고) | 카드사 수수료 | 비고 |
|------|--------------------------|-------------|------|
| 영세 | 3억 원 이하 | **0.5\~1.1%** | 신용카드 0.5%, 체크카드 0.25% |
| 중소1 | 3\~5억 원 | **1.1\~1.4%** | |
| 중소2 | 5\~10억 원 | **1.25\~1.6%** | |
| 중소3 | 10\~30억 원 | **1.5\~1.9%** | **수수료 협상이 가능해지는 구간** |
| 일반 | 30억 원 초과 | 2.0\~2.5% | 적극적 협상 가능 |

신규 사업자는 매출 데이터가 없어서 처음에 '일반' 등급(가장 높은 수수료)으로 적용된다. 하지만 국세청에 매출신고가 반영되면 영세/중소 등급으로 자동 재산정되고, 그동안 더 낸 차액을 환급받을 수 있다. 사업 시작 후 첫 재산정 시점(1월 또는 7월)에 반드시 확인하자.

### PG사별 등급별 실제 수수료 비교

카드사 우대수수료는 PG 수수료에 포함된 구조다. 아래는 **카드사 수수료 + PG 마진이 합산된 가맹점이 실제로 내는 수수료**다. (2025년 2월 기준, 신용카드)

| PG사 | 일반 (30억+) | 영세 (3억↓) | 중소1 (3\~5억) | 중소2 (5\~10억) | 중소3 (10\~30억) |
|------|-------------|------------|--------------|---------------|---------------|
| 토스페이먼츠 | 3.20% | 1.65% | 2.25% | 2.40% | 2.65% |
| NHN KCP | 3.20% | 1.72% | 2.32% | 2.47% | 2.72% |
| 나이스페이먼츠 | 3.20% | 1.70% | 2.30% | 2.46% | 2.70% |
| KG이니시스 | 3.20% | 1.90% | 2.50% | 2.65% | 2.90% |

**일반 등급 수수료는 전부 3.20%로 동일하다.** 차이가 나는 건 영세/중소 등급에서의 PG 마진이다. 가입비는 대부분 22만 원 내외이고, 제휴 링크나 프로모션으로 면제되는 경우도 많다. 정산 주기도 D+5\~7 영업일로 대동소이하다.

결국 PG사를 고를 때 수수료율 차이보다 더 중요한 건 **가입비 부담과 연동 편의성**이다. 부트페이를 통해 가입하면 가입비 22만 원에 PG 심사·연동이 한 번에 처리되고, 부트페이 서비스 이용료가 PG 수수료에 포함되어 있어 별도 비용이 없다. 하나의 SDK로 여러 PG사를 붙일 수 있어서 초기 비용과 개발 공수를 동시에 줄일 수 있다.

### 수수료 협상, 현실적으로 언제 가능한가

수수료율은 테이블로 정해져 있다. 결국 **수수료 협상력 = 매출**이다. PG사 입장에서 수수료를 깎아줄 이유는 딱 하나, "이 가맹점이 빠지면 아까운 거래량"이 있느냐다.

**수수료 협상은 국세청 매출신고 기준 중소3 등급(연매출 10\~30억 원) 이상부터 현실적으로 가능하다.** 그 이하 구간은 카드사 우대수수료가 자동 적용되기 때문에, PG사가 추가로 깎아줄 여지가 거의 없다. "수수료 좀 깎아주세요"라고 연락하면 PG사 영업 담당자가 가장 먼저 요구하는 건 **최근 3개월간 PG 거래내역**이다. 이걸 보고 실제 거래 규모를 확인한 다음에야 협상이 시작된다. 거래내역을 내밀 수 없는 규모라면, 협상 테이블에 앉을 수 없다.

| 연매출 구간 (국세청 기준) | 수수료 협상 가능성 | 현실적 전략 |
|------------------------|-----------------|-----------|
| 3억 이하 (영세) | **불가** — 카드사 우대수수료 자동 적용 | 가입비·연관리비가 낮은 PG 선택에 집중 |
| 3\~10억 (중소1\~2) | **거의 불가** — 우대수수료가 이미 낮음 | 정산 주기, 결제 성공률에 집중 |
| 10\~30억 (중소3) | **협상 가능** — 3개월 거래내역으로 증명 | PG 영업 담당자 배정 요청. 수수료·정산 주기 동시 협상 |
| 30억 초과 (일반) | **적극 협상** — 복수 PG 견적 비교 가능 | 멀티PG 도입 검토, 연 단위 계약으로 추가 할인 |

> **실무 팁**: 연 매출 10억 미만이라면 수수료 협상에 시간을 쓰지 마라. PG사도 사업이다. 월 거래액 수백만 원인 가맹점의 수수료를 건별로 조정해줄 인센티브가 없다. 이 구간에서 실제로 줄일 수 있는 비용은 PG 가입비·연관리비 같은 고정 비용이다. 부트페이처럼 통합결제 SDK를 통해 가입하면 가입비에 PG 연동이 포함되어 있어, 별도 PG 가입비를 내지 않아도 된다. 수수료 협상은 매출이 10억을 넘기고, 최근 3개월 거래내역을 자신 있게 내밀 수 있는 규모가 됐을 때 하면 된다.

## 4. 심사에서 실제로 떨어지는 사유들

PG 심사가 까다롭다는 인식이 있는데, 실제로는 대부분의 반려가 **미리 알았으면 피할 수 있는 것들**이다.

### 반려 3대 사유

**1. 사이트 미구축**

PG사는 심사 시 실제 서비스 URL을 확인한다. "아직 개발 중이에요"는 통하지 않는다.

최소한 갖춰야 하는 것:
- 상품/서비스 소개 페이지 (가격 포함)
- 이용약관 페이지
- 개인정보처리방침 페이지
- 회사/사업자 정보 (상호, 대표자, 사업자번호, 연락처)

**2. 위험업종 해당**

아래 업종은 사전에 PG사에 문의해야 한다:

| 분류 | 예시 | 대응 |
|------|------|------|
| 입점 불가 | 도박/사행성, 불법 의약품 | PG 결제 자체가 불가 |
| 고위험 | 포인트/크레딧 충전, 게임아이템 거래, 크라우드펀딩 | PG사에 사전 문의. 추가 서류 요구 가능 |
| 조건부 가능 | 구독 서비스, 이용권/쿠폰형, 예약+선결제형 | 서비스 모델을 상세히 설명하면 대부분 통과 |

**3. 서류 미비 / 불일치**

- 사업자등록증의 업종 코드와 실제 서비스가 다른 경우
- 정산 계좌 명의와 사업자 대표가 다른 경우
> **참고**: 과거에는 카드사 심사에 통신판매업 신고번호가 필수였지만, 현재는 필수가 아니다. 통신판매업 신고를 하려면 PG사에서 발급하는 구매안전서비스 이용확인증(에스크로 확인증)이 필요하다. PG사 계약 시 에스크로 확인증 발급을 요청하면 1~2일 내에 발급받을 수 있다.

### 심사 통과율 높이는 실무 팁

1. **서류를 한 번에 제출하라** — 서류 보완 요청이 오면 재심사 큐로 들어가서 7일이 추가된다
2. **서비스 설명서를 첨부하라** — 특히 디지털 콘텐츠, 구독, 예약형 서비스는 비즈니스 모델 설명 문서를 함께 제출하면 반려 확률이 낮아진다
3. **환불/교환 정책을 명시하라** — 이용약관에 환불 조건이 없으면 심사에서 걸린다

## 5. 결제 플로우 전체 시퀀스

실제 결제가 일어나는 과정을 이해하면, 기획자는 각 단계에서 뭘 결정해야 하는지, 개발자는 뭘 구현해야 하는지가 명확해진다.

```
[고객: 주문서 작성]
    ↓
[프론트엔드: 결제 요청 (SDK 호출)]
    ↓
[PG사: 결제창 표시 → 카드 인증]
    ↓
[PG사 → 카드사: 승인 요청]
    ↓
[카드사: 승인/거절 응답]
    ↓
[PG사 → 서버: 웹훅으로 결제 결과 전달]
    ↓
[서버: 주문 상태 업데이트 + 고객 안내]
    ↓
[PG사: 정산일에 가맹점 계좌로 입금]
```

### 각 단계에서 결정해야 하는 것

| 단계 | 기획자가 결정 | 개발자가 구현 |
|------|-------------|-------------|
| 주문서 | 어떤 정보를 받을지 (배송지, 쿠폰 등) | 주문 데이터 저장, 금액 계산 |
| 결제 요청 | 어떤 결제수단을 노출할지 (카드, 계좌이체, 간편결제) | SDK 초기화, 결제 파라미터 전달 |
| 인증/승인 | - (PG사가 처리) | - |
| 결제 결과 | 성공/실패 시 사용자에게 보여줄 화면 | 웹훅 수신, 결제 검증, DB 업데이트 |
| 정산 | 정산 주기 확인, 캐시플로우 계획 | 정산 대사(PG 내역과 DB 비교) 로직 |

### 코드로 보면 이렇다

결제 요청은 프론트엔드에서 SDK를 호출하는 것으로 시작한다. 아래는 Bootpay SDK 기준 예시다:

```javascript
// 프론트엔드: 결제 요청
const response = await Bootpay.requestPayment({
  application_id: '내_APPLICATION_ID',
  price: 29900,
  order_name: '프리미엄 플랜 (1개월)',
  order_id: 'order_' + Date.now(),
  pg: '나이스페이',                   // 사용할 PG사
  method: 'card',                    // 결제수단
  user: {
    username: '홍길동',
    phone: '01012345678'
  }
})

// 결제 완료 시 서버에서 검증
if (response.event === 'done') {
  await verifyPayment(response.receipt_id)
}
```

결제가 완료되면 서버에서 결제 내역을 검증한다. **금액 위변조 방지를 위해 서버에서 반드시 재확인해야 한다:**

```javascript
// 서버: 결제 검증
import { Bootpay } from '@bootpay/backend-js'

app.post('/api/payment/verify', async (req, res) => {
  const { receipt_id } = req.body

  await Bootpay.getAccessToken()
  const receipt = await Bootpay.receiptPayment(receipt_id)

  if (receipt.status === 1 && receipt.price === 주문금액) {
    // 검증 성공 → 주문 상태 업데이트
    await db.orders.update({
      bootpay_receipt_id: receipt_id,
      status: 'paid',
      paid_at: new Date()
    }, { where: { order_id } })
    res.json({ success: true })
  } else {
    // 검증 실패 → 결제 취소
    await Bootpay.cancelPayment({
      receipt_id,
      cancel_price: receipt.price,
      cancel_reason: '금액 불일치'
    })
    res.json({ success: false })
  }
})
```

웹훅도 함께 구현해야 한다. 고객이 결제 완료 후 브라우저를 닫아버리면 프론트엔드 콜백이 실행되지 않는데, 웹훅은 PG사가 서버로 직접 보내기 때문에 결제 결과를 놓치지 않는다:

```javascript
// 서버: 웹훅 수신
app.post('/webhook/bootpay', async (req, res) => {
  const { receipt_id, status, price, order_id } = req.body

  // 위변조 방지를 위해 결제 조회 API로 재검증
  await Bootpay.getAccessToken()
  const receipt = await Bootpay.getReceipt(receipt_id)

  if (receipt.status === status && receipt.price === price) {
    switch (status) {
      case 1:  // 결제 완료
        await db.orders.update(
          { status: 'paid', paid_at: new Date() },
          { where: { order_id } }
        )
        break
      case 20: // 결제 취소
        await db.orders.update(
          { status: 'refunded' },
          { where: { bootpay_receipt_id: receipt_id } }
        )
        break
    }
  }

  res.status(200).json({ status: 200 })
})
```

핵심 패턴은 **프론트에서 결제 요청 → 서버에서 금액 검증 → 웹훅으로 결과 수신** 3단계다.

> **실무 팁**: 결제만 구현하고 환불/취소 처리를 안 만드는 경우가 정말 많습니다. 라이브 첫날부터 "결제 취소해주세요" CS가 들어옵니다. 전체 취소와 부분 취소 API를 반드시 함께 구현하고, 환불 요청을 처리할 수 있는 관리자 화면도 준비하세요.

## 6. 자주 하는 질문

**Q. 사업자등록 없이 결제 테스트할 수 있나요?**

대부분의 PG사가 테스트 키를 발급해준다. 테스트 환경에서는 사업자 없이도 연동 개발이 가능하다. 다만 **실결제(라이브)를 위해서는 사업자등록이 필수**다.

**Q. 간이과세자도 PG 가입이 되나요?**

된다. 현재는 카드사 심사에 통신판매업 신고번호가 필수가 아니므로, 면제 대상이라면 PG 가입 후 필요할 때 신고해도 된다.

**Q. 개발자 없이 결제를 붙일 수 있나요?**

쇼핑몰 빌더(카페24, 아임웹 등)를 사용하면 코드 없이 결제를 붙일 수 있다. 자체 서비스에 결제를 넣으려면 개발자가 필요하지만, 통합결제 SDK를 쓰면 연동 자체는 1~3일이면 가능하다.

## 체크리스트: 결제 도입 준비물 한 장 정리

### 사업 준비

- [ ] 사업자등록 완료
- [ ] 통신판매업 신고 완료 (간이과세자 직전 반기 50건 미만이면 면제. 카드사 심사에는 현재 필수 아님)
- [ ] 정산 받을 사업자/법인 명의 계좌 준비

### 사이트 준비 (PG 심사용)

- [ ] 상품/서비스 소개 페이지 (가격 포함)
- [ ] 이용약관 페이지
- [ ] 개인정보처리방침 페이지
- [ ] 사업자 정보 표시 (상호, 대표자, 사업자번호, 연락처, 이메일)
- [ ] 환불/교환 정책 명시

### PG 심사 서류 (개인사업자 기준)

- [ ] PG 가입 신청서
- [ ] 사업자등록증 사본
- [ ] 대표자 신분증 사본
- [ ] 인감증명서
- [ ] 통신판매업 신고증
- [ ] 정산 계좌 통장 사본

### 개발 준비

- [ ] HTTPS 적용 (SSL 인증서)
- [ ] 결제 SDK 연동 (테스트 키로 먼저)
- [ ] 웹훅 수신 엔드포인트 구현
- [ ] 결제 검증 로직 (금액 위변조 방지)
- [ ] 환불/취소 API 구현
- [ ] 관리자 화면에서 결제 내역 조회 + 환불 처리 가능하도록

### 라이브 전 최종 확인

- [ ] 테스트 키 → 운영 키 교체
- [ ] 웹훅 URL이 운영 도메인을 가리키는지 확인
- [ ] HTTPS 인증서 유효한지 확인
- [ ] 결제 성공/실패/취소 시나리오 실결제 테스트
- [ ] 영세/중소 우대수수료 적용 일정 확인 (1월/7월 재산정)
