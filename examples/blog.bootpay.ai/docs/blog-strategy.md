# Bootpay 블로그 전략 v2

> blog.bootpay.ai — "결제 기획하려는데 뭐부터 해야 하나요?"에 대한 답

## 전략 전환: SEO 물량 → 커뮤니티에서 퍼지는 가이드

### 왜 바꾸는가

기존 전략은 "PG수수료비교", "결제연동 방법" 같은 키워드별 낱개 글 20개로 SEO를 잡는 것이었다.
문제는 이 시장에서 포트원이 200+개 글로 이미 SEO를 선점했고, 같은 방식으로는 이기기 어렵다는 것.

더 큰 문제: **실제로 결제를 기획하려는 사람이 원하는 건 낱개 글이 아니라 "한 번에 그림이 잡히는 가이드"다.**

### 노리는 장면

개발 커뮤니티(GeekNews, disquiet, 개발자 카톡방, Reddit r/korea 등)에서 이런 질문이 올라온다:

- "사이드 프로젝트에 결제 넣으려는데 뭐부터 해요?"
- "SaaS 구독 모델 만드려는데 정기결제 어떻게 해요?"
- "중개 플랫폼인데 판매자 정산 어떻게 하죠?"

→ 누군가 **"이거 보세요"** 하고 링크 하나 던져주는 글. 그게 목표.

### 퍼지는 콘텐츠의 조건

1. 질문자의 상황에 정확히 맞을 것
2. 읽고 나면 **다음 행동이 명확**해질 것
3. 겪어본 사람만 아는 **삽질 포인트**가 있을 것
4. 비교표, 체크리스트 — **저장해두고 다시 볼** 포맷일 것
5. 특정 제품을 밀지 않는 **중립적 교육 콘텐츠**일 것 (→ 신뢰 → 자연스러운 Bootpay 인지)

## 경쟁사 콘텐츠 갭 분석

### 포트원

| 주제 | 콘텐츠 양 | 깊이 | 빈틈 |
|------|----------|------|------|
| 결제 도입 | 많음 (10+편) | PG 비교는 상세, 아키텍처는 약함 | 결제 시스템 설계론, 테스트 환경 구축, 법적 고려사항 없음 |
| 구독/정기결제 | 적음 (3편) | 빌링키 설명 위주 | 구독 모델 설계, 이탈 방지, 프로레이션, 쿠폰 등 기획 관점 전무 |
| 마켓플레이스 정산 | 많음 (8편) | 법률(전금법) 대응은 강함 | 정산 모델 비교, 아키텍처 설계, 분쟁 처리 없음. 제품 홍보 톤 |
| 결제 운영 | 중간 (8편) | 멀티PG 반복 | 실무 운영 가이드, 장애 플레이북, CS 대응, fraud 탐지 없음 |

**전체 톤**: "포트원 쓰면 해결됩니다" — 리드젠 마케팅 콘텐츠. 도메인 교육은 약함.

### 토스페이먼츠

| 주제 | 콘텐츠 양 | 깊이 | 빈틈 |
|------|----------|------|------|
| 결제 도입 | 많음 (8~10편) | 기획자 시리즈가 좋음 | 업종별 가이드, 비즈니스 의사결정 관점 부족 |
| 구독/정기결제 | 적음 (4~5편) | 개발 연동만 존재 | 기획/설계 관점 전무 — 플랜 구조, 이탈 방지, 환불 정책 등 없음 |
| 마켓플레이스 정산 | 적음 (3~4편) | 제품(지급대행) 소개 수준 | 정산 설계 전략 콘텐츠 전무 |
| 결제 운영 | 산발적 (5~6편) | 단편적 트러블슈팅 | 체계적 운영 가이드, 모니터링 체계, 장애 대응 없음 |

**전체 톤**: 기술 블로그와 사장님 매거진 이원화. 둘 다 자사 제품(결제위젯, 상점관리자) 전환 유도 중심.

### 양쪽 모두 없는 것 (= Bootpay 기회)

1. **"기획자가 먼저 읽어야 할" 구독 설계 가이드** — 빌링키 연동법은 있지만, 플랜 구조/프로레이션/이탈 방지 등 기획 관점이 없음
2. **마켓플레이스 정산 설계론** — 둘 다 자사 제품 홍보 수준. "어떤 정산 모델이 우리에게 맞는가"를 다루는 글이 없음
3. **결제 운영 실무 가이드** — 장애 대응 플레이북, 정산 대사, fraud 탐지 등 라이브 이후 콘텐츠가 거의 없음
4. **제품 중립적 교육 콘텐츠** — 양쪽 모두 "우리 제품 쓰세요"로 귀결. 커뮤니티에서 공유하기 민망한 톤

## 콘텐츠 구성: 4개의 "결정적 가이드"

### 가이드 1: 결제 처음 붙이려는데 뭐부터 해야 하나요?

> **커뮤니티 질문 원형**: "사이드 프로젝트에 결제 넣으려는데", "쇼핑몰 만드는데 결제 어떻게"

**다루는 것:**

1. **결제가 붙기까지의 전체 그림**
   - 사업자등록 → PG심사 → 연동 → 테스트 → 라이브, 전체 타임라인
   - 각 단계별 소요 시간과 병렬 처리 가능한 것들

2. **"나는 어떤 케이스인가" 분기표**
   - 개인사업자 vs 법인
   - 실물 상품 vs 디지털 콘텐츠 vs 서비스
   - 국내만 vs 해외 포함
   - → 케이스별로 준비물과 선택지가 달라짐을 보여줌

3. **PG사 선택: 수수료 비교가 아니라 "우리 상황에 맞는 선택"**
   - 월 거래액 규모별 추천 (소액이면 수수료보다 심사 편의성이 중요)
   - 결제수단 조합별 PG사 지원 현황
   - 정산 주기가 왜 중요한지 (캐시플로우 관점)

4. **심사에서 실제로 떨어지는 사유들**
   - 위험 업종 기준과 대응 전략
   - 서류 미비로 지연되는 흔한 실수
   - 심사 통과율 높이는 실무 팁

5. **결제 플로우 전체 시퀀스**
   - 주문서 → 결제 요청 → 인증 → 승인 → 웹훅 → 정산
   - 각 단계에서 기획자가 결정해야 하는 것과 개발자가 구현해야 하는 것

6. **체크리스트: 결제 도입 전 준비물 한 장 정리**

**경쟁사 대비 차별점:**
- 포트원은 PG 비교표는 상세하지만 "내 상황에 맞는 판단"을 도와주지 않음
- 토스페이먼츠 기획자 시리즈(Part 1, 2)가 좋은 시도지만, 결제수단 설명에 그치고 전체 여정을 커버하지 않음
- 양쪽 다 결제 시스템 설계(멱등성, 상태 머신, 테스트 환경)에 대한 기술적 깊이가 없음

**목표 분량**: 5000~7000자. 한 번 읽으면 전체 그림이 잡히는 수준.

### 가이드 2: 구독/정기결제 기획하려는데 뭘 알아야 하나요?

> **커뮤니티 질문 원형**: "SaaS 구독 모델 만드려는데", "정기결제 어떻게 구현해요", "빌링키가 뭔가요"

**다루는 것:**

1. **구독 결제가 일반 결제와 다른 점**
   - 빌링키 개념: 왜 필요한지, 어떻게 작동하는지
   - 인증 결제 vs 비인증 결제 — 언제 뭘 쓰는지
   - PG사별 정기결제 지원 현황과 차이점

2. **기획 단계에서 먼저 결정해야 하는 것들** (← 이 부분이 양쪽 다 없음)
   - 과금 모델: 정액제 / 종량제 / 시트당 과금 / 하이브리드
   - 플랜 구조: 몇 개의 플랜? 연/월 가격 차이? 무료 플랜?
   - 무료체험: 기간, 카드 등록 시점, 자동 전환 여부
   - 업그레이드/다운그레이드: 즉시 적용 vs 다음 결제일 적용, 프로레이션(일할 계산)
   - 해지 흐름: 즉시 해지 vs 기간 만료 해지, 해지 사유 수집

3. **결제 실패 시 설계** (← 가장 많이 놓치는 부분)
   - 재시도 정책: 며칠 간격? 몇 번? (보통 3일 간격 × 3회)
   - 유예 기간(grace period): 결제 실패 후 서비스 유지 기간
   - 카드 만료/변경 시 빌링키 갱신 플로우
   - 고객 알림: 결제 실패 안내 이메일/푸시 타이밍

4. **환불 정책 설계**
   - 구독 환불 계산: 이용일수 기준? 고정 금액?
   - 부분 환불 시나리오: 플랜 다운그레이드, 중도 해지
   - 공정거래위원회 기준과 실무 간 간극

5. **의사결정 맵**: "우리 서비스 유형 → 추천 구독 구조"
   - 콘텐츠 구독 (넷플릭스형)
   - SaaS 도구 (노션형)
   - 정기배송 (마켓컬리형)
   - 각각의 추천 플랜 구조와 주의점

**경쟁사 대비 차별점:**
- 포트원: 빌링키·슈퍼빌링키 기술 설명은 있지만, 기획자가 결정해야 할 것들(플랜 설계, 이탈 방지, 환불 정책)이 전혀 없음
- 토스페이먼츠: 빌링키 발급·스케줄링 API 연동 가이드만 있음. "구독 비즈니스를 어떻게 설계하는가"는 없음
- **양쪽 다 "개발자가 빌링키를 이렇게 구현하세요"에 그침. "기획자가 먼저 이걸 정해야 합니다"가 없음**

**목표 분량**: 6000~8000자. 기획자가 읽고 의사결정을 내릴 수 있는 수준.

### 가이드 3: 마켓플레이스/플랫폼인데 정산을 어떻게 하죠?

> **커뮤니티 질문 원형**: "중개 플랫폼인데 판매자한테 정산 어떻게", "수수료 떼고 보내려면"

**다루는 것:**

1. **플랫폼 결제의 구조: 돈의 흐름**
   - 구매자 → PG → 플랫폼 → 판매자, 전체 플로우 다이어그램
   - "내가 돈을 받았다가 나눠주는 건가, PG가 나눠주는 건가" — 이 질문이 핵심

2. **정산 모델 3가지와 선택 기준**
   - **모델 A: 플랫폼 수취 후 지급** — 플랫폼이 전액 받고, 수수료 떼고 판매자에게 이체
     - 장점: 구현 단순. 단점: 전금법상 자금이체업 등록 필요할 수 있음
   - **모델 B: PG 분리정산** — PG가 결제 시점에 플랫폼/판매자 몫을 분리
     - 장점: 법적 리스크 낮음. 단점: PG 지원 필요, 유연성 제한
   - **모델 C: 에스크로** — 구매 확정 전까지 PG가 보관
     - 장점: 소비자 보호. 단점: 정산 지연
   - **판단 가이드**: 월 거래액, 판매자 수, 업종에 따른 추천 모델

3. **기획자가 놓치는 세무/법률 이슈**
   - 세금계산서: 플랫폼이 발행? 판매자가 발행?
   - 원천징수: 판매자가 개인일 때 (3.3% 원천징수)
   - 전자금융거래법 개정: 2024.9 시행, 자금이체업 등록 의무
   - 부가세 처리: 플랫폼 수수료의 부가세

4. **정산 주기 설계**
   - D+7? 구매확정 후? 월 정산?
   - 캐시플로우 관점: 환불 대비 유보금(reserve) 필요성
   - 판매자 입장의 정산 주기 기대치

5. **환불 시 정산 취소 처리**
   - 이미 정산한 금액의 환불: 판매자에게 반환 청구? 다음 정산에서 차감?
   - 부분 환불 시 수수료 재계산
   - 정산 분쟁 시나리오와 대응

6. **규모별 현실적 접근**
   - 월 거래 1000만 원 이하: 수동 정산(엑셀)으로 충분
   - 월 거래 1000만~1억: 정산 자동화 필요
   - 월 거래 1억+: 정산 시스템 자체 구축 or 솔루션 도입

**경쟁사 대비 차별점:**
- 포트원: 파트너 정산 제품 홍보 + 전금법 대응은 잘 다루지만, "어떤 정산 모델이 맞는가"를 스스로 판단하게 돕는 글이 없음
- 토스페이먼츠: 지급대행 제품 소개 수준. 정산 설계 전략 콘텐츠 전무
- **양쪽 다 "우리 정산 솔루션 쓰세요"로 귀결. 정산을 처음 설계하는 기획자를 위한 중립적 교육이 없음**
- 한국어로 마켓플레이스 정산을 제대로 다루는 글 자체가 희소 → 쓰기만 하면 퍼짐

**목표 분량**: 6000~8000자.

### 가이드 4: 결제 붙였는데 운영하면서 뭘 챙겨야 하나요?

> **커뮤니티 질문 원형**: "결제 장애 나면 어떡해요", "정산 안 맞을 때", "결제 성공률이 갑자기 떨어졌어요"

**다루는 것:**

1. **결제 성공률 모니터링**
   - 정상 범위: 카드 95%+, 간편결제 98%+ (이 아래면 문제)
   - 떨어졌을 때 보는 순서: PG 상태 → 카드사 점검 → 우리 쪽 에러 로그
   - 결제수단별/시간대별 성공률 추적 방법

2. **정산 대사(reconciliation)**
   - PG 정산 내역과 우리 DB가 안 맞는 이유들
   - 대사 자동화: 일일/주간 체크 스크립트 설계
   - 오차 허용 범위와 에스컬레이션 기준

3. **장애 대응 플레이북**
   - PG사 장애 vs 우리 쪽 장애 구분법
   - PG사 장애 시: 폴백 결제(멀티PG), 고객 안내 문구, CS 대응
   - 우리 쪽 장애 시: 웹훅 누락 복구, 미승인 건 처리, 이중결제 방지
   - 장애 후 체크리스트: 누락된 결제 건 확인, 정산 영향 파악

4. **환불/취소 처리 플로우**
   - 전체 취소 vs 부분 취소 vs 부분 환불: 언제 뭘 쓰는지
   - PG사별 부분취소 지원 차이
   - 환불 기한: 카드사별 차이와 초과 시 대응

5. **부정결제(fraud) 탐지**
   - 실제 패턴: 도용 카드 소액 테스트 → 고액 결제, 새벽 시간대 집중 등
   - FDS(이상거래탐지) 기본 룰 설계
   - 차지백(chargeback) 대응 프로세스

6. **운영 체크리스트**
   - 매일: 전일 결제 성공률, 정산 대사, 미처리 건 확인
   - 매주: 결제수단별 트렌드, 환불률, CS 인입 패턴
   - 매월: PG사 청구서 확인, 수수료 정산, 보안 점검
   - 매년: PG 계약 갱신, 수수료 재협상, 인증서 갱신

**경쟁사 대비 차별점:**
- 포트원: "멀티PG 쓰면 장애 대비 가능" 반복. 실제 장애 발생 시 뭘 하는지 플레이북이 없음
- 토스페이먼츠: 에러 코드별 대응, FDS 안내 등 단편적 트러블슈팅만 존재. 체계적 운영 가이드 없음
- **양쪽 다 "결제 연동"까지만 다루고, "라이브 이후"를 다루는 콘텐츠가 거의 없음**
- 결제 운영은 실무자만 아는 영역 → 이 글 하나면 운영팀 온보딩 자료로 쓸 수 있음

**목표 분량**: 5000~7000자.

## CTA 설계: 세일즈가 아니라 자연스러운 연결

양쪽 경쟁사의 가장 큰 문제는 **모든 글이 "우리 제품 쓰세요"로 끝난다는 것**. 커뮤니티에서 이런 글을 공유하면 "광고잖아"라는 반응이 나온다.

Bootpay 가이드의 CTA 원칙:
- 글 본문은 **100% 중립적 교육 콘텐츠**. 특정 제품을 밀지 않음
- 글 끝에 "이 가이드에서 설명한 것들을 직접 구현하려면" 섹션으로 자연스럽게 연결:
  - 개발자: "Bootpay SDK로 시작하기 →" (ai-docs 링크)
  - 기획자: "도입 상담 신청 →" (문의 폼)
- 중간중간 코드 예제가 필요한 부분에서 Bootpay SDK를 "예시 중 하나"로 자연스럽게 노출
  - "예를 들어 Bootpay SDK로 빌링키를 발급하면:" (코드 블록)
  - "포트원이나 토스페이먼츠에서도 비슷한 API를 제공합니다"

→ 이렇게 하면 **"광고 아니고 진짜 좋은 글"**로 인식되면서, 자연스럽게 Bootpay를 알게 됨.

## 시즌 2: 실전 사례편 (가이드 1~4 이후)

가이드 1~4가 "이론/기획 관점"이라면, 실전 사례편은 **"실제로 이렇게 했습니다"**다.
가이드를 읽고 "좋은데, 실제로 적용하면 어떻게 되는 거지?"라는 다음 질문에 답하는 콘텐츠.

### 사례 1: SaaS 구독 결제 — "월 9,900원짜리 구독 서비스를 처음부터 만든 과정"

- 과금 모델 선택 (정액제 vs 시트당 → 정액제로 결정한 이유)
- 플랜 3개 설계 과정 (Free/Pro/Team — 각 플랜의 가격과 기능 경계를 어떻게 정했나)
- 연/월 결제 할인율 결정, 중도 해지 환불 정책 수립
- 결제 실패 재시도 정책 적용 결과 (dunning으로 비자발적 이탈 몇 % 복구했는지)
- 삽질 포인트: 실제로 겪은 문제와 해결

### 사례 2: 이커머스 첫 결제 도입 — "사이드 프로젝트에서 월 매출 1000만 원까지"

- 개인사업자로 PG 심사 통과한 과정 (서류 준비, 반려 경험)
- PG사 선택 기준 (MVP 단계 → 성장 단계에서 기준이 바뀐 시점)
- 영세/중소 우대수수료 적용 전후 비용 차이
- 정산 주기가 캐시플로우에 미친 영향
- 삽질 포인트: 테스트→라이브 전환, 첫 환불 처리

### 사례 3: 플랫폼 정산 설계 — "중개 플랫폼에서 판매자 정산을 자동화한 과정"

- 정산 모델 선택 (플랫폼 수취 후 지급 vs 분리정산 → 어떤 기준으로 결정했나)
- 세금계산서/원천징수 처리
- 환불 발생 시 정산 취소 처리
- 수동 정산(엑셀) → 자동화 전환 시점과 방법
- 삽질 포인트: 정산 오차, 분쟁 처리

**포맷**: 각 사례 3000~5000자. 가이드 본문 링크로 "더 자세한 기획 가이드는 여기"를 연결.
**차별점**: 경쟁사에 실전 사례가 거의 없음. 특히 숫자(매출, 이탈률, 복구율)가 있는 사례는 한국어 콘텐츠에서 희소.

## 글 작성 원칙 (개정)

1. **"친구한테 설명하듯"** — 전문 용어 쓸 때 반드시 한 줄 설명 추가
2. **의사결정을 도와줄 것** — 정보 나열이 아니라 "이런 상황이면 이걸 선택해라" 판단 기준 제공
3. **실제 삽질 경험** — "보통 여기서 막힙니다", "이건 문서에 안 나와 있지만" 같은 실전 팁
4. **비교표/체크리스트 필수** — 저장해두고 다시 볼 수 있는 포맷
5. **제품 중립** — 본문에서 Bootpay를 밀지 않음. CTA만 자연스럽게 연결
6. **연도 표기 지양** — "2026년 최신"보다 구조적으로 오래 가는 콘텐츠. (수수료 같은 건 별도 업데이트 노트로)
7. **코드는 설명의 도구** — 전체 구현이 아니라, 개념을 이해시키기 위한 핵심 스니펫만
