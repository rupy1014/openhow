# Bootpay YouTube 채널 컨셉

## 채널 정체성

```yaml
채널명: 부트페이 (Bootpay 공식)
컨셉: "이런게 된다고?" — 기획자를 위한 결제 실무 채널
타겟: 결제/구독 서비스를 기획하는 PM, 기획자, 사업 담당자
톤: 친구 PM한테 커피 한 잔 하면서 듣는 실무 팁
```

## 왜 이 채널인가?

결제 관련 YouTube 콘텐츠는 대부분 **개발자 대상**(API 연동, SDK 사용법)이다.
기획자가 필요한 콘텐츠 — PG 선택, 수수료 구조, 심사 프로세스, 구독 설계 — 는 거의 없다.

```yaml
시장 빈틈:
  개발자 대상: 포화 (공식 문서, 블로그, 강의 많음)
  기획자 대상: 거의 없음
    - PG 수수료 비교? → 블로그 글 몇 개뿐
    - 구독결제 설계? → 체계적 콘텐츠 전무
    - 심사 반려 대응? → 커뮤니티 질문만 떠돌아다님

우리의 기회:
  "결제 기획하려는데 뭐부터 해야 해?" → 이 영상 보세요
  "구독결제 빌링키가 뭐예요?" → 이 영상 보세요
```

## 캐릭터: 태섭

```yaml
역할: 1인 설명자 (narrator)
페르소나:
  - 32세 남성, 결제/핀테크 분야 경험이 풍부한 PM
  - PG 연동부터 구독결제 설계까지 직접 해본 실무자
  - 친구에게 설명하듯 편하게 말하는 스타일

이미지:
  - ai-jobdori 프로젝트의 캐릭터 이미지 공유
  - 스마트 캐주얼 (셔츠, 가디건, 청바지)
  - 따뜻한 미소, 신뢰감 있는 인상

말투:
  - "~거든요", "~잖아요", "~에요"
  - "이게 중요한 게요," (강조 전환)
  - "근데 여기서 함정이 하나 있어요." (삽질 포인트 전)
  - "이거 모르면 진짜 고생해요." (경험 기반)
  - "자, 그러면" (섹션 전환)

인사:
  - 오프닝: "안녕하세요. 부트페이입니다."
  - 클로징: "도움이 됐다면 구독 부탁드려요. 다음 영상에서 만나요."
```

## 영상 포맷

### 구조 (1인 설명)

```yaml
영상 길이: 5~10분 (집중력 유지)

구조:
  훅 (0:00~0:30):
    - "이런게 된다고?" 느낌의 충격적 사실/질문
    - 시청자가 "어 뭐지?" 하고 멈추게 만드는 첫 문장
    - 예: "PG 심사에서 업종 코드 하나 때문에 두 달을 날렸다는 거 아세요?"

  인트로 (0:30~1:00):
    - "안녕하세요. 부트페이입니다."
    - 오늘 주제 한 줄 소개

  본론 (1:00~N:00):
    - 가이드 핵심 내용을 영상용으로 재구성
    - 비교표, 숫자, 프로세스 시각화
    - 삽질 포인트 강조

  정리 (마지막 1분):
    - 핵심 3줄 요약
    - "하나만 기억하신다면: {핵심}"
    - 블로그 가이드 링크 안내

  아웃트로 (마지막 15초):
    - 구독 유도 + 다음 영상 예고
```

### 차별화 요소

```yaml
훅 패턴:
  - "이거 아세요?" (몰랐던 사실)
  - "이거 안 하면 어떻게 될까요?" (리스크)
  - "실제로 이런 일이 있었어요." (사례)

삽질 포인트:
  - 가이드의 "💡 실무 팁"을 영상에서는 더 극적으로 연출
  - "근데 여기서 함정이 하나 있어요..." 패턴

비교/시각화:
  - 가이드의 비교표 → comparison 씬
  - 가이드의 체크리스트 → flow 씬
  - 수수료/비용 → chart/stats 씬
```

## 콘텐츠 로드맵

```yaml
시즌 1 — 기존 가이드 영상화:
  EP01: "결제 기획, 뭐부터 해야 하나요?" (payment-getting-started)
  EP02: "구독결제, 이것만 알면 됩니다" (subscription-planning)
  EP03: "마켓플레이스 정산, 어떻게 설계하나요?" (marketplace-settlement)
  EP04: "결제 운영, 장애 나면 어쩌죠?" (payment-operations)

시즌 2 — 심화/트렌드:
  - PG 수수료 완전 비교 (매년 업데이트)
  - 해외결제 연동 가이드
  - 결제 보안/인증 트렌드
  - 실제 사례 분석
```

## 제품 중립 정책

```yaml
원칙:
  - 본문에서 Bootpay를 밀지 않음
  - "Bootpay 쓰면 이게 쉽다" ← 이런 거 금지
  - 정보 자체가 가치 → 신뢰 → 자연스러운 전환

CTA (영상 끝에서만):
  - "더 자세한 건 블로그 가이드에 정리해뒀어요. 링크는 설명란에."
  - "직접 구현하시려면 Bootpay 개발자 문서도 참고해보세요."

경쟁사:
  ❌ 포트원(PortOne) 이름 언급 절대 금지
  ✅ PG사(토스페이먼츠, 나이스페이먼츠, KCP, 이니시스) 중립 비교 OK
  ✅ 해외 서비스(Stripe, Paddle) 참고 사례 OK
```

## 씬 타입 (Bootpay 영상용)

ai-jobdori 기반이지만 1인 설명에 맞게 단순화:

```yaml
# 인트로/아웃트로
greeting:         # 캐릭터 인사 (립싱크 가능)
  필드: tts, display_text, image, lipsync

# 본론 (content)
narration:        # 기본 설명 (가장 많이 사용)
  필드: tts, display_text, highlight

stats:            # 수치 강조
  필드: tts, stat, detail

comparison:       # 비교표
  필드: tts, items[]

flow:             # 프로세스/단계
  필드: tts, steps[]

chart:            # 데이터 차트
  필드: tts, chart_title, chart_data[], source

screenshot:       # 화면 캡처
  필드: tts, image, caption

quote:            # 인용/강조 문구
  필드: tts, source

# 참고: speaker 필드는 항상 "narrator" (1인 설명)
# 참고: code 씬은 기획자 대상이므로 거의 사용 안 함
```
