### 반복되는 삽질 포인트 (커뮤니티 답변에서 추출)

1. 빌링(billing)과 결제(payment)를 같은 것으로 착각하고 시작
2. PG 선택 후에야 스케줄링 지원 여부가 다르다는 걸 알게 됨
3. 빌링키 발급은 됐는데, "언제 얼마를 청구할지"의 비즈니스 로직은 본인이 짜야 한다는 걸 모름
4. 카드 만료 시 빌링키가 무효화되는데, 고객에게 갱신을 안내하는 플로우를 미리 안 만들어둠
5. 무료체험 → 유료 전환 시 고객 동의 절차를 법적으로 어떻게 해야 하는지 모름

### 결제 실패 재시도 (dunning) 업계 표준

| 서비스 | 재시도 정책 | Grace Period | 특이사항 |
|--------|-----------|-------------|---------|
| Stripe | 권장: 8회 / 2주 이내 (Smart Retries) | 설정 가능 | ML 기반 500+ 속성으로 최적 재시도 시점 예측. 실패 결제 56% 복구 |
| YouTube | 3일간 주기적 재시도 | 실패 후 30일 일시중지 | 3일 이내 해결 시 혜택 유지 |
| Google Workspace | 기본 결제 실패 시 유예기간 시작 | 다음 달 초까지 | 유예기간 내 미해결 시 계정 정지 |
| Apple (앱스토어) | 자동 재시도 | Billing Grace Period 활성화 가능 | 유예기간 중 유료 콘텐츠 접근 유지 |
| Adobe | 남은 구독 기간 동안 계속 재시도 | 구독 기간 내 | 결제 완료 시 즉시 접근 복원 |

> 출처: Stripe docs, Google/Apple/Adobe 고객센터
> 확인일: 2026-03-17

**한국 시장 실무 표준 (종합)**:
- 일반적 재시도: 3일 간격 x 3회 (총 9일)
- Grace period: 결제 실패 후 3~7일 서비스 유지
- 고객 알림: 실패 즉시 + 재시도 전 + 서비스 중단 전 (최소 3회)
- Stripe 데이터: 비자발적 이탈(involuntary churn)이 전체 이탈의 25%. 복구된 구독자는 평균 7개월 추가 유지

## 4. 삽질 포인트 후보

### 기획 단계 삽질

1. **"과금 모델을 안 정하고 결제 연동부터 시작한다"**
   - 정액제/종량제/시트당 과금에 따라 빌링키 사용 방식, DB 설계, 청구 로직이 완전히 달라짐
   - 나중에 바꾸면 DB 마이그레이션 + 기존 고객 전환 이슈 발생

2. **"PG사별 스케줄링 지원 차이를 모르고 선택한다"**
   - 토스페이먼츠: 스케줄링 미지원 (직접 구현)
   - 포트원: schedule API 제공
   - 이 차이가 개발 공수에 2~4주 차이를 만들 수 있음

3. **"무료체험 → 유료 전환 시 법적 동의 절차를 빠뜨린다"**
   - 2025년부터 다크패턴 규제 시행. 무료체험 종료 후 자동 과금 시 사전 동의 없으면 위반
   - 결제 7일 전 안내 + 해지 방법 안내 필수

4. **"해지를 어렵게 만들면 이탈이 줄 것이라고 생각한다"**
   - 다크패턴 규제로 과태료 리스크 (실제 부과 사례: 쿠팡 250만, 웨이브 400만)
   - 가입 경로와 동일한 경로로 해지 가능해야 함
   - 오히려 해지 사유 수집 → 윈백 캠페인이 더 효과적

### 운영 단계 삽질

5. **"카드 만료 시 빌링키 갱신 플로우를 안 만들어둔다"**
   - 카드 유효기간은 보통 5년. 서비스 운영 2~3년차부터 대량 만료 시작
   - 일부 카드사는 갱신카드로 자동연결, 일부는 수동 업데이트 필요
   - 만료 1~2개월 전 안내 이메일 + 원클릭 카드 변경 페이지가 없으면 대량 이탈

6. **"결제 실패 시 즉시 서비스 차단한다"**
   - 일시적 잔액 부족, 카드사 점검 등 일시적 사유가 대부분
   - Grace period 없이 즉시 차단하면 고객 이탈 + CS 폭증
   - 업계 표준: 3~7일 유예 + 3회 재시도

7. **"환불 정책을 미리 정하지 않는다"**
   - 구독 환불은 단건 환불보다 복잡: 이용일수 기준? 고정금액? 남은 기간 일할?
   - 공정위 기준: 가분적 콘텐츠는 미제공 부분 환불 가능
   - 정책 미수립 시 CS마다 다른 기준 적용 → 고객 불만 + 법적 리스크

8. **"업그레이드/다운그레이드 시 프로레이션을 고려하지 않는다"**
   - 월 1만원 플랜에서 월 3만원 플랜으로 월 중간에 업그레이드하면?
   - 차액을 일할 계산해서 즉시 청구? 다음 결제일에 반영? 차액만 청구?
   - Paddle 기준 5가지 빌링모드: (1) 프로레이션+즉시청구 (2) 프로레이션+다음결제일 (3) 전액+즉시 (4) 전액+다음결제일 (5) 청구안함
   - 한국 PG는 이 로직을 자동으로 해주지 않음. 전부 직접 구현

## 5. 권장 목차

기존 blog-strategy.md 목차에 리서치 결과를 반영한 조정안:

### 구독/정기결제 기획하려는데 뭘 알아야 하나요?

**도입부**: "구독 서비스, 결제 연동보다 기획이 먼저다"
- 구독 서비스 구축에 평균 6~9개월 (StepPay/Disquiet 데이터)
- 결제 연동은 전체의 20%. 나머지 80%가 기획 의사결정

**1. 구독 결제, 일반 결제와 뭐가 다른가**
- 빌링키: 친구한테 설명하듯 (카드번호 대신 쓰는 암호화 키)
- 인증 vs 비인증 결제 — 표로 비교
- PG사별 정기결제 지원 현황 비교표 (스케줄링 지원 여부 포함)
  - 삽질 포인트: "토스 골랐는데 스케줄링을 직접 짜야 한다고?"

**2. 기획자가 먼저 결정해야 하는 5가지** ← 핵심 차별화 섹션
- (1) 과금 모델: 정액제 / 종량제 / 시트당 / 하이브리드 — 의사결정 플로우차트
- (2) 플랜 구조: 몇 개? 연/월 차이는? 무료 플랜은? — 넷플릭스/노션/Slack 실제 사례
- (3) 무료체험: 기간, 카드 등록 시점, 자동전환 여부 — 법적 주의사항 포함
- (4) 업/다운그레이드: 프로레이션(일할 계산) 4가지 방식 비교표
- (5) 해지 흐름: 다크패턴 규제 체크리스트 포함

**3. 결제 실패 — 가장 많이 놓치는 설계**
- 왜 중요한가: 비자발적 이탈이 전체 이탈의 25% (Stripe 데이터)
- 재시도 정책: 3일 간격 x 3회 (업계 표준 근거)
- Grace period: 3~7일 서비스 유지
- 카드 만료 대응: 만료 2개월 전부터 안내
- 고객 알림 타이밍: 실패 즉시 / 재시도 전 / 서비스 중단 전
- 삽질 포인트: "결제 실패 = 즉시 차단? 그러면 CS 폭발합니다"

**4. 환불 정책: 공정위 기준과 실무 사이**
- 디지털콘텐츠 청약철회 7일 규정
- 가분적 콘텐츠(구독)의 환불 계산법
- 부분 환불 시나리오 3가지
- 환불 정책 설계 체크리스트

**5. 우리 서비스에 맞는 구독 구조는?** ← 의사결정 맵
- 콘텐츠 구독 (넷플릭스형): 정액제 + 등급별 플랜 + 동시접속 제한
- SaaS 도구 (노션형): 시트당 과금 + 무료 플랜 + 연/월 차이
- 정기배송 (마켓컬리형): 정액제 + 건너뛰기 + 배송일 변경
- 각각의 추천 구조, 주의점, PG 선택 팁

**마무리**: 체크리스트 — "구독 결제 기획 전 확인 20항목"

## 6. 해외 참고

### Stripe

- **Smart Retries**: ML 기반 500+ 속성으로 최적 재시도 시점 예측. 실패 결제 56% 복구, AI 기반이 고정 스케줄 대비 9% 더 복구
- **비자발적 이탈 데이터**: 이탈 구독의 25%가 결제 실패 때문. 복구된 구독자는 평균 7개월 추가 유지 — "새 고객 획득과 같은 가치"
- **권장 재시도 설정**: 2주 내 8회 재시도
- **Customer Portal**: 고객이 직접 결제수단 변경, 플랜 업/다운그레이드 관리
- **한국 적용 인사이트**: 한국 PG에는 Smart Retries 같은 기능이 없으므로 직접 재시도 로직 구현 필요. 하지만 "3일 간격 x 3회" 정도의 단순 재시도만으로도 상당한 복구 효과

> 출처: [Stripe Revenue Recovery](https://docs.stripe.com/billing/revenue-recovery), [Smart Retries](https://docs.stripe.com/billing/revenue-recovery/smart-retries), [How We Built Smart Retries](https://stripe.com/blog/how-we-built-it-smart-retries)

### Paddle

- **프로레이션 5가지 빌링모드**: (1) prorate + bill now (2) prorate + bill next (3) full amount + now (4) full amount + next (5) do_not_bill
- **분 단위 정밀 계산**: 프로레이션을 분 단위로 계산 (일 단위가 아님)
- **변경 미리보기**: 업/다운그레이드 전 고객에게 예상 청구액 보여주기 가능
- **한국 적용 인사이트**: 한국 PG는 프로레이션 자동 계산을 지원하지 않음. 5가지 빌링모드 중 서비스에 맞는 것을 선택하고 직접 구현해야 함. 가장 일반적: "프로레이션 + 다음 결제일 반영"

> 출처: [Paddle Proration](https://developer.paddle.com/concepts/subscriptions/proration), [Upgrade/Downgrade Guide](https://developer.paddle.com/build/subscriptions/replace-products-prices-upgrade-downgrade)

### StepPay (한국)

- **한국판 구독 결제 플랫폼**: Stripe Billing에 해당하는 한국 서비스
- **제공 기능**: 정액제/종량제/계정수 기반 요금제, 무료체험 자동전환, 할인 적용, 구독매니저 SDK (고객 셀프 관리)
- **참고 포인트**: "주기와 가격만 설정하면 되며, 반복 결제 스케줄러를 만들 필요 없음" — 직접 구현 vs 솔루션 사용의 의사결정 기준이 될 수 있음
- **한국 적용 인사이트**: PG 직접 연동 vs StepPay 같은 구독 전문 솔루션 사용의 트레이드오프를 가이드에서 언급하면 실용적

> 출처: [StepPay 정기결제](https://steppay.kr/recurring-billing), [StepPay 블로그](https://www.steppay.kr/blog/subscription-era)
